# QR Code

## Upstream docs

- Ark UI: https://ark-ui.com/docs/components/qr-code
- Zag API: https://zagjs.com/api/mdx/components/react/qr-code

## Purpose

`QrCode` renders a QR code as an Ark UI-managed SVG with optional overlay and download support.

## Upstream model to preserve

The wrapper follows Ark UI `QrCode` directly. `Root` owns the encoded value, generation options,
element ids, and QR code API. `Frame` renders the SVG, `Pattern` renders the QR modules, `Overlay`
renders centered custom content, and `DownloadTrigger` exports the current frame as an image.

Keep the component composition explicit and Ark-shaped. Do not add an automatic single-component
render path that hides `Frame` or `Pattern`.

## Current behavior contract

- `QrCode` is the short root export and is equivalent to `QrCode.Root`.
- `defaultValue` creates uncontrolled state; `value` plus `onValueChange(details)` creates
  controlled state.
- `encoding` passes Ark/Zag QR generation options such as `ecc: 'L' | 'M' | 'Q' | 'H'` and
  `boostEcc`.
- `id`, `pixelSize`, and `ids` are forwarded to Ark.
- `QrCode.DownloadTrigger` requires `fileName` and `mimeType`; it preserves Ark's `quality` option.
- `QrCode.RootProvider` is preserved for externally owned Ark QR state. Import `useQrCode()` from
  `@moduix/react`; keep other advanced state APIs on `@ark-ui/react/qr-code` as escape hatches.

## Anatomy and exported parts

```text
QrCode / QrCode.Root
├─ QrCode.Frame
│  └─ QrCode.Pattern
├─ QrCode.Overlay
└─ QrCode.DownloadTrigger

QrCode.RootProvider
└─ same part tree connected to useQrCode() from @moduix/react
```

| Export                   | `data-slot`                | Notes                                      |
| ------------------------ | -------------------------- | ------------------------------------------ |
| `QrCode` / `QrCode.Root` | `qr-code-root`             | Ark root, value state, ids, and variables. |
| `QrCode.RootProvider`    | `qr-code-root-provider`    | Root for an external `useQrCode()` API.    |
| `QrCode.Frame`           | `qr-code-frame`            | SVG frame generated by Ark.                |
| `QrCode.Pattern`         | `qr-code-pattern`          | SVG path for QR modules.                   |
| `QrCode.Overlay`         | `qr-code-overlay`          | Centered overlay content.                  |
| `QrCode.DownloadTrigger` | `qr-code-download-trigger` | Button that downloads the current image.   |

No flat part aliases such as `QrCodeRoot` or `QrCodeFrame` are exported.

## Composition

```tsx
import { QrCode } from '@moduix/react';

export function QrCodeDemo() {
  return (
    <QrCode defaultValue="https://moduix.dev/docs/qr-code">
      <QrCode.Frame>
        <QrCode.Pattern />
      </QrCode.Frame>
    </QrCode>
  );
}
```

Use `QrCode.Overlay` with high error correction when central content covers part of the code:

```tsx
<QrCode defaultValue="https://moduix.dev/docs/qr-code" encoding={{ ecc: 'H' }}>
  <QrCode.Frame>
    <QrCode.Pattern />
  </QrCode.Frame>
  <QrCode.Overlay>MX</QrCode.Overlay>
</QrCode>
```

## Upstream feature coverage

- Basic rendering: supported with `Root` / `Frame` / `Pattern`.
- Controlled value: supported through `value` and `onValueChange(details)`.
- Download: supported through `DownloadTrigger` with `fileName`, `mimeType`, and optional
  `quality`.
- Error correction: supported through `encoding.ecc` and `encoding.boostEcc`.
- Fill customization: supported through `className`, SVG `style`, and `--moduix-qr-code-fill`.
- Overlay: supported through `Overlay`; consumers should use stronger error correction when the
  overlay is large.
- Root provider: supported through moduix `useQrCode()` and `QrCode.RootProvider`.
- `asChild`: preserved on every Ark part.
- `id` and `ids`: preserved for stable machine, root, and frame ids.

## Accessibility and state

Ark renders the root as a `div`, the frame as an `svg`, the pattern as a `path`, the overlay as a
`div`, and the download trigger as a native `button`. Refs forward to those Ark elements. The QR
image is visual content; provide surrounding text, labels, or adjacent copy when the encoded value
needs to be available to assistive technology.

There is no roving focus or QR-specific keyboard navigation. `DownloadTrigger` is a native button
and keeps normal button focus, keyboard activation, `disabled`, and `asChild` behavior from Ark.

State attributes are minimal for QR code. Ark supplies `data-scope="qr-code"` and `data-part` on
parts, plus runtime CSS variables on the root:

| Part | Runtime CSS variables                                      | Notes                 |
| ---- | ---------------------------------------------------------- | --------------------- |
| Root | `--qrcode-pixel-size`, `--qrcode-width`, `--qrcode-height` | Generated by Ark/Zag. |

## Defaults and styling

The download trigger follows Button's `--moduix-size-md` default; the visual QR overlay remains independently sized.

Moduix adds default classes, stable `data-slot` hooks, SVG sizing, overlay styling, and a
button-like default style for `DownloadTrigger`. The root has no border, background, padding, or
radius by default.

## Intentional sugar and differences from upstream

- Moduix adds CSS Modules, token-backed defaults, and stable `data-slot` hooks.
- Moduix keeps `QrCode` as the short root import with attached Ark parts and re-exports
  `useQrCode()` for the normal `RootProvider` path.
- Moduix does not add form integration, hidden inputs, value text, or local state around Ark.
- Moduix does not force a default overlay or download button; consumers opt into those parts.

## Agent notes

- Keep the wrapper thin and preserve Ark callback details.
- Keep `Frame` and `Pattern` explicit in examples and docs.
- When changing visual defaults, keep `--moduix-qr-code-*` tokens, docs examples, and registry output in
  sync.
- Keep advanced Ark state APIs other than `useQrCode()` out of the moduix surface; consumers can
  import them from Ark directly when needed.
