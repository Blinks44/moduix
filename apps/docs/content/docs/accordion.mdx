## Basic

Start with `Accordion`, `AccordionItem`, `AccordionHeader`, `AccordionTrigger`, and
`AccordionPanel`. This is the recommended path for FAQs, settings groups, and other stacked
disclosure content.

## Anatomy

The common consumer path is short:

```text
Accordion
└─ AccordionItem[value]
   ├─ AccordionHeader
   │  └─ AccordionTrigger
   │     ├─ label
   │     └─ AccordionTriggerIcon (optional)
   └─ AccordionPanel
      └─ content
```

| Part                   | Role                                                                                                             |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `Accordion`            | Root state machine. Controls open values, keyboard navigation, `multiple`, `disabled`, and `orientation`.        |
| `AccordionItem`        | One collapsible row. Its `value` identifies the row for `defaultValue`, controlled `value`, and change handlers. |
| `AccordionHeader`      | Semantic heading wrapper for a trigger. Use `render` when the heading element needs to change.                   |
| `AccordionTrigger`     | Interactive button that opens or closes the matching panel.                                                      |
| `AccordionTriggerIcon` | Optional helper that renders the default plus icon. Replace its children or omit it when you want a custom cue.  |
| `AccordionPanel`       | The collapsible content region. Put padding/content wrappers inside it so panel height animations stay clean.    |

## Composition

### When to stay high-level

Stay with `Accordion`, `AccordionItem`, `AccordionHeader`, `AccordionTrigger`, and
`AccordionPanel` for most disclosure lists, FAQ blocks, and settings sections.

`Accordion` keeps Base UI root behavior intact. Use it for `value`, `defaultValue`,
`onValueChange`, `multiple`, `disabled`, `orientation`, `loopFocus`, `keepMounted`, and
`hiddenUntilFound`.

Keep `AccordionTriggerIcon` when the default plus icon works. It is narrow sugar for the common
visual affordance and can be omitted entirely when the trigger label is enough.

### When to go custom

Switch to lower-level rendering only when heading semantics or trigger visuals need to change.
`AccordionHeader` supports `render` for custom heading levels, and `AccordionTriggerIcon` accepts
custom children when the default icon should be replaced.

### Default props

| Prop               | Default    | Values / Notes                                      |
| ------------------ | ---------- | --------------------------------------------------- |
| `multiple`         | `false`    | Allows multiple panels to stay open                 |
| `disabled`         | `false`    | Disables the whole accordion                        |
| `orientation`      | `vertical` | `vertical`, `horizontal`                            |
| `loopFocus`        | `true`     | Arrow-key focus wraps between triggers              |
| `keepMounted`      | `false`    | Closed panels stay in the DOM when enabled          |
| `hiddenUntilFound` | `false`    | Uses `hidden="until-found"` on closed panel content |

### Behavioral notes

- `value`, `defaultValue`, and `onValueChange` always work with arrays of item values, even when
  `multiple` is not enabled.
- Put spacing and layout wrappers inside `AccordionPanel` so height measurement and open/close
  animation stay clean.
- When `AccordionTrigger` uses the `render` prop, the custom rendered element owns its styling and
  moduix does not attach the default trigger classes.
- The shipped styles are optimized for vertical accordions. `orientation="horizontal"` changes
  keyboard semantics, but most consumer styling still assumes vertical layout.

### Controlled

Control `value` from React state when the open panels need to coordinate with surrounding UI.

<Preview>
  <ControlledAccordionExample />

  <Preview.Code>
    {`
      import {
        Accordion,
        AccordionHeader,
        AccordionItem,
        AccordionPanel,
        AccordionTrigger,
        AccordionTriggerIcon,
      } from "moduix";
      import { useState } from "react";

      export function ControlledAccordionDemo() {
        const [value, setValue] = useState(["getting-started"] as string[]);

        return (
          <Accordion value={value} onValueChange={setValue}>
            {items.map((item) => (
              <AccordionItem key={item.value} value={item.value}>
                <AccordionHeader>
                  <AccordionTrigger>
                    {item.title}
                    <AccordionTriggerIcon />
                  </AccordionTrigger>
                </AccordionHeader>
                <AccordionPanel>
                  <div className="accordion-panel-content">{item.description}</div>
                </AccordionPanel>
              </AccordionItem>
            ))}
          </Accordion>
        );
      }
    `}

  </Preview.Code>

  <Preview.CSS>{accordionExampleCss}</Preview.CSS>
</Preview>

## Styling

Use `className` on the exported parts for most visual changes. The root ships with a compact
default width through `--accordion-width: 22rem` and `--accordion-max-width: 100%`, so override
those variables or set width directly when the surrounding layout needs a different size.

### Styling hooks

- `data-slot="accordion-root"`, `accordion-item`, `accordion-header`, `accordion-trigger`,
  `accordion-trigger-icon`, and `accordion-panel` are stable selectors on the exported parts.
- Base UI state attributes are exposed to consumers: `data-open` on items and panels,
  `data-panel-open` on triggers, `data-disabled` on root, item, trigger, and panel, and
  `data-orientation` on the root and panel.
