## Choosing the right component

Use `Menubar` for persistent application command bars where top-level triggers stay visible.

- Choose **`Menubar`** for desktop-style app menus, editor command bars, and vertical command rails.
- Choose **`Menu`** for a single trigger that opens one dropdown.
- Choose **`NavigationMenu`** when the popup content is site navigation, not command-style actions.

## Basic

Start with `Menubar`, `MenubarMenu`, `MenubarTrigger`, and `MenubarContent`. This is the
recommended path for application-style command bars.

## Anatomy

The normal consumer path is one visible bar with one or more top-level menus:

```text
Menubar
└─ MenubarMenu
   ├─ MenubarTrigger
   └─ MenubarContent
      ├─ MenubarItem / MenubarLinkItem
      ├─ MenubarCheckboxItem / MenubarRadioItem
      ├─ MenubarSeparator
      └─ MenubarSubmenu
         ├─ MenubarSubmenuTrigger
         │  └─ MenubarSubmenuTriggerIcon
         └─ MenubarSubmenuContent
```

`MenubarContent` internally handles `MenubarPortal`, `MenubarPositioner`, `MenubarPopup`, and
`MenubarViewport`. The low-level popup parts stay exported for custom composition.

| Part                    | Role                                                                     |
| ----------------------- | ------------------------------------------------------------------------ |
| `Menubar`               | Visible root with orientation and roving focus behavior.                 |
| `MenubarMenu`           | One top-level trigger and popup scope inside the bar.                    |
| `MenubarTrigger`        | Interactive trigger in the visible menubar row or column.                |
| `MenubarContent`        | Default popup wrapper with built-in portal, positioner, popup, viewport. |
| `MenubarItem`           | Command-style action row.                                                |
| `MenubarLinkItem`       | Link row that keeps menu behavior.                                       |
| `MenubarCheckboxItem`   | Persistent boolean row.                                                  |
| `MenubarRadioItem`      | Exclusive selection row.                                                 |
| `MenubarSubmenu`        | Nested branch inside popup content.                                      |
| `MenubarItemShortcut`   | Trailing shortcut hint.                                                  |
| `MenubarSubmenuContent` | Submenu popup wrapper with nested defaults.                              |

## Composition

### When to stay high-level

Stay with `Menubar`, `MenubarMenu`, `MenubarTrigger`, and `MenubarContent` for most application
command bars.

`Menubar` keeps the Base UI root behavior intact, so use it for `orientation`, `modal`,
`disabled`, and `loopFocus`.

`MenubarMenu` owns the open state for one top-level menu. Use it for `open`, `defaultOpen`,
`onOpenChange`, `highlightItemOnHover`, `disabled`, and `handle`.

`MenubarContent` is the high-level popup wrapper. It accepts popup `className`, forwards Base UI
popup props, supports the shared positioning props (`side`, `sideOffset`, `align`, `alignOffset`,
`arrowPadding`, `collisionAvoidance`, `collisionBoundary`, `collisionPadding`), and adds
`showArrow` for the built-in popup arrow.

Use `MenubarItem` for imperative actions, `MenubarLinkItem` for navigation,
`MenubarCheckboxItem` and `MenubarRadioItem` for persistent app state, and `MenubarSubmenu` when a
top-level menu needs a nested branch.

### When to go custom

Switch to `MenubarPortal`, `MenubarBackdrop`, `MenubarPositioner`, `MenubarPopup`, `MenubarArrow`,
and `MenubarViewport` only when popup structure, layering, or viewport behavior needs to change.

Use `createMenubarMenuHandle` when a trigger and its `MenubarMenu` root must render in different
places.

If `MenubarTrigger` uses `render={...}`, moduix does not merge the default trigger class. That path
is useful when the host component already owns its surface styling.

### Default props

`MenubarContent` defaults:

| Prop         | Default | Values / Notes                            |
| ------------ | ------- | ----------------------------------------- |
| `showArrow`  | `false` | Enables the built-in arrow                |
| `sideOffset` | `6`     | Any number accepted by Base UI positioner |

`MenubarSubmenuContent` keeps the same public props, but its nested alignment defaults are tuned
for connected submenus:

| Prop          | Default                 | Values                           |
| ------------- | ----------------------- | -------------------------------- |
| `showArrow`   | `false`                 | `true`, `false`                  |
| `sideOffset`  | `({ side }) => 4 \| -4` | `number \| ({ side }) => number` |
| `alignOffset` | `({ side }) => 4 \| -4` | `number \| ({ side }) => number` |

### Behavioral notes

- `Menubar` is persistent UI, so top-level triggers stay rendered even when every popup is closed.
- Use `closeOnClick` on action rows when selecting the command should immediately close the popup.
- `MenubarCheckboxItem` and `MenubarRadioItem` support `indicator="start" | "end" | "none"` to
  control the reserved indicator column.
- Set `orientation="vertical"` when the menubar should become a stacked command rail instead of a
  horizontal app bar.
- `MenubarSubmenuContent` already applies nested offsets, so only override them when submenu
  alignment needs to change deliberately.

## Examples

### Groups And Controls

Use groups, radio items, and checkbox items when menubar commands change application state.

<Preview>
  <GroupsAndControlsMenubarExample />

  <Preview.Code>
    {`
      import {
        Menubar,
        MenubarCheckboxItem,
        MenubarCheckboxItemIndicator,
        MenubarContent,
        MenubarGroup,
        MenubarGroupLabel,
        MenubarItemText,
        MenubarMenu,
        MenubarRadioGroup,
        MenubarRadioItem,
        MenubarRadioItemIndicator,
        MenubarSeparator,
        MenubarTrigger,
      } from "moduix";
      import { useState } from "react";

      export function GroupsAndControlsMenubarDemo() {
        const [sortBy, setSortBy] = useState("name");
        const [showSidebar, setShowSidebar] = useState(true);

        return (
          <Menubar>
            <MenubarMenu>
              <MenubarTrigger>View</MenubarTrigger>
              <MenubarContent>
                <MenubarGroup>
                  <MenubarGroupLabel>Sort</MenubarGroupLabel>
                  <MenubarRadioGroup value={sortBy} onValueChange={setSortBy}>
                    <MenubarRadioItem value="name">
                      <MenubarRadioItemIndicator />
                      <MenubarItemText>Name</MenubarItemText>
                    </MenubarRadioItem>
                    <MenubarRadioItem value="date">
                      <MenubarRadioItemIndicator />
                      <MenubarItemText>Date Modified</MenubarItemText>
                    </MenubarRadioItem>
                  </MenubarRadioGroup>
                </MenubarGroup>

                <MenubarSeparator />

                <MenubarGroup>
                  <MenubarGroupLabel>Panels</MenubarGroupLabel>
                  <MenubarCheckboxItem checked={showSidebar} onCheckedChange={setShowSidebar}>
                    <MenubarCheckboxItemIndicator />
                    <MenubarItemText>Sidebar</MenubarItemText>
                  </MenubarCheckboxItem>
                </MenubarGroup>
              </MenubarContent>
            </MenubarMenu>
          </Menubar>
        );
      }
    `}

### Styling hooks

Use the part that owns the visual concern:

- `Menubar` for the visible bar shell, spacing, radius, and orientation-specific layout
- `MenubarTrigger` for top-level trigger chrome and open/focus states
- `MenubarContent` or `MenubarPopup` for popup surface, sizing, and motion
- `MenubarItem`, `MenubarCheckboxItem`, and `MenubarRadioItem` for row layout and state styling
- `MenubarItemText`, `MenubarItemShortcut`, and `MenubarGroupLabel` for secondary row content
- `MenubarBackdrop`, `MenubarArrow`, and `MenubarViewport` for optional layered composition

The main public hooks include `data-slot="menubar-root"`, `data-slot="menubar-trigger"`,
`data-slot="menubar-popup"`, `data-slot="menubar-item"`, `data-slot="menubar-submenu-trigger"`,
`data-slot="menubar-checkbox-item"`, and `data-slot="menubar-item-shortcut"`.

The shipped styles also use Base UI state attributes directly:

- `data-orientation="horizontal" | "vertical"` on the menubar root
- `data-starting-style` and `data-ending-style` on the backdrop and popup
- `data-highlighted`, `data-popup-open`, and `data-disabled` on interactive rows and triggers
- `data-indicator-position="start" | "end" | "none"` on checkbox and radio rows
- `data-side` on the popup arrow for direction-specific positioning

`MenubarTrigger render={...}` opts out of the default trigger CSS class, so custom trigger hosts
own their full surface and spacing contract.
