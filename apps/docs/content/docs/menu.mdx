## Choosing the right component

Use `Menu` for trigger-anchored dropdown actions, lightweight selections, and nested action trees.

- Choose **`Menu`** for overflow actions, row actions, and command lists opened from a button.
- Choose **`ContextMenu`** when the same interaction should open from right-click or a long press.
- Choose **`Menubar`** for persistent app-level menus in a horizontal bar.
- Choose **`NavigationMenu`** when the popup content is site navigation, not command-style actions.

## Basic

Start with `Menu`, `MenuTrigger`, and `MenuContent`. This is the recommended path for standard
action menus.

## Anatomy

The normal consumer path stays short:

```text
Menu
├─ MenuTrigger
│  └─ MenuTriggerIcon
└─ MenuContent
   ├─ MenuItem
   ├─ MenuSeparator
   └─ MenuSubmenu
      ├─ MenuSubmenuTrigger
      │  └─ MenuSubmenuTriggerIcon
      └─ MenuSubmenuContent
         └─ MenuItem
```

`MenuContent` internally handles `MenuPortal`, `MenuPositioner`, `MenuPopup`, and the optional
`MenuArrow`. Those lower-level parts stay exported for custom composition.

| Part                 | Role                                                               |
| -------------------- | ------------------------------------------------------------------ |
| `Menu`               | Root state, open state, and keyboard interactions.                 |
| `MenuTrigger`        | Control that opens the menu.                                       |
| `MenuContent`        | Default popup wrapper with built-in portal, positioner, and popup. |
| `MenuItem`           | Command-style action row.                                          |
| `MenuLinkItem`       | Link row that keeps menu behavior.                                 |
| `MenuSubmenu`        | Nested branch inside another menu.                                 |
| `MenuCheckboxItem`   | Persistent boolean row.                                            |
| `MenuRadioItem`      | Exclusive selection row.                                           |
| `MenuItemShortcut`   | Trailing shortcut hint.                                            |
| `MenuTriggerIcon`    | Optional trigger icon helper.                                      |
| `MenuSubmenuContent` | Submenu popup wrapper with nested defaults.                        |

## Composition

### When to stay high-level

Stay with `Menu`, `MenuTrigger`, and `MenuContent` for most action menus.

`Menu` keeps the Base UI root behavior intact, so use it for `open`, `defaultOpen`,
`onOpenChange`, `modal`, and `handle`.

`MenuContent` is the high-level popup wrapper. It accepts popup `className`, forwards Base UI popup
props, supports the shared positioning props (`side`, `sideOffset`, `align`, `alignOffset`,
`arrowPadding`, `collisionAvoidance`, `collisionBoundary`, `collisionPadding`), and adds
`showArrow` for the built-in popup arrow.

Use `MenuItem` for imperative actions, `MenuLinkItem` for navigation, `MenuCheckboxItem` and
`MenuRadioItem` for persistent app state, and `MenuSubmenu` when the menu needs a nested branch.
Use `tone="destructive"` on `MenuItem` or `MenuLinkItem` when destructive rows should keep a soft
destructive highlight instead of the shared dark highlight fill.

### When to go custom

Switch to `MenuPortal`, `MenuBackdrop`, `MenuPositioner`, `MenuPopup`, `MenuArrow`, and
`MenuViewport` only when popup structure, layering, or scrolling behavior needs to change.

Use `createMenuHandle` when the trigger and menu root must render in different places.

If `MenuTrigger` uses `render={...}`, moduix does not merge the default trigger class. That path is
useful when the host component already owns its surface styling.

### Default props

`MenuContent` defaults:

| Prop         | Default | Values / Notes                            |
| ------------ | ------- | ----------------------------------------- |
| `showArrow`  | `false` | Enables the built-in arrow                |
| `sideOffset` | `8`     | Any number accepted by Base UI positioner |

`MenuSubmenuContent` keeps the same public props, but its nested alignment defaults are tuned for
connected submenus:

| Prop          | Default                 | Values                           |
| ------------- | ----------------------- | -------------------------------- |
| `showArrow`   | `false`                 | `true`, `false`                  |
| `sideOffset`  | `({ side }) => 4 \| -4` | `number \| ({ side }) => number` |
| `alignOffset` | `({ side }) => 4 \| -4` | `number \| ({ side }) => number` |

### Behavioral notes

- Use `closeOnClick` on action rows when the selection should immediately close the menu.
- Use `tone="destructive"` for delete, archive, remove, or reset actions that should read as
  dangerous in both resting and highlighted states.
- `MenuCheckboxItem` and `MenuRadioItem` support `indicator="start" | "end" | "none"` to control
  the reserved indicator column.
- `openOnHover` and `delay` belong on `MenuTrigger`, not on the menu root.
- `MenuSubmenuContent` already applies nested offsets, so only override them when submenu alignment
  needs to change deliberately.

### Groups And Controls

Use groups, radio items, and checkbox items when the menu changes application state.

<Preview>
  <GroupsAndControlsMenuExample />

  <Preview.Code>
    {`
      import {
        Button,
        Menu,
        MenuCheckboxItem,
        MenuCheckboxItemIndicator,

        MenuRadioItem,
        MenuRadioItemIndicator,
        MenuSeparator,
        MenuTrigger,
        MenuTriggerIcon,
      } from "moduix";
      import { useState } from "react";

      export function GroupsAndControlsMenu() {
        const [sortBy, setSortBy] = useState("date");

              View
              <MenuTriggerIcon />
            </MenuTrigger>
            <MenuContent>
              <MenuGroup>
                <MenuGroupLabel>Sort</MenuGroupLabel>
                <MenuRadioGroup value={sortBy} onValueChange={setSortBy}>
                  <MenuRadioItem value="date">
                    <MenuRadioItemIndicator />
                    <MenuItemText>Date</MenuItemText>

                    <MenuRadioItemIndicator />
                    <MenuItemText>Name</MenuItemText>
                  </MenuRadioItem>
                </MenuRadioGroup>

              <MenuSeparator />
              <MenuGroup>
                <MenuGroupLabel>Workspace</MenuGroupLabel>
                <MenuCheckboxItem checked={showMinimap} onCheckedChange={setShowMinimap}>
                  <MenuCheckboxItemIndicator />
                  <MenuItemText>Minimap</MenuItemText>
                </MenuCheckboxItem>
              </MenuGroup>
            </MenuContent>
          </Menu>
        );
      }
    `}

### Styling hooks

Use the part that owns the visual concern:

- `MenuTrigger` for trigger chrome and focus styling
- `MenuContent` or `MenuPopup` for the popup surface, sizing, and motion
- `MenuItem`, `MenuCheckboxItem`, and `MenuRadioItem` for row layout and state styling
- `tone="destructive"` on `MenuItem` and `MenuLinkItem` for destructive row styling
- `MenuItemText`, `MenuItemShortcut`, and `MenuGroupLabel` for secondary row content
- `MenuBackdrop` and `MenuArrow` for optional layered composition

The main public hooks include `data-slot="menu-trigger"`, `data-slot="menu-popup"`,
`data-slot="menu-item"`, `data-slot="menu-submenu-trigger"`, `data-slot="menu-checkbox-item"`,
and `data-slot="menu-item-shortcut"`.

The shipped styles also use Base UI state attributes directly:

- `data-starting-style` and `data-ending-style` on the backdrop and popup
- `data-highlighted`, `data-popup-open`, and `data-disabled` on interactive rows
- `data-indicator-position="start" | "end" | "none"` on checkbox and radio rows
- `data-side` on the popup arrow for direction-specific positioning

`MenuTrigger render={...}` opts out of the default trigger CSS class, so custom trigger hosts own
their full surface and spacing contract.
