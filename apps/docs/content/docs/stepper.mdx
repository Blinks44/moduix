Use `Stepper` with `StepperList` and explicit `StepperItem step={...}` values when you need ordered
progress without a large wizard abstraction.

## Anatomy

The normal consumer path stays small and explicit:

```text
Stepper
└─ StepperList
   └─ StepperItem
      └─ StepperTrigger
         ├─ StepperIndicator
         └─ StepperContent
            ├─ StepperTitle
            └─ StepperDescription
```

| Part                 | Role                                                               |
| -------------------- | ------------------------------------------------------------------ |
| `Stepper`            | Navigation landmark. Owns `currentStep` and `orientation`.         |
| `StepperList`        | Ordered list of visible steps.                                     |
| `StepperItem`        | One numbered step. Resolves `inactive`, `current`, or `completed`. |
| `StepperTrigger`     | Optional interactive wrapper. Defaults to a native button.         |
| `StepperIndicator`   | Number or completed check icon.                                    |
| `StepperContent`     | Layout wrapper for step text.                                      |
| `StepperTitle`       | Main step label.                                                   |
| `StepperDescription` | Secondary helper text.                                             |

## Composition

### When to stay high-level

Stay with `Stepper`, `StepperList`, `StepperItem`, and `StepperTrigger` for most onboarding flows,
setup checklists, and multi-step navigation.

The component stays intentionally narrow:

- `currentStep` derives item status only
- panel rendering stays in consumer code
- form state, validation, and route state stay in consumer code
- step navigation stays explicit through `StepperTrigger`

Use the default button trigger when steps should update local UI state. Use `render={<div />}` for a
display-only progress indicator.

### When to go custom

Use `render` on `StepperTrigger` when navigation should come from your own element, such as a plain
anchor or a router link component.

Use `status="inactive" | "current" | "completed"` on `StepperItem` when the visible progress should
not follow the numeric `currentStep` order.

### Default props

| Prop          | Default      | Values / Notes                                  |
| ------------- | ------------ | ----------------------------------------------- |
| `orientation` | `horizontal` | `horizontal` or `vertical` on `Stepper`         |
| `aria-label`  | `"Steps"`    | Accessible label on the root navigation element |
| `type`        | `button`     | Applied by `StepperTrigger` on the default path |

### Behavioral notes

- `StepperItem` requires `step`; the component does not auto-number from child position.
- When `currentStep` is set, lower steps become `completed`, the matching step becomes `current`,
  and higher steps stay `inactive`.
- Current steps automatically receive `aria-current="step"` on both `StepperItem` and
  `StepperTrigger`.
- Disable `StepperTrigger` directly when a step should stay visible but not reachable yet.

## Examples

### Controlled State

Keep the current step in React state when each step should be clickable.

<Preview>
  <ControlledStepperExample />

  <Preview.Code>
    {`
      import {
        Stepper,
        StepperContent,
        StepperDescription,
        StepperIndicator,
        StepperItem,
        StepperList,
        StepperTitle,
        StepperTrigger,
      } from "moduix";
      import { useState } from "react";

      export function ControlledStepperDemo() {
        const [currentStep, setCurrentStep] = useState(2);

        return (
          <Stepper currentStep={currentStep}>
            <StepperList>
              {steps.map((item) => (
                <StepperItem key={item.step} step={item.step}>
                  <StepperTrigger onClick={() => setCurrentStep(item.step)}>
                    <StepperIndicator />
                    <StepperContent>
                      <StepperTitle>{item.title}</StepperTitle>
                    </StepperContent>
                  </StepperTrigger>
                </StepperItem>
              ))}
            </StepperList>
          </Stepper>
        );
      }
    `}

  </Preview.Code>

  <Preview.Data>
    {`      const steps = [
        { step: 1, title: "Account" },
        { step: 2, title: "Profile" },
        { step: 3, title: "Billing" },
      ];`}

  </Preview.Data>
</Preview>

Style `Stepper` through `className`, the public `data-slot` and `data-state` hooks, and the public

`packages/ui/src/styles/theme.css`.

### Styling hooks

The useful public hooks are:

- `className` on `Stepper`, `StepperList`, `StepperItem`, `StepperTrigger`, `StepperIndicator`, `StepperContent`, `StepperTitle`, and `StepperDescription`
- `data-slot="stepper-root"`, `stepper-list`, `stepper-item`, `stepper-trigger`, `stepper-indicator`, `stepper-content`, `stepper-title`, and `stepper-description`
- `data-orientation="horizontal" | "vertical"` on the root, list, item, trigger, indicator, and content
- `data-state="inactive" | "current" | "completed"` on `StepperItem`, `StepperTrigger`, `StepperIndicator`, `StepperContent`, `StepperTitle`, and `StepperDescription`
- disabled styling hooks through native `[disabled]`, `[aria-disabled='true']`, and `[data-disabled]` on `StepperTrigger`
