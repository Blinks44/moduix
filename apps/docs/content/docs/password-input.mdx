## Choosing the right component

Use `PasswordInput` when a single secure value should stay masked by default while still allowing a
user-controlled reveal action.

- Choose **`Input`** for plain text-like fields without a visibility toggle.
- Choose **`PasswordInput`** for passwords, API keys, and other secure strings.
- Choose **`InputGroup`** when the field needs a custom inline action that is not password visibility.

## Anatomy

```tsx
<PasswordInput>
  <PasswordInput.Label />
  <PasswordInput.Control>
    <PasswordInput.Input />
    <PasswordInput.VisibilityTrigger>
      <PasswordInput.Indicator />
    </PasswordInput.VisibilityTrigger>
  </PasswordInput.Control>
</PasswordInput>
```

| Part                                   | Role                                                                           |
| -------------------------------------- | ------------------------------------------------------------------------------ |
| `PasswordInput` / `PasswordInput.Root` | Owns password visibility state and field state.                                |
| `PasswordInput.RootProvider`           | Renders the same root part from a `usePasswordInput` state object.             |
| `PasswordInput.Label`                  | Labels the input and receives required, disabled, invalid, and readonly state. |
| `PasswordInput.Control`                | Positions the input and visibility trigger together.                           |
| `PasswordInput.Input`                  | Native input part whose `type` switches between `password` and `text`.         |
| `PasswordInput.VisibilityTrigger`      | Button that toggles the visibility state.                                      |
| `PasswordInput.Indicator`              | Shows visible/hidden content; moduix supplies default icons.                   |
| `PasswordInput.Context`                | Inline render-prop access to Ark password input state.                         |

## Composition

### Recommended composition

Render the Ark password-input parts directly: root, label, control, input, visibility trigger, and
indicator. Put value props such as `value`, `defaultValue`, `placeholder`, and `onChange` on
`PasswordInput.Input`; put visibility, form-state, ids, translations, and password-manager props on
the root.

### When to go custom

Use `asChild` only when a part must render one custom semantic element. Replace the default
indicator icons by passing `fallback` for the hidden state and children for the visible state.

### Default props

| Prop                             | Default              | Notes                                                  |
| -------------------------------- | -------------------- | ------------------------------------------------------ |
| `autoComplete`                   | `"current-password"` | Set on the input through the root machine.             |
| `defaultVisible`                 | `false`              | Initial uncontrolled visibility state.                 |
| `visible`                        | uncontrolled         | Controlled visibility state.                           |
| `onVisibilityChange`             | -                    | Receives Ark details; read `details.visible`.          |
| `translations.visibilityTrigger` | Ark default          | Function used for localized visibility trigger labels. |
| `ignorePasswordManagers`         | `false`              | Adds supported password-manager ignore hints.          |

### Behavioral notes

- Control visibility with `visible` and `onVisibilityChange(details)`.
- Control the field value on `PasswordInput.Input` with native `value` and `onChange`.
- Use root props such as `name`, `required`, `disabled`, `readOnly`, `invalid`, `ids`, and
  `translations` directly from Ark UI.
- Use `usePasswordInput` with `PasswordInput.RootProvider` when another component needs the same
  visibility state outside the rendered tree.
- `PasswordInput.Input`, `PasswordInput.VisibilityTrigger`, and `PasswordInput.Indicator` expose
  `data-state="visible" | "hidden"` for styling.

### Strength Meter

Keep password value state on `PasswordInput.Input` and compose extra UI after the control.

<Preview>
  <PasswordInputStrengthMeterExample />

  <Preview.Code>
    {`
      import { PasswordInput } from "@moduix/react";
      import { useState } from "react";

      export function PasswordInputStrengthMeterDemo() {
        const [password, setPassword] = useState("asdfasdf");
        const strength = getPasswordStrength(password);

        return (
          <PasswordInput>
            <PasswordInput.Label>Password</PasswordInput.Label>
            <PasswordInput.Control>
              <PasswordInput.Input
                value={password}
                onChange={(event) => setPassword(event.currentTarget.value)}
                placeholder="Enter your password"
              />
              <PasswordInput.VisibilityTrigger>
                <PasswordInput.Indicator />
              </PasswordInput.VisibilityTrigger>
            </PasswordInput.Control>
            {strength ? (
              <div className="password-input-demo-strength-meter">
                <div className="password-input-demo-strength-bar">
                  <div className="password-input-demo-strength-fill" data-strength={strength} />
                </div>
                <div className="password-input-demo-strength-label">{strength} password</div>
              </div>
            ) : null}
          </PasswordInput>
        );
      }

      function getPasswordStrength(password: string) {
        if (!password) return null;
        if (password.length >= 10 && /[0-9]/.test(password) && /[^a-zA-Z0-9]/.test(password)) {
          return "strong";
        }
        if (password.length >= 6) return "medium";
        return "weak";
      }
    `}

  </Preview.Code>

  <Preview.CSS>
    {`

        height: var(--spacing-2);
        overflow: hidden;
        background: var(--color-muted);
        border-radius: var(--radius-full);
      }

      .password-input-demo-strength-fill[data-strength="medium"] {
        width: 60%;

      }

### With Validation

Set `invalid` on the root to let Ark propagate invalid state to the label, input, control, and
indicator parts.

<Preview>
  <PasswordInputValidationExample />

  <Preview.Code>
    {`
      import { PasswordInput } from "@moduix/react";
      import { useState } from "react";

      export function PasswordInputValidationDemo() {
        const [password, setPassword] = useState("");
        const isValid = password.length >= 8;
        const invalid = !isValid && password.length > 0;

        return (
          <PasswordInput invalid={invalid}>
            <PasswordInput.Label>Password (min 8 characters)</PasswordInput.Label>
            <PasswordInput.Control>
              <PasswordInput.Input
                value={password}
                onChange={(event) => setPassword(event.currentTarget.value)}
                placeholder="Enter your password"
              />
              <PasswordInput.VisibilityTrigger>
                <PasswordInput.Indicator />
              </PasswordInput.VisibilityTrigger>
            </PasswordInput.Control>
            {password.length > 0 ? (
              <p className="password-input-demo-validation-message" data-valid={isValid}>
                {isValid ? "Password is valid." : "Password must be at least 8 characters."}
              </p>
            ) : null}
          </PasswordInput>
        );
      }
    `}

  </Preview.Code>

## Styling

`PasswordInput` styles the control shell, input text area, and visibility trigger with
`--password-input-*` tokens layered over the shared input contract.

### Styling Hooks

- Root and root provider expose `data-slot="password-input-root"` and
  `data-slot="password-input-root-provider"`.
- Ark parts keep `data-scope="password-input"` and `data-part` values for `root`, `label`,
  `control`, `input`, `visibility-trigger`, and `indicator`.
- State styling uses `data-state="visible" | "hidden"` on input, trigger, and indicator, plus
  `data-disabled`, `data-invalid`, `data-readonly`, and `data-required` where Ark provides them.
- Ark does not expose password-input-specific runtime CSS variables; moduix adds
  `--password-input-*` variables for visual defaults.
