## API Reference

`PasswordInput` does not wrap a dedicated Base UI primitive. It is a moduix composition component
built from the local `InputGroup`, `InputGroupInput`, and `InputGroupButton` wrappers, then narrowed
to the common password-field workflow.

## Choosing the right component

Use `PasswordInput` when the field value should stay masked by default but users still need an
inline reveal action.

- Choose **`Input`** for plain text-like fields without inline actions.
- Choose **`PasswordInput`** when the field needs a built-in visibility toggle.
- Choose **`InputGroup`** when the field needs a different inline action, prefix, suffix, or
  composed shell.

## Anatomy

The normal consumer path is short:

```text
Field (optional wrapper)
└─ PasswordInput
   ├─ input
   └─ visibility toggle button
```

| Part                     | Role                                                               |
| ------------------------ | ------------------------------------------------------------------ |
| `PasswordInput`          | Grouped field shell plus password-specific show/hide behavior.     |
| inner grouped input      | The real input element used for form value, validation, and focus. |
| visibility toggle button | Icon-only button that reveals or masks the current password value. |

## Composition

### When to stay high-level

Stay with `PasswordInput` for the standard password-field path. It keeps the `Input` contract for
value, validation, native attributes, and sizing while owning one narrow extra behavior: visibility
toggling.

Use `defaultVisible` to set the initial reveal state. Use `onVisibleChange` when the parent should
react to reveal and hide events.

Set `size` on `PasswordInput` to resize the whole grouped control. The shell, nested input, and
toggle stay aligned automatically.

`PasswordInput` is designed to work directly inside `Field`, so labels, descriptions, errors, and
native validation stay declarative without extra wiring.

### When to go custom

Switch away from `PasswordInput` when the field needs more than a built-in visibility toggle:

- use `Input` for plain text-like fields
- use `InputGroup` when the inline action, prefix, suffix, or shell structure needs to change

`PasswordInput` intentionally does not expose a public `type` prop. The component owns the
`password` and `text` switching internally so the API stays password-specific.

### Default props

| Prop                     | Default                                            | Notes                                                       |
| ------------------------ | -------------------------------------------------- | ----------------------------------------------------------- |
| `size`                   | `md`                                               | Supports `xs`, `sm`, `md`, `lg`, and `xl`.                  |
| `defaultVisible`         | `false`                                            | Starts the field masked unless you opt into reveal-on-load. |
| `visibilityToggleLabels` | `{ show: "Show password", hide: "Hide password" }` | Accessible toggle labels for the two states.                |
| toggle button `type`     | `button`                                           | Prevents accidental form submission.                        |

### Behavioral notes

- The ref points to the real input element, not the toggle button.
- `className` styles the grouped shell, not the nested input.
- `disabled` disables both the input and toggle button.
- `readOnly` keeps the input non-editable while still allowing reveal and hide.
- The toggle button keeps input focus on pointer interaction, so reveal and hide does not steal
  focus from the field.

## Examples

### Controlled Value

Use `value` and `onValueChange` when the password value should stay in React state.

<Preview>
  <ControlledPasswordInputExample />

  <Preview.Code>
    {`
      import { Field, PasswordInput } from "moduix";
      import { useState } from "react";

      export function ControlledPasswordInputDemo() {
        const [value, setValue] = useState("");

        return (
          <Field className="password-input-demo-field">
            <Field.Label>Workspace password</Field.Label>
            <PasswordInput
              autoComplete="new-password"
              value={value}
              onValueChange={setValue}
              placeholder="Type to control value"
            />

## Styling

Style `PasswordInput` through `className`, `data-slot` hooks, and the inherited `--input-group-*`
CSS variables shown below.

### Styling hooks

Use the part that owns the visual concern:

- `className` on `PasswordInput` for shell-wide overrides
- `data-slot="password-input-root"` for grouped shell targeting
- `data-slot="password-input-toggle"` for the reveal and hide button
- `data-slot="input-group-input"` for the nested input element
- `data-size="xs" | "sm" | "md" | "lg" | "xl"` plus field state attributes such as `data-disabled`,
  `data-invalid`, `data-valid`, `data-filled`, and `data-focused`
