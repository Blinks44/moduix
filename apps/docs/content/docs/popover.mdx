## Choosing the right component

Use `Popover` for compact, contextual content that stays anchored to one trigger and can contain
interactive UI.

- Choose **`Tooltip`** for short, non-interactive hints or labels.
- Choose **`Dialog`** when the content needs a dedicated overlay workflow, stronger interruption, or
  a larger surface.
- Choose **`Popover`** when users need lightweight actions, status details, or short forms without
  leaving the current layout.

## Basic

Start with `Popover`, `PopoverTrigger`, and `PopoverContent`. This is the recommended path for
most menus, notifications, and compact forms.

## Anatomy

The normal consumer path stays short:

```text
Popover
├─ PopoverTrigger
│  └─ trigger content
└─ PopoverContent
   ├─ PopoverHeader (optional)
   │  ├─ PopoverTitle (optional)
   │  └─ PopoverDescription (optional)
   ├─ PopoverBody (optional)
   └─ PopoverFooter (optional)
      └─ PopoverClose (optional)
```

`PopoverContent` internally renders `PopoverPortal`, `PopoverPositioner`, and `PopoverPopup`.
`PopoverBackdrop`, `PopoverArrow`, and `PopoverViewport` stay explicit building blocks for advanced
composition.

| Part                 | Role                                                                            |
| -------------------- | ------------------------------------------------------------------------------- |
| `Popover`            | Root state and behavior. Controls `open`, `defaultOpen`, `modal`, and `handle`. |
| `PopoverTrigger`     | Interactive anchor that opens and closes the popup.                             |
| `PopoverContent`     | Default popup wrapper for the common case.                                      |
| `PopoverHeader`      | Layout wrapper for title and description.                                       |
| `PopoverTitle`       | Accessible popup heading.                                                       |
| `PopoverDescription` | Supporting text inside the popup.                                               |
| `PopoverBody`        | Optional free-form content region.                                              |
| `PopoverFooter`      | Optional action row.                                                            |
| `PopoverClose`       | Built-in dismiss action.                                                        |

## Composition

### When to stay high-level

Stay with `Popover`, `PopoverTrigger`, and `PopoverContent` for most contextual actions and compact
information panels.

`Popover` keeps the Base UI root contract intact. Use it for `open`, `defaultOpen`,
`onOpenChange`, `modal`, and `handle`.

`PopoverContent` is the default popup path. It accepts popup `className`, supports the shared
positioning props (`side`, `sideOffset`, `align`, `alignOffset`, `arrowPadding`,
`collisionAvoidance`, `collisionBoundary`, `collisionPadding`), and adds one convenience prop:

- `showArrow` for the built-in popup arrow

### When to go custom

Switch to `PopoverPortal`, `PopoverBackdrop`, `PopoverPositioner`, `PopoverPopup`,
`PopoverArrow`, and `PopoverViewport` when you need portal props, backdrop control, custom arrow
content, or a different popup structure.

`PopoverContent` does not expose portal-level props such as `keepMounted` or `container`. Use
explicit `PopoverPortal` composition when those matter.

### Default props

| Prop         | Default | Values / Notes                            |
| ------------ | ------- | ----------------------------------------- |
| `showArrow`  | `false` | Enables the built-in `PopoverArrow`       |
| `sideOffset` | `8`     | Any number accepted by Base UI positioner |

### Behavioral notes

- When `render` is provided to `PopoverTrigger`, moduix does not add the default trigger class.
  Style the rendered element directly or pass a component such as `Button`.
- Keep `nativeButton={false}` when `PopoverTrigger` renders a non-button element such as `div` or
  `span`.
- Use `initialFocus` and `finalFocus` on `PopoverContent` when focus needs to move predictably on
  open and close.
- If you use `modal` or `modal="trap-focus"`, keep a `PopoverClose` action inside the popup so
  keyboard users always have a clear exit path.
- `PopoverViewport` is for panel-style inner transitions. Do not use it as a generic scroll
  container. For tall content, add a scrollable wrapper inside `PopoverBody` instead.

### Placement

Pass `side`, alignment props, and collision options to `PopoverContent` when popup placement needs
to be explicit.

<Preview>

  <Preview.Code>
    {`
      import {
        Button,
        Popover,
        PopoverContent,
        PopoverDescription,
        PopoverHeader,
        PopoverTitle,
        PopoverTrigger,
      } from "moduix";
      import { useState } from "react";

      const sides = ["top", "right", "bottom", "left"] as const;

      export function SideControlPopoverDemo() {
        const [side, setSide] = useState("bottom" as (typeof sides)[number]);

        return (
          <div className={styles.stack}>
            <div className={styles.sideButtons}>
              {sides.map((item) => (
                <button
                  key={item}

                  <PopoverTitle>Placement</PopoverTitle>

### Styling hooks

The useful public hooks are concentrated on the rendered parts:

- `PopoverContent className` styles `PopoverPopup`, not the portal or positioner
- `data-slot="popover-trigger"`, `popover-backdrop`, `popover-positioner`, `popover-popup`,
  `popover-arrow`, `popover-title`, `popover-description`, `popover-header`, `popover-body`,
  `popover-footer`, and `popover-close`
- Base UI popup state attributes such as `data-open`, `data-closed`, `data-starting-style`, and
  `data-ending-style`
- `PopoverTrigger` state attributes such as `data-popup-open` and `data-disabled`
- `PopoverArrow[data-side]` for side-specific arrow styling
- `PopoverViewport` transition state such as `data-current`, `data-previous`, and
  `data-activation-direction`
