## Install with shadcn (optional)

If you want `Input` to become part of your own project source instead of staying in
`node_modules`, install it from the moduix GitHub registry:

<Tabs items={['npm', 'pnpm', 'bun']}>
  <Tab value="npm">

```bash
npx shadcn@latest add Blinks44/moduix/input
```

  </Tab>
  <Tab value="pnpm">

```bash
pnpm dlx shadcn@latest add Blinks44/moduix/input
```

  </Tab>
  <Tab value="bun">

```bash
bunx shadcn@latest add Blinks44/moduix/input
```

  </Tab>
</Tabs>

This install path also pulls in the shared moduix foundation files that `Input` depends on, such
as styles, icons, and small utilities.

`Input` is a thin styled wrapper over the Base UI primitive. The recommended path is to compose it
inside `Field` so labels, descriptions, and validation stay declarative.

## Anatomy

```text
Field (optional wrapper)
└─ Input
```

| Part    | Role                                                                     |
| ------- | ------------------------------------------------------------------------ |
| `Input` | Native input element with Base UI state integration and style variables. |

## Composition

`Input` keeps the primitive contract intact. Pass Base UI props such as `value`, `defaultValue`,
`onValueChange`, `render`, function `className`, and function `style` directly to the component.

Use `size` for the visual size and `htmlSize` only when you need the native HTML `size` attribute
without giving up the design-system size prop.

```tsx
<Input size="lg" htmlSize={24} placeholder="Search projects" />
```

### Defaults

| Prop        | Default | Values                                 |
| ----------- | ------- | -------------------------------------- |
| `size`      | `md`    | `xs`, `sm`, `md`, `lg`, `xl`           |
| `htmlSize`  | -       | Any valid native `input` `size` number |
| `className` | -       | `string` or Base UI state callback     |

`Input` is a single styled root, so there are no slot props, class name maps, or internal layout
parts to learn. Style the element directly with `className` or the public `--input-*` CSS
variables.

## Examples

### Controlled

Use `value` and `onValueChange` when the input value should stay in React state.

<Preview>
  <ControlledInputExample />

  <Preview.Code>
    {`
      import { Field, FieldLabel, Input } from "moduix";
      import { useState } from "react";

      function ControlledInput() {
        const [value, setValue] = useState("");

        return (
          <Field className={styles.field}>
            <FieldLabel>Username</FieldLabel>
            <Input
              value={value}
              onValueChange={setValue}
              placeholder="Type to control value"
            />
          </Field>
        );
      }
    `}

  </Preview.Code>

  <Preview.CSS>
    {`
      .field {
        width: min(20rem, 100%);
      }
    `}

  </Preview.CSS>
</Preview>

### Native Attributes

Use native input attributes for keyboard, autocomplete, validation, and form behavior. Reach for
`htmlSize` only when the native `size` attribute matters.

<Preview>
  <InputNativeAttributesExample />

      export function NativeInputAttributesDemo() {
        return (
          <Field className={styles.field}>

              htmlSize={8}
              inputMode="numeric"
              maxLength={6}
              name="security-code"
              placeholder="000000"
              type="text"

### Field Validation

Use `Field` when the input needs labels, errors, native validity, and Base UI validation state.

<Preview>
  <InputFieldValidationExample />

  <Preview.Code>
    {`
      import { Field, FieldError, FieldLabel, Input } from "moduix";

      export function InputFieldValidationDemo() {
        return (
          <Field className={styles.field} validationMode="onBlur">
            <FieldLabel>Email</FieldLabel>
            <Input required type="email" placeholder="name@example.com" />
            <FieldError match="valueMissing">Please enter your email.</FieldError>
            <FieldError match="typeMismatch">Enter a valid email address.</FieldError>

## Custom Composition

You can render `Input` outside `Field` when another form abstraction already provides labeling. In
that case, keep the accessible name explicit with `aria-label`, `aria-labelledby`, or a native
`label`.

<Preview>
  <StandaloneInputExample />

  <Preview.Code>
    {`
      import { Input } from "moduix";

      export function StandaloneInputDemo() {
        return (
          <div className={styles.field}>
            <Input aria-label="Search projects" placeholder="Search projects" />
          </div>
        );
      }
    `}

  </Preview.Code>

  <Preview.CSS>
    {`
