## Choosing the right component

Use `Input` for plain single-line text-like values when one styled native input is enough.

- Choose **`Input`** for standard text, email, search, or code fields.
- Choose **`InputGroup`** when the control needs inline addons or action buttons in the same shell.
- Choose **`PasswordInput`** when the field needs a built-in visibility toggle.
- Choose **`NumberField`** when the value is numeric and needs parsing, stepping, or formatting.
- Choose **`Textarea`** when the content can span multiple lines.

## Anatomy

The public shape is intentionally small:

```text
Field (optional wrapper)
└─ Input
```

| Part    | Role                                                                     |
| ------- | ------------------------------------------------------------------------ |
| `Input` | Native input element with Base UI state integration and style variables. |

## Composition

### When to stay high-level

Stay with `Input` for most plain text-like fields. The moduix wrapper keeps the Base UI input
contract intact and adds the shared visual sizing surface.

Pass Base UI props such as `value`, `defaultValue`, `onValueChange`, `render`, function
`className`, and function `style` directly to `Input`.

Use `size` for visual density. Use `htmlSize` only when you need the native HTML `size` attribute
without giving up the design-system `size` prop.

### When to go custom

There is no lower-level moduix `Input` composition tree to drop into. When the field needs more
structure, switch to the component that owns that behavior:

- `InputGroup` for inline prefixes, suffixes, or action buttons
- `PasswordInput` for a visibility toggle
- `NumberField` for numeric parsing and step controls

Use standalone `Input` outside `Field` when another form abstraction already provides the label and
validation wiring. In that case, keep the accessible name explicit with a real `label`,
`aria-label`, or `aria-labelledby`.

### Default props

| Prop        | Default | Values                                 |
| ----------- | ------- | -------------------------------------- |
| `size`      | `md`    | `xs`, `sm`, `md`, `lg`, `xl`           |
| `htmlSize`  | -       | Any valid native `input` `size` number |
| `className` | -       | `string` or Base UI state callback     |

### Behavioral notes

- Use `value` with `onValueChange` for controlled inputs, or `defaultValue` for uncontrolled ones.
- `disabled` removes the field from interaction and form submission; `readOnly` keeps it focusable
  and submittable.
- `Input` works directly with `Field`, so you do not need `FieldControl` for this component.
- Prefer semantic native attributes such as `type`, `autoComplete`, `inputMode`, and `name` before
  adding custom input logic.

## Examples

### Controlled

Use `value` and `onValueChange` when the input value should stay in React state.

<Preview>

    {`

          <Field className={styles.field}>

### Styling hooks

`Input` exposes a single styled root, so the useful public hooks stay simple:

- `data-slot="input-root"` for direct element targeting
- `data-size="xs" | "sm" | "md" | "lg" | "xl"` for size-specific styling
- Base UI state attributes such as `data-disabled`, `data-invalid`, `data-valid`, `data-filled`,
  `data-focused`, `data-dirty`, and `data-touched`
- native `:read-only` and `[data-readonly]` matching when the field is readonly
