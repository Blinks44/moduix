# Composition Patterns

Many moduix components are compound wrappers around Base UI primitives. They expose visible parts
for product markup and usually keep service slots internal by default so behavior stays consistent.
Some components, such as `Dialog`, also export those low-level parts when manual composition is a
real customization path.

These patterns stay the same whether you import from `moduix` as a package or install source files
into your own project with `shadcn`.

## Default Path First

moduix prefers explicit visible parts over generic slot prop bags. In practice, the usual order is:

1. Start with the high-level wrapper part the component gives you, such as `DialogContent` or
   `PopoverContent`.
2. Use `className` on the visible part for styling.
3. Drop to low-level parts only when you need to own portal behavior, overlay structure, or a
   different DOM arrangement.

## Service Slots

Service slots are infrastructure layers that usually should not be composed manually in app code:

| Slot         | Responsibility                                                                  |
| ------------ | ------------------------------------------------------------------------------- |
| `Portal`     | Moves popup/modal layers outside local DOM stacking context.                    |
| `Positioner` | Calculates anchored placement, collision handling, offsets, and sticky logic.   |
| `Viewport`   | Provides controlled clipping/scroll area for content and transition containers. |
| `Backdrop`   | Optional overlay for modal separation and pointer-event shielding.              |
| `Arrow`      | Optional anchored pointer between trigger and popup surface.                    |

In moduix wrappers (`PopoverContent`, `SelectContent`, `DialogContent`, `DrawerContent`, and
similar), these slots are rendered internally for the default path. When a component exports its own
service slots, prefer the high-level wrapper first and drop to manual composition only when you need
control over portal behavior, overlay presence, or DOM structure.

## Public Customization Contract

Use this contract consistently when customizing a component:

| Concern                          | API surface                                          |
| -------------------------------- | ---------------------------------------------------- |
| Main visible surface             | `className`                                          |
| Common wrapper defaults          | high-level content wrapper (`DialogContent`)         |
| Deep structure / portal behavior | exported low-level parts or component-specific props |
| Optional internal UI             | composition first, component-specific props second   |

Prefer the simplest public path the component gives you. For `Dialog`, that means `DialogContent`
for the common case and `DialogPortal` / `DialogBackdrop` / `DialogViewport` / `DialogPopup` only
when you intentionally need to own that structure. The wrapper can still carry small behavior sugar
when it clearly matches the primitive contract, such as `DialogContent` automatically adapting to
non-modal pointer behavior from `Dialog`.

## When To Compose Manually

Compose primitives manually only when you intentionally need behavior outside the moduix wrapper
contract (for example a custom library-level abstraction or explicit viewport behavior). For regular
product screens, prefer wrapper parts to keep behavior, accessibility, and styling consistent.

## Wrapper Example (Dialog)

If your product has its own design system package, you can create a thin wrapper over moduix and
keep a stable API for app teams.

```tsx
import type { ComponentProps } from 'react';
import { clsx } from 'clsx';
import {
  Dialog as DialogBase,
  DialogClose as DialogCloseBase,
  DialogBody as DialogBodyBase,
  DialogCloseIcon as DialogCloseIconBase,
  DialogContent as DialogContentBase,
  DialogDescription as DialogDescriptionBase,
  DialogFooter as DialogFooterBase,
  DialogHeader as DialogHeaderBase,
  DialogTitle as DialogTitleBase,
  DialogTrigger as DialogTriggerBase,
} from 'moduix';
import styles from './AppDialog.module.css';

type DialogProps = ComponentProps<typeof DialogBase>;
type DialogContentProps = ComponentProps<typeof DialogContentBase>;

export function Dialog(props: DialogProps) {
  return <DialogBase {...props} />;
}

export function DialogTrigger(props: ComponentProps<typeof DialogTriggerBase>) {
  return <DialogTriggerBase {...props} />;
}

export function DialogContent({ className, children, ...props }: DialogContentProps) {
  return (
    <DialogContentBase className={clsx(styles.popup, className)} {...props}>
      {children}
    </DialogContentBase>
  );
}

export function DialogHeader({ className, ...props }: ComponentProps<typeof DialogHeaderBase>) {
  return <DialogHeaderBase className={clsx(styles.header, className)} {...props} />;
}

export function DialogBody({ className, ...props }: ComponentProps<typeof DialogBodyBase>) {
  return <DialogBodyBase className={clsx(styles.body, className)} {...props} />;
}

export function DialogFooter({ className, ...props }: ComponentProps<typeof DialogFooterBase>) {
  return <DialogFooterBase className={clsx(styles.footer, className)} {...props} />;
}

export function DialogTitle({ className, ...props }: ComponentProps<typeof DialogTitleBase>) {
  return <DialogTitleBase className={clsx(styles.title, className)} {...props} />;
}

export function DialogDescription({
  className,
  ...props
}: ComponentProps<typeof DialogDescriptionBase>) {
  return <DialogDescriptionBase className={clsx(styles.description, className)} {...props} />;
}

export function DialogClose({ className, ...props }: ComponentProps<typeof DialogCloseBase>) {
  return <DialogCloseBase className={clsx(styles.close, className)} {...props} />;
}

export function DialogCloseIcon({
  className,
  ...props
}: ComponentProps<typeof DialogCloseIconBase>) {
  return <DialogCloseIconBase className={clsx(styles.closeIcon, className)} {...props} />;
}
```

This pattern gives teams one product-level import while preserving moduix composition, accessibility,
and escape hatches. If your product wrapper needs to own the full overlay structure, start from the
low-level parts on that component page instead of rebuilding the contract from scratch.
