# Composition Patterns

moduix keeps Ark composition visible. Instead of folding behavior into one large prop surface, most
components expose the stateful root and the parts that own semantics, layout, and styling.

That gives you a predictable customization ladder:

- use the shipped parts directly
- re-export them behind product-specific names
- wrap only the parts that need repeated product styling or layout

These patterns stay the same whether you consume `@moduix/react` or install source through the
registry.

## Start from the documented part tree

Start with the component page's anatomy before creating wrappers. Use `className`, CSS variables,
`data-slot`, and Ark state attributes on the parts that actually own those responsibilities.

Root-only exports such as `Popover`, `Tooltip`, and `Select` are the short form of `*.Root`.
Popup-like and dialog-like roots portal their overlay subtree automatically, so the structure stays
visible without adding a separate `Portal` wrapper.

For example:

```text
Dialog / Dialog.Root
├─ Dialog.Trigger
├─ Overlay subtree (automatically portalled)
│  ├─ Dialog.Backdrop
│  └─ Dialog.Positioner
│     └─ Dialog.Content
│        ├─ Dialog.Title
│        ├─ Dialog.Description
│        └─ Dialog.CloseTrigger
```

Use `portalled={false}` for inline rendering and `portalRef` for a custom container when the
component supports those props. Use `Dialog.RootProvider` as the alternative state entry point when
the state object lives outside the rendered subtree.

## What the structural parts are for

| Part         | Responsibility                                                                 |
| ------------ | ------------------------------------------------------------------------------ |
| `Backdrop`   | Provides modal separation and outside-interaction targeting.                   |
| `Positioner` | Owns viewport or anchored placement and scrolling behavior.                    |
| `Content`    | Renders the interactive surface and receives state attributes.                 |
| `Arrow`      | Connects anchored popup content to its trigger when the primitive supports it. |

Do not hide these nodes inside product wrappers unless your product contract intentionally removes
their override points.

## Use `asChild` when another element should own the DOM node

Use `asChild` when a moduix part should attach its behavior to an existing semantic element.

```tsx
import { Button, Popover } from '@moduix/react';

export function FiltersPopover() {
  return (
    <Popover positioning={{ gutter: 8 }}>
      <Popover.Trigger asChild>
        <Button variant="outline">Filters</Button>
      </Popover.Trigger>
      <Popover.Positioner>
        <Popover.Content>Choose one or more filters.</Popover.Content>
      </Popover.Positioner>
    </Popover>
  );
}
```

Keep a single semantic child that can receive Ark props. If the child component swallows refs,
event handlers, or attributes, `asChild` will not behave correctly.

## Share `ids` when composed parts should read as one control

When two primitives must point at the same trigger or description node, share IDs instead of
hand-writing aria relationships.

```tsx
import { Avatar, Tooltip } from '@moduix/react';
import { useId } from 'react';

export function AssigneeTooltip() {
  const id = useId();

  return (
    <Tooltip.Root ids={{ trigger: id }}>
      <Tooltip.Trigger asChild>
        <Avatar.Root ids={{ root: id }}>
          <Avatar.Image src="https://bit.ly/sage-adebayo" alt="Segun Adebayo" />
          <Avatar.Fallback>SA</Avatar.Fallback>
        </Avatar.Root>
      </Tooltip.Trigger>
      <Tooltip.Positioner>
        <Tooltip.Content>Segun Adebayo is online</Tooltip.Content>
      </Tooltip.Positioner>
    </Tooltip.Root>
  );
}
```

This keeps the accessibility wiring in Ark instead of duplicating `aria-*` attributes by hand.

## Re-export the product API first

If your product wants shorter imports or a branded surface, start with re-exports before writing
new abstractions. Re-exporting keeps the original behavior intact while giving your application a
stable local API.

```tsx
import { Dialog as DialogBase } from '@moduix/react';

export const Dialog = DialogBase.Root;
export const DialogTrigger = DialogBase.Trigger;
export const DialogBackdrop = DialogBase.Backdrop;
export const DialogPositioner = DialogBase.Positioner;
export const DialogContent = DialogBase.Content;
export const DialogTitle = DialogBase.Title;
export const DialogDescription = DialogBase.Description;
export const DialogCloseTrigger = DialogBase.CloseTrigger;
```

This is the lowest-risk customization path:

- consumers import from your product layer instead of directly from `moduix`
- Ark behavior stays untouched
- moving from package usage to copy-owned usage stays straightforward
- wrappers can be added later only where they are actually needed

## Add wrappers only on the parts that need product styling

When design requirements go beyond a few utility classes, wrap only the parts that need custom
styling. Keep the structure visible, and forward the rest of the props unchanged.

}

## Move state outside the tree only when ownership changes

Keep Ark detail objects intact. For example, dialog open state still uses
`onOpenChange(details)` and reads `details.open`.

Use component state helpers only when state needs to live outside the rendered subtree:

- `Component.Context` for one-off inline reads
- `useComponentContext()` inside descendants that already live under the root
- `useComponent()` plus `Component.RootProvider` when state is created outside the rendered tree

Do not render `Root` and `RootProvider` for the same state instance.

## A simple rule for choosing the right level

- Only need usage? Stay on the shipped parts.
- Need product import paths? Re-export first.
- Need repeated product visuals? Wrap the specific parts that own those visuals.
- Need different behavior? Drop to the explicit part tree instead of inventing new top-level props.
