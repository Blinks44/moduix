# Composition Patterns

moduix is built around explicit parts. Instead of hiding behavior behind one large component with a
dozen styling props, most multi-part components keep their structure visible with APIs such as
`Dialog.Root`, `Dialog.Trigger`, and `Dialog.Content`.

That gives you three useful levels of control:

- use the package directly and style the shipped parts
- re-export the parts behind product-specific names
- wrap selected parts with your own classes, layout, or design tokens

These patterns are the same whether you consume the package or install source through the registry.

## Start from the documented part tree

Start with the component's documented part tree. Use `className`, CSS variables, `data-slot`, and
Ark state attributes for styling. Use `asChild` when one semantic child must own the rendered node.

Overlay components keep structural parts explicit:

```text
Root
├─ Trigger
└─ Portal
   ├─ Backdrop
   └─ Positioner
      └─ Content
```

`Portal`, `Backdrop`, `Positioner`, and `Content` remain visible so applications can independently
configure mount targets, overlays, scrolling, layout, and surface styling.

## What the structural parts are for

| Part         | Responsibility                                                                 |
| ------------ | ------------------------------------------------------------------------------ |
| `Portal`     | Moves an overlay outside local stacking contexts.                              |
| `Backdrop`   | Provides modal separation and outside-interaction targeting.                   |
| `Positioner` | Owns viewport or anchored placement and scrolling behavior.                    |
| `Content`    | Renders the interactive surface and receives state attributes.                 |
| `Arrow`      | Connects anchored popup content to its trigger when the primitive supports it. |

Do not hide these nodes inside product wrappers unless the product contract intentionally removes
their override points.

## Keep state and callbacks close to the primitive

Keep Ark callback detail objects intact. For example, dialog open state uses
`onOpenChange(details)` and reads `details.open`.

Use component context for state access:

- `Component.Context` for an inline render prop.
- `useComponentContext()` inside reusable descendants.
- `useComponent()` with `Component.RootProvider` when state is owned outside the rendered tree.

Do not render `Root` and `RootProvider` for the same state instance.

## Re-export the product API first

If your product wants shorter imports or a more branded surface, start with re-exports before
writing new abstractions. Re-exporting keeps the original behavior intact while giving your
application a stable local API.

```tsx
import type { ComponentProps } from 'react';
import { Dialog as DialogBase } from '@moduix/react';

export const Dialog = DialogBase.Root;
export const DialogTrigger = DialogBase.Trigger;
export const DialogBackdrop = DialogBase.Backdrop;
export const DialogPositioner = DialogBase.Positioner;
export const DialogContent = DialogBase.Content;
export const DialogTitle = DialogBase.Title;
export const DialogDescription = DialogBase.Description;
export const DialogCloseTrigger = DialogBase.CloseTrigger;

export function ProductDialog(props: ComponentProps<typeof DialogBase.Root>) {
  return <DialogBase.Root {...props} />;
}
```

This is the lowest-risk customization path:

- consumers import from your product layer instead of directly from `moduix`
- Ark behavior stays untouched
- moving from package usage to copy-owned usage stays straightforward
- wrappers can be added later only where they are actually needed

## Add wrappers only on the parts that need product styling

When design requirements go beyond a few utility classes, wrap only the parts that need custom
styling. Keep the structure visible, and forward the rest of the props unchanged.

import type { ComponentProps, ReactNode } from 'react';
import { clsx } from 'clsx';
import { Dialog as DialogBase } from '@moduix/react';
import styles from './ProductDialog.module.css';

export const Dialog = DialogBase.Root;
export const DialogTrigger = DialogBase.Trigger;

export function DialogBackdrop({
  className,
  ...props
}: ComponentProps<typeof DialogBase.Backdrop>) {
  return <DialogBase.Backdrop className={clsx(styles.backdrop, className)} {...props} />;
}

export function DialogPositioner(props: ComponentProps<typeof DialogBase.Positioner>) {
  return <DialogBase.Positioner {...props} />;
}

export function DialogContent({ className, ...props }: ComponentProps<typeof DialogBase.Content>) {
  return <DialogBase.Content className={clsx(styles.content, className)} {...props} />;
}

export function ProductDialog({ children }: { children: ReactNode }) {
  return (
    <DialogBase.Root>
      <DialogBase.Trigger>Open</DialogBase.Trigger>
      <DialogBackdrop />
      <DialogBase.Positioner>
        <DialogContent>{children}</DialogContent>
      </DialogBase.Positioner>
    </DialogBase.Root>
  );
}

This is the practical replacement for the old repeated `Custom Styling` examples:

- keep one documented structural example
- show where classes belong
- leave component-specific visual decisions in the product codebase

## A simple rule for choosing the right level

- Only need usage? Stay on the shipped parts.
- Need product import paths? Re-export first.
- Need repeated product visuals? Wrap the specific parts that own those visuals.
- Need different behavior? Drop to the explicit part tree instead of inventing new top-level props.
