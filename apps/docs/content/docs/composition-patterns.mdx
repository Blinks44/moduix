# Composition Patterns

moduix components expose typed parts that stay close to their underlying primitive contracts.
Multi-part Ark UI components use namespace APIs such as `Dialog.Root`, `Dialog.Trigger`, and
`Dialog.Content`.

These patterns are the same whether you consume the package or install source through the registry.

## Keep the primitive structure visible

Start with the component's documented part tree. Use `className`, CSS variables, `data-slot`, and
Ark state attributes for styling. Use `asChild` when one semantic child must own the rendered node.

## Infrastructure parts

| Part         | Responsibility                                                                 |
| ------------ | ------------------------------------------------------------------------------ |
| `Portal`     | Moves an overlay outside local stacking contexts.                              |
| `Backdrop`   | Provides modal separation and outside-interaction targeting.                   |
| `Positioner` | Owns viewport or anchored placement and scrolling behavior.                    |
| `Content`    | Renders the interactive surface and receives state attributes.                 |
| `Arrow`      | Connects anchored popup content to its trigger when the primitive supports it. |

Do not hide these nodes inside product wrappers unless the product contract intentionally removes
their override points.

## State and callbacks

Keep Ark callback detail objects intact. For example, dialog open state uses
`onOpenChange(details)` and reads `details.open`.

Use component context for state access:

- `Component.Context` for an inline render prop.
- `useComponentContext()` inside reusable descendants.
- `useComponent()` with `Component.RootProvider` when state is owned outside the rendered tree.

Do not render `Root` and `RootProvider` for the same state instance.

## Product wrapper example

A product design system can wrap moduix parts without replacing the Ark composition model:

```tsx
import type { ComponentProps } from 'react';
import { clsx } from 'clsx';
import { Dialog as DialogBase, Portal } from 'moduix';
import styles from './AppDialog.module.css';

export const Dialog = DialogBase.Root;

export function DialogTrigger(props: ComponentProps<typeof DialogBase.Trigger>) {
  return <DialogBase.Trigger {...props} />;
}

export function DialogBackdrop({
  className,
  ...props
}: ComponentProps<typeof DialogBase.Backdrop>) {
  return <DialogBase.Backdrop className={clsx(styles.backdrop, className)} {...props} />;
}

export function DialogPositioner({
  className,
  ...props
}: ComponentProps<typeof DialogBase.Positioner>) {
  return <DialogBase.Positioner className={clsx(styles.positioner, className)} {...props} />;
}

export function DialogContent({ className, ...props }: ComponentProps<typeof DialogBase.Content>) {
  return <DialogBase.Content className={clsx(styles.content, className)} {...props} />;
}

export function AppDialog({ children }: { children: React.ReactNode }) {
  return (
    <DialogBase.Root>
      <DialogTrigger>Open</DialogTrigger>
      <Portal>
        <DialogBackdrop />
        <DialogPositioner>
          <DialogContent>{children}</DialogContent>
        </DialogPositioner>
      </Portal>
    </DialogBase.Root>
  );
}
```

This keeps product styling local while preserving Ark focus management, accessibility, lifecycle,
state attributes, and structural override points.
