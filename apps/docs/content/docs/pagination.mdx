## Basic

Start with `Pagination` and `usePagination` when the page window should be calculated for you and
page changes stay in local UI state.

<Preview>
  <PaginationExample />

  <Preview.Code>
    {`
      import {
        Pagination,
        PaginationContent,
        PaginationEllipsis,
        PaginationItem,
        PaginationLink,
        PaginationNext,
        PaginationPrevious,
        usePagination,
      } from "moduix";
      import { useState } from "react";

      export function PaginationDemo() {
        const [page, setPage] = useState(5);
        const pagination = usePagination({ count: 10, page });

        return (
          <Pagination aria-label="Search results pages">
            <PaginationContent>
              <PaginationItem>
                <PaginationPrevious
                  aria-disabled={!pagination.canPreviousPage || undefined}
                  onClick={() => {
                    if (pagination.canPreviousPage) {
                      setPage(pagination.previousPage);
                    }
                  }}
                />
              </PaginationItem>
              {pagination.items.map((item, index) => (
                <PaginationItem key={String(item) + "-" + index}>
                  {typeof item !== "number" ? (
                    <PaginationEllipsis />
                  ) : (
                    <PaginationLink
                      isActive={item === pagination.page}
                      onClick={() => setPage(item)}
                    >
                      {item}
                    </PaginationLink>
                  )}
                </PaginationItem>
              ))}
              <PaginationItem>
                <PaginationNext
                  aria-disabled={!pagination.canNextPage || undefined}
                  onClick={() => {
                    if (pagination.canNextPage) {
                      setPage(pagination.nextPage);
                    }
                  }}
                />
              </PaginationItem>
            </PaginationContent>
          </Pagination>
        );
      }
    `}

  </Preview.Code>

## Anatomy

The public structure stays explicit:

```text
Pagination
└─ PaginationContent
   ├─ PaginationItem
   │  └─ PaginationPrevious
   ├─ PaginationItem
   │  └─ PaginationLink
   ├─ PaginationItem
   │  └─ PaginationEllipsis
   └─ PaginationItem
      └─ PaginationNext
```

| Part                 | Role                                                            |
| -------------------- | --------------------------------------------------------------- |
| `Pagination`         | Navigation landmark.                                            |
| `PaginationContent`  | Toolbar-like root for keyboard navigation and item arrangement. |
| `PaginationItem`     | Lightweight layout wrapper for one pagination element.          |
| `PaginationLink`     | Page control with active styling via `isActive`.                |
| `PaginationPrevious` | Previous-page shortcut with an icon-only default control.       |
| `PaginationNext`     | Next-page shortcut with an icon-only default control.           |
| `PaginationEllipsis` | Non-interactive collapsed-range marker.                         |

## Composition

### When to stay high-level

Stay with `Pagination`, `PaginationContent`, and `usePagination` for most paginated lists, search
results, and tables.

`usePagination` only calculates the visible range. It does not own routing, URL state, fetching, or
table state. Keep those concerns in application code and map them into `PaginationLink`,
`PaginationPrevious`, and `PaginationNext`.

When `href` and `render` are omitted, `PaginationLink`, `PaginationPrevious`, and `PaginationNext`
render native `button` elements automatically. Use that path for local state and client-side
interactions.

### When to go custom

Skip `usePagination` and compose the visible items manually when the surrounding layout already
knows the exact page window, when the range is intentionally short, or when routing owns the page
math elsewhere.

Use `href="/items?page=4"` for plain anchors and `render={<Link />}` when the app standardizes on a
router-specific link component.

### Default props

| Prop            | Default      | Values / Notes                                                    |
| --------------- | ------------ | ----------------------------------------------------------------- |
| `aria-label`    | `Pagination` | Applied on `Pagination` when no explicit label is passed          |
| `siblingCount`  | `1`          | `usePagination` pages shown on each side of the current page      |
| `boundaryCount` | `1`          | `usePagination` pages always shown at the start and end           |
| `isActive`      | `false`      | Sets `aria-current="page"` and active styling on `PaginationLink` |

### Behavioral notes

- `PaginationContent` uses a roving-focus toolbar-style contract for keyboard navigation.
- `PaginationPrevious` and `PaginationNext` add accessible labels automatically when they stay
  icon-only.
- Disabled styling applies through `[aria-disabled='true']`, `[data-disabled]`, and `[disabled]`.
- `PaginationItem` is only a layout wrapper. If surrounding list semantics matter, own that markup
  in app code.

### Router Links

Use `href` for plain anchors and `render={<Link />}` when the app standardizes on a router link
component.

```tsx
import {
  Pagination,
  PaginationContent,
  PaginationEllipsis,
  PaginationItem,
  PaginationLink,
  PaginationNext,
  PaginationPrevious,
  usePagination,
} from 'moduix';
import Link from 'next/link';
import { usePathname, useSearchParams } from 'next/navigation';

export function PaginationLinks({ pageCount }: { pageCount: number }) {
  const pathname = usePathname();
  const searchParams = useSearchParams();
  const page = Number(searchParams.get('page') ?? '1');
  const pagination = usePagination({ count: pageCount, page });

  const createHref = (nextPage: number) => {
    const params = new URLSearchParams(searchParams);
    params.set('page', String(nextPage));
    return `${pathname}?${params.toString()}`;
  };

  return (
    <Pagination aria-label="Articles pages">
      <PaginationContent>
        <PaginationItem>
          <PaginationPrevious
            render={<Link href={createHref(pagination.previousPage)} />}
            aria-disabled={!pagination.canPreviousPage || undefined}
          />
        </PaginationItem>
        {pagination.items.map((item, index) => (
          <PaginationItem key={String(item) + '-' + index}>
            {typeof item !== 'number' ? (
              <PaginationEllipsis />
            ) : (
              <PaginationLink
                render={<Link href={createHref(item)} />}
                isActive={item === pagination.page}
              >
                {item}
              </PaginationLink>
            )}
          </PaginationItem>
        ))}
        <PaginationItem>
          <PaginationNext
            render={<Link href={createHref(pagination.nextPage)} />}
            aria-disabled={!pagination.canNextPage || undefined}
          />
        </PaginationItem>
      </PaginationContent>
    </Pagination>
  );
}
```

Keep `createHref` close to the router integration so page math, current search params, and URL
shape stay owned by the app.

## Styling

Style `Pagination` through `className`, `data-slot="pagination-*"` hooks, and the public
`--pagination-*` CSS variables shown below.

### Styling hooks

- `data-slot="pagination-root"` for the outer navigation landmark
- `data-slot="pagination-content"` for toolbar layout and gap overrides
- `data-slot="pagination-link"` for page, previous, and next controls
- `data-slot="pagination-ellipsis"` for collapsed-range marker styling
- `aria-current="page"` and `isActive` for active page state targeting
- `[aria-disabled='true']`, `[data-disabled]`, and `[disabled]` for disabled visual states
