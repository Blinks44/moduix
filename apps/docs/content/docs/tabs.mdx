## Choosing the right component

Use `Tabs` when users switch between mutually exclusive views that share one page region.

- Choose **`Tabs`** when one active option should reveal one active panel.
- Choose **`Accordion`** when sections should stay stacked and expand independently.
- Choose **`ToggleGroup`** when the control changes state or mode, but does not own a panel region.

## Basic

Start with `Tabs`, `TabsList`, `TabsTab`, and `TabsPanel`. This is the recommended path for
settings screens, dashboard panes, and compact view switchers.

## Anatomy

The normal consumer path is short. `TabsList` renders the default indicator for you, so you only
add `TabsIndicator` when you need direct control over indicator props or styling.

```text
Tabs
├─ TabsList
│  ├─ TabsTab[value]
│  ├─ TabsTab[value]
│  └─ TabsIndicator (optional explicit override)
└─ TabsPanel[value]
```

| Part            | Role                                                                                  |
| --------------- | ------------------------------------------------------------------------------------- |
| `Tabs`          | Root state, selected value, orientation, and moduix `variant`.                        |
| `TabsList`      | Groups tabs, owns roving-focus list behavior, and auto-renders the default indicator. |
| `TabsTab`       | Interactive tab button. Its `value` must match a `TabsPanel` value.                   |
| `TabsPanel`     | Content region for the active tab. Use `keepMounted` when hidden DOM must persist.    |
| `TabsIndicator` | Optional escape hatch for custom indicator props or styling.                          |

## Composition

### When to stay high-level

Stay with `Tabs`, `TabsList`, `TabsTab`, and `TabsPanel` for most layouts. `Tabs` owns selected
state, `variant`, and `orientation`. `TabsList` already includes the default indicator, so the
common path stays small.

### When to go custom

Add an explicit `TabsIndicator` inside `TabsList` when you need direct indicator control, such as a
custom `className` or `renderBeforeHydration`. An explicit indicator replaces the built-in one for
that list.

Use `render` on `TabsTab` together with `nativeButton={false}` when tabs should render as links or
other non-button elements.

### Default props

| Prop                       | Default      | Values / Notes                           |
| -------------------------- | ------------ | ---------------------------------------- |
| `Tabs.variant`             | `default`    | `default`, `line`                        |
| `Tabs.orientation`         | `horizontal` | `horizontal`, `vertical`                 |
| `TabsList.loopFocus`       | `true`       | Roving focus wraps around the list       |
| `TabsList.activateOnFocus` | `false`      | Focus moves without selecting by default |

### Behavioral notes

- Use `onValueChange` when the selected panel must coordinate with React state.
- Pass an explicit enabled `defaultValue` or controlled `value` in SSR-sensitive layouts where the
  first tab might be disabled.
- Use `keepMounted` on `TabsPanel` when inactive content must preserve local state, measurements, or
  form inputs.
- `Tabs` does not impose width. Let the surrounding layout or your own `className` decide how wide
  the control should be.

### Controlled Value

Control the active tab from React state when the selected panel needs to coordinate with other UI.

<Preview>
  <ControlledTabsExample />

  <Preview.Code>
    {`
      import {
        Tabs,
        TabsList,
        TabsPanel,
        TabsTab,
      } from "moduix";
      import { useState } from "react";

      export function ControlledTabsDemo() {
        const [value, setValue] = useState("projects");

        return (
          <Tabs value={value} onValueChange={setValue}>
            <TabsList>
              {items.map((item) => (
                <TabsTab key={item.value} value={item.value}>
                  {item.title}
                </TabsTab>
              ))}

            <TabsList>

              <TabsPanel key={item.value} value={item.value}>

              </TabsPanel>
            ))}
          </Tabs>

## Styling

Style `Tabs` through `className`, stable `data-slot="tabs-*"` hooks, `data-variant`, and the
public `--tabs-*` CSS variables shown below.

### Styling hooks

Use the part that owns the visual concern:

- `Tabs` for layout width, overall gap, and variant-level overrides
- `TabsList` for list chrome, borders, background, and indicator container spacing
- `TabsTab` for label layout, icon spacing, active state, and focus styling
- `TabsIndicator` for custom indicator color, size, and transition behavior
- `TabsPanel` for panel surface, padding, and mounted-panel focus styling

The main public hooks include `data-slot="tabs-root"`, `data-slot="tabs-list"`,
`data-slot="tabs-tab"`, `data-slot="tabs-indicator"`, and `data-slot="tabs-panel"`.

Useful state hooks from the shipped primitive include `data-variant`, `data-orientation`,
`data-active`, and `data-disabled`.
