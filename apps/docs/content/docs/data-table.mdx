# Data Table

`Data Table` is a composition pattern, not a component exported by moduix. Use the semantic and
styled `Table` parts for markup, then let TanStack Table own the data-grid state. Each product has
different columns, filters, row actions, and server contracts, so keep that behavior close to the
feature that needs it.

## Installation

Install TanStack Table in the application that owns the data-grid behavior:

```bash
npm install @tanstack/react-table
```

`Table` already comes from `@moduix/react`. It deliberately does not include TanStack Table, so
applications that only need a semantic table do not pay for data-grid logic.

## Basic

Define typed `ColumnDef` entries, create a table instance with `getCoreRowModel()`, and map its
header groups and visible cells into the moduix table anatomy. Use `flexRender` for headers and
cells: column definitions may contain plain values or React render functions.

Keep the columns alongside the feature's data type. This is the natural place for formatting,
numeric alignment, row actions, and business-specific filter definitions.

## Stable column widths

Use native `Table.ColumnGroup` and `Table.Column htmlWidth` for columns whose width should stay
stable. Combine them with `table-layout: fixed` on the table root: the remaining column takes the
available space, while a changing sort icon cannot resize its column.

For a sortable header, render a real button inside `Table.ColumnHeader`. The example uses Lucide
icons with a consistent size and sets `aria-sort="none" | "ascending" | "descending"` on sortable
column headers. The selection column uses moduix `Checkbox` with the built-in indeterminate state
for the current page.

## Add features deliberately

TanStack Table is headless. Choose the state and row models for the behavior that belongs to this
specific data view instead of turning every grid feature on by default.

| Need                           | Add to this table                                                                                                                            |
| ------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------- |
| Row actions                    | Render a `Menu` in an actions column; action handlers stay with the product feature.                                                         |
| Row selection                  | Control `rowSelection`, configure `getRowId` with a durable record ID, and render moduix `Checkbox` controls in a selection column.          |
| Column visibility              | Control `columnVisibility` and expose `column.toggleVisibility()` through a `Menu`.                                                          |
| Server filtering or pagination | Keep query state in the route or data layer, use TanStack's `manual*` options, and pass the response rows and page count to the table.       |
| Large datasets                 | Add virtualization to the table body only after measuring; keep the native `table`, `thead`, `tbody`, `tr`, `th`, and `td` semantics intact. |

The TanStack Table [row models guide](https://tanstack.com/table/latest/docs/guide/row-models) lists
the available client-side transforms. Their [row selection guide](https://tanstack.com/table/latest/docs/guide/row-selection)
explains controlled selection state and page-level selection, while the [React adapter](https://tanstack.com/table/latest/docs/framework/react/react-table)
documents `useReactTable`. The moduix [Table](/docs/table) page documents the rendered structure
and its styling hooks.

With `manualPagination`, keep the full `rowSelection` state for actions that span pages. TanStack's
selected-row models can only return records currently supplied to the table, even when the selection
state also contains IDs from other pages.

## Reuse after the pattern stabilizes

If the same product table appears in more than one place, extract its columns, toolbar, or
pagination controls into local feature components. Keep the resulting API product-specific:
`PaymentsTable` is more maintainable than a broad `DataTable` wrapper that attempts to predict every
filter, server response, action, or selection rule.

## Accessibility

Keep the native table parts in the rendered output. Use `Table.ColumnHeader` for headers,
`Table.Cell` for data, and a short caption when the surrounding heading does not identify the data.
Interactive controls inside cells need descriptive accessible names, and sortable headers should
expose their current direction through `aria-sort`. The filter input and selection controls in the
interactive example have explicit accessible names rather than relying on placeholder or icon text.
