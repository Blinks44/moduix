## Anatomy

```text
Avatar.Root
├─ Avatar.Image
└─ Avatar.Fallback
   └─ initials | icon | custom content
```

| Part              | Role                                                     |
| ----------------- | -------------------------------------------------------- |
| `Avatar.Root`     | Shared size, shape, color, and Ark loading state owner.  |
| `Avatar.Image`    | Visible image when the source loads successfully.        |
| `Avatar.Fallback` | Identity fallback while the image is absent or rejected. |

## Composition

### Recommended composition

`Avatar.Root` owns the Ark image-loading lifecycle. Use it for `ids`, `onStatusChange(details)`,
and for the moduix `size` styling shortcut.

Use `Avatar.Image` and `Avatar.Fallback` together for normal profile media. Keep `Avatar.Fallback`
in the tree even when the image is expected to load; Ark swaps visibility through
`data-state="visible" | "hidden"`.

Use `size` on `Avatar.Root` for the standard token sizes, or override `--avatar-size`,
`--avatar-font-size`, and `--avatar-line-height` when dimensions need a custom value.

All exported parts support `asChild`. Use that when another element must own the DOM node. This is
the Ark composition path that replaces the old Base UI `render` contract.

### When to go custom

Use `Avatar.Root asChild` when the avatar itself should be the interactive element. When external
state ownership is needed, keep the Ark `RootProvider` and `useAvatar` model in mind even if the
common consumer path stays with `Avatar.Root`.

For frameworks such as Next.js, keep Ark image visibility semantics intact when composing a custom
image implementation.

### Default props

| Surface       | Prop   | Default | Values / Notes               |
| ------------- | ------ | ------- | ---------------------------- |
| `Avatar.Root` | `size` | `md`    | `xs`, `sm`, `md`, `lg`, `xl` |

### Behavioral notes

- `Avatar.Image` is visible only after a successful image load.
- `Avatar.Fallback` remains visible while the image is missing or has failed.
- `onStatusChange` follows Ark shape and receives `details` (`details.status` contains the current
  loading state).
- `ids` can be passed to `Avatar.Root` when surrounding composition needs stable DOM ids.
- If `Avatar.Root`, `Avatar.Image`, or `Avatar.Fallback` is made polymorphic through `asChild`, the
  child element must keep the right accessible semantics.

### Events

Use `onStatusChange(details)` when surrounding UI should react to the avatar image lifecycle.

  <AvatarStatusExample />

  <Preview.Code>
    {`

      import { useState } from "react";

      export function AvatarStatusDemo() {
        const [status, setStatus] = useState("idle");

        return (
          <div className="flex flex-col items-start gap-3">

## Styling

Use `className` on exported parts for structural styling changes. Prefer Ark state attributes for
load-driven styling and moduix CSS variables for tokens and sizing.

### CSS Properties

<AvatarCssPropertiesPanel />

### Styling hooks

- `data-slot="avatar-root"`, `avatar-image`, and `avatar-fallback` are stable selectors on the
  exported parts.
- Ark also exposes `data-scope="avatar"` and `data-part="root" | "image" | "fallback"` on the
  rendered nodes.
- `Avatar.Image` and `Avatar.Fallback` expose `data-state="visible" | "hidden"` for state-aware
  styling.
- `data-size` is added on `Avatar.Root` when the moduix `size` prop is used.
