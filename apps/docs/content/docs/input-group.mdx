## API Reference

`InputGroup` does not wrap a dedicated Base UI primitive. It is a moduix composition helper built
from the local `Input` and `Button` components, so the public API lives on these exported parts:
`InputGroup`, `InputGroupInput`, `InputGroupAddon`, `InputGroupText`, and `InputGroupButton`.

## Choosing the right component

Use `InputGroup` when one single-line input and its inline affordances should read as one merged
control.

- Choose **`Input`** when the field only needs a plain input.
- Choose **`InputGroup`** when the field needs inline prefixes, suffixes, helper text, or actions.
- Choose **`PasswordInput`** when the grouped action is specifically a built-in visibility toggle.
- Choose **`NumberField`** when the value is numeric and needs parsing, stepping, or formatting.
- Choose **`Textarea`** when the content can span multiple lines.

## Anatomy

The normal consumer path is short:

```text
InputGroup
├─ InputGroupAddon (optional)
├─ InputGroupInput
├─ InputGroupText (optional)
└─ InputGroupButton (optional)
```

| Part               | Role                                                     |
| ------------------ | -------------------------------------------------------- |
| `InputGroup`       | Shared shell for border, radius, focus ring, and sizing. |
| `InputGroupInput`  | The `Input` rendered inside the shell.                   |
| `InputGroupAddon`  | Filled prefix or suffix surface with a separator.        |
| `InputGroupText`   | Plain inline text without the filled addon background.   |
| `InputGroupButton` | Inline action based on `Button`.                         |

## Composition

### When to stay high-level

Stay with one `InputGroupInput` per group, then add addons, helper text, or buttons before or
after it as needed.

Set `size` on `InputGroup` to keep the whole row aligned. `InputGroupInput` and
`InputGroupButton` inherit that size automatically, so the default path stays short.

`InputGroupInput` accepts the same props as `Input`, including controlled values, native input
attributes, and Base UI field validation state.

`InputGroupButton` defaults to `variant="ghost"` and `type="button"`, so inline actions work
inside forms without submitting them accidentally.

### When to go custom

`InputGroup` is already the low-level moduix composition path for merged field shells. There is no
deeper namespaced primitive tree below it.

Switch away from `InputGroup` when the control stops being one input with inline decoration:

- use `Input` for plain fields without extra inline parts
- use `PasswordInput` when the grouped action is just a visibility toggle
- use `NumberField` when the input needs numeric behavior instead of visual grouping

Use `role="group"` with `aria-label` or `aria-labelledby` only for standalone widgets where the
group itself needs an explicit grouped semantic. When the control lives inside `Field`, prefer
`FieldLabel` and keep `InputGroup` purely visual.

### Default props

| Part / Prop                | Default              | Notes                                                     |
| -------------------------- | -------------------- | --------------------------------------------------------- |
| `InputGroup.size`          | `md`                 | Supports `xs`, `sm`, `md`, `lg`, and `xl`.                |
| `InputGroupInput.size`     | inherited from group | Override only when one input should intentionally differ. |
| `InputGroupButton.variant` | `ghost`              | Accepts the same variants as `Button`.                    |
| `InputGroupButton.size`    | inherited from group | Override for icon or mixed-density actions.               |
| `InputGroupButton.type`    | `button`             | Use `submit` or `reset` when the action should do that.   |

### Behavioral notes

- The shell click-to-focus behavior targets the first grouped input, so multiple grouped inputs are
  not a supported composition.
- Shell clicks focus the input for plain primary-button clicks, but skip interactive descendants and
  do nothing when the grouped input is `disabled` or `readOnly`.
- Invalid, disabled, and readonly shell styling is derived from the nested input state.
- For inline edit flows, keep one mounted `InputGroupInput`, switch it between `readOnly` and
  editable modes, and swap only the trailing actions.

### Inline Editing

For single-line inline editing, keep one mounted `InputGroupInput`, switch it between `readOnly`
and editable modes, and swap only the action buttons. If the readonly state should look flatter or
more text-like, override the `--input-group-readonly-*` shell variables instead of styling only the
nested input.

<Preview>
  <InputGroupInlineEditingExample />

  <Preview.Code>
    {`
      import {
        CheckIcon,
        CloseIcon,
        Field,
        FieldLabel,
        InputGroup,
        InputGroupButton,
        InputGroupInput,
        PencilIcon,
      } from "moduix";
      import { useEffect, useRef, useState } from "react";

      export function InlineEditingInputGroupDemo() {
        const [editing, setEditing] = useState(false);
        const [value, setValue] = useState("Workspace display name");
        const [draft, setDraft] = useState(value);
        const inputRef = useRef(null as HTMLInputElement | null);

        useEffect(() => {
          if (editing) {
            inputRef.current?.focus();
          }
        }, [editing]);

        const handleEdit = () => {
          setDraft(value);
          setEditing(true);
        };

        const handleCancel = () => {
          setDraft(value);
          setEditing(false);
        };

        const handleSave = () => {
          setValue(draft);
          setEditing(false);
        };

        return (
          <Field className="input-group-demo-field">
            <FieldLabel>Display name</FieldLabel>
            <InputGroup>
              <InputGroupInput
                ref={inputRef}
                aria-label="Display name"
                readOnly={!editing}
                value={editing ? draft : value}
                onValueChange={setDraft}
              />
              {editing ? (
                <>
                  <InputGroupButton aria-label="Cancel editing" size="icon-md" onClick={handleCancel}>
                    <CloseIcon />
                  </InputGroupButton>
                  <InputGroupButton aria-label="Save changes" size="icon-md" onClick={handleSave}>
                    <CheckIcon />
                  </InputGroupButton>
                </>
              ) : (
                <InputGroupButton aria-label="Edit display name" size="icon-md" onClick={handleEdit}>
                  <PencilIcon />
                </InputGroupButton>
              )}
            </InputGroup>
          </Field>
        );
      }
    `}

  </Preview.Code>

  <Preview.CSS>
    {`
      .input-group-demo-field {
        width: min(16rem, 100%);
      }
    `}

  </Preview.CSS>
</Preview>

### Styling hooks

Use the part that owns the visual concern:

- `InputGroup` for the visible shell, radius, border, and focus ring
- `InputGroupInput` for text alignment and input-specific local tweaks
- `InputGroupAddon` for filled prefix and suffix surfaces
- `InputGroupText` for plain inline helper text
- `InputGroupButton` for grouped actions

The main public hooks are `data-slot="input-group-root"`, `data-slot="input-group-input"`,
`data-slot="input-group-addon"`, `data-slot="input-group-text"`, and
`data-slot="input-group-button"`.

The shell does not mirror child state with its own `data-invalid`, `data-disabled`, or
`data-readonly` attributes. Root invalid, disabled, and readonly styling is derived from the nested
input state.
