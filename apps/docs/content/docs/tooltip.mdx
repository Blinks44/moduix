## Choosing the right component

Use `Tooltip` for short, non-essential labels or hints that help sighted users on hover or focus.

- Choose **`Tooltip`** for compact labels and short contextual hints.
- Choose **`Popover`** when the popup needs richer layout, actions, or interactive content.
- Choose **`PreviewCard`** when the popup should expose a richer preview surface tied to a trigger.

## Basic

Start with `Tooltip`, `TooltipTrigger`, and `TooltipContent`. Keep an accessible name on the
trigger itself: tooltips do not replace `aria-label` or visible text.

## Anatomy

The normal consumer path is short:

```text
Tooltip
├─ TooltipTrigger
└─ TooltipContent
   └─ TooltipPortal
      └─ TooltipPositioner
         └─ TooltipPopup
            └─ TooltipViewport
               └─ content
```

Turn on `showArrow` when the built-in arrow adds value:

```text
Tooltip
├─ TooltipTrigger
└─ TooltipContent
   └─ TooltipPortal
      └─ TooltipPositioner
         └─ TooltipPopup
            ├─ TooltipArrow
            └─ TooltipViewport
               └─ content
```

Use the low-level parts directly when portal placement, popup structure, or advanced positioning
needs to change.

| Part                | Role                                                                                                     |
| ------------------- | -------------------------------------------------------------------------------------------------------- |
| `Tooltip`           | Root state, open lifecycle, controlled props, and optional `handle` linking.                             |
| `TooltipProvider`   | Shared timing config for related tooltips.                                                               |
| `TooltipTrigger`    | Interactive anchor element. Opens the tooltip on hover or focus and should keep its own accessible name. |
| `TooltipContent`    | Default popup wrapper with built-in portal, positioner, popup, optional arrow, and viewport.             |
| `TooltipPortal`     | Mount target for custom portal placement.                                                                |
| `TooltipPositioner` | Floating layout and positioning layer.                                                                   |
| `TooltipPopup`      | Visible tooltip surface.                                                                                 |
| `TooltipArrow`      | Optional default arrow part.                                                                             |
| `TooltipViewport`   | Inner content wrapper that preserves Base UI transition behavior between triggers.                       |

## Composition

### When to stay high-level

Stay with `Tooltip`, `TooltipTrigger`, and `TooltipContent` for most icon buttons, compact labels,
and settings controls.

`Tooltip` keeps the Base UI root behavior intact. Use it for `open`, `defaultOpen`,
`onOpenChange`, `disableHoverablePopup`, `trackCursorAxis`, and shared-handle flows.

`TooltipContent` is the high-level popup wrapper. It accepts popup `className`, supports the shared
positioning props (`side`, `sideOffset`, `align`, `alignOffset`, `arrowPadding`,
`collisionAvoidance`, `collisionBoundary`, `collisionPadding`), and adds one tooltip-specific
convenience:

- `showArrow` for the built-in popup arrow

### When to go custom

Switch to `TooltipPortal`, `TooltipPositioner`, `TooltipPopup`, `TooltipArrow`, and
`TooltipViewport` when you need portal props, custom DOM structure, a custom arrow setup, or less
common positioner props such as `anchor`, `disableAnchorTracking`, `sticky`, or
`positionMethod`.

### Default props

| Prop           | Default | Values / Notes                            |
| -------------- | ------- | ----------------------------------------- |
| `showArrow`    | `false` | Enables the built-in arrow                |
| `sideOffset`   | `8`     | Any number accepted by Base UI positioner |
| `closeOnClick` | `true`  | Default `TooltipTrigger` click behavior   |

### Behavioral notes

- `TooltipContent className` styles the visible popup surface, not the portal, positioner, or
  viewport.
- `TooltipTrigger` applies the moduix trigger styling only when `render` is omitted. With
  `render={<Button />}`, styling belongs to the rendered element.
- `TooltipTrigger disabled` only disables tooltip behavior. If the rendered control itself should
  be disabled, pass `disabled` to that control as well.
- `TooltipViewport` is part of the default wrapper contract. It is not a generic scroll container.
- Use `TooltipProvider` when nearby tooltips should share delay timing and reopen instantly after
  one another.
- Use `createTooltipHandle` when one tooltip needs to follow detached triggers or several triggers
  need to share payload-driven content.

composition.

### Side Control

        Button,
        Tooltip,
        TooltipContent,
        TooltipTrigger,

      import { useState } from "react";
      import styles from "./tooltip-demo.module.css";

      type TooltipSide = (typeof tooltipSides)[number];

      export function SideControlTooltip() {
        const [side, setSide] = useState("top" as TooltipSide);

        return (
          <div className={styles.stack}>
            <div className={styles.sideButtons}>
              {tooltipSides.map((item) => (
                <button
                  key={item}
                  type="button"
                  className={styles.sideButton}

                Hover or focus

            </Tooltip>
          </div>

### Detached Trigger

Use `createTooltipHandle` when the trigger and tooltip content cannot live in the same tree.

<Preview>
  <DetachedTriggerTooltipExample />

  <Preview.Code>
    {`
      import {
        Button,
        Tooltip,
        TooltipContent,
        TooltipTrigger,
        createTooltipHandle,
      } from "moduix";
      import { useMemo } from "react";

      export function DetachedTriggerTooltip() {
        const tooltipHandle = useMemo(() => createTooltipHandle(), []);

        return (
          <>
            <TooltipTrigger
              handle={tooltipHandle}
              render={<Button />}
              aria-label="Detached tooltip"
            >
              Detached trigger
            </TooltipTrigger>
            <Tooltip handle={tooltipHandle}>
              <TooltipContent>Linked with handle.</TooltipContent>
            </Tooltip>
          </>
        );
      }
    `}

  </Preview.Code>
</Preview>

### Multiple Triggers

Share one tooltip between several triggers by linking them with the same handle and rendering the
active trigger payload.

<Preview>
  <MultipleTriggersTooltipExample />

  <Preview.Code>
    {`
      import {
        Button,
        InfoIcon,
        PlusIcon,
        ShareIcon,
        Tooltip,
        TooltipContent,
        TooltipProvider,
        TooltipTrigger,
        createTooltipHandle,
      } from "moduix";
      import { useMemo } from "react";
      import styles from "./tooltip-demo.module.css";

      export function MultipleTriggersTooltip() {
        const tooltipHandle = useMemo(
          () => createTooltipHandle<{ text: string }>(),
          [],
        );

        return (
          <TooltipProvider delay={250}>
            <div className={styles.row}>
              <TooltipTrigger
                aria-label="Create"
                handle={tooltipHandle}
                payload={{ text: "Create" }}
                render={<Button variant="ghost" size="icon-md" />}
              >
                <PlusIcon className={styles.icon} />
              </TooltipTrigger>
              <TooltipTrigger
                aria-label="Share"
                handle={tooltipHandle}
                payload={{ text: "Share" }}
                render={<Button variant="ghost" size="icon-md" />}
              >
                <ShareIcon className={styles.icon} />
              </TooltipTrigger>
              <TooltipTrigger
                aria-label="Details"
                handle={tooltipHandle}
                payload={{ text: "Details" }}
                render={<Button variant="ghost" size="icon-md" />}
              >
                <InfoIcon className={styles.icon} />
              </TooltipTrigger>

              <Tooltip handle={tooltipHandle}>
                {({ payload }) => <TooltipContent>{payload?.text}</TooltipContent>}
              </Tooltip>
            </div>
          </TooltipProvider>
        );
      }
    `}

  </Preview.Code>

  <Preview.CSS>
    {`
      .icon {
        width: 1rem;
        height: 1rem;
      }

      .row {
        display: flex;
        align-items: center;
        justify-content: center;
        flex-wrap: wrap;
        gap: var(--spacing-2);
      }
    `}

  </Preview.CSS>
</Preview>

### Styling hooks

Use the part that owns the visual concern:

- `TooltipTrigger` for trigger chrome, hover/open feedback, and focus styling
- `TooltipContent` or `TooltipPopup` for popup surface overrides
- `TooltipArrow` for arrow size or stroke changes
- `TooltipViewport` for payload-transition layout and inner content behavior

The main public hooks include `data-slot="tooltip-trigger"`, `tooltip-positioner`,
`tooltip-popup`, `tooltip-arrow`, and `tooltip-viewport`.

Base UI also adds state attributes such as `data-popup-open`, `data-trigger-disabled`,
`data-starting-style`, `data-ending-style`, `data-instant`, `data-side`, `data-align`,
`data-current`, `data-previous`, and `data-transitioning` on the relevant parts.
