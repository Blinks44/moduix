## Choosing the right component

Use `Switch` for immediate on/off settings where changing the control should toggle one boolean
state.

- Choose **`Switch`** for settings such as notifications, reminders, or feature flags.
- Choose **`Checkbox`** when the control behaves like a submitted selection rather than an immediate
  setting toggle.
- Choose **`RadioGroup`** when users must choose one option from several mutually exclusive values.

## Anatomy

```text
Switch / Switch.Root
├─ Switch.Control
│  └─ Switch.Thumb
├─ Switch.Label
└─ Switch.HiddenInput
```

| Part                                  | Role                                                                 |
| ------------------------------------- | -------------------------------------------------------------------- |
| `Switch` / `Switch.Root`              | Root label element. Owns checked state, ids, form props, and `size`. |
| `Switch.Control`                      | Visual track and focus ring target. Auto-renders `Switch.Thumb`.     |
| `Switch.Thumb`                        | Movable thumb for default visuals or custom thumb content.           |
| `Switch.Label`                        | Ark label text connected to the switch.                              |
| `Switch.HiddenInput`                  | Native input for form submission, reset, validation, and name/value. |
| `Switch.RootProvider`                 | Root label driven by an external `useSwitch()` state instance.       |
| `Switch.Context` / `useSwitchContext` | Inline or child-part access to the Ark switch state.                 |

## Composition

### Recommended composition

Keep the Ark part tree visible. Render `Switch.Control`, `Switch.Label`, and
`Switch.HiddenInput` inside `Switch` / `Switch.Root`.

`Switch.Control` renders the default `Switch.Thumb` when its children are omitted. Compose
`Switch.Thumb` directly when the thumb needs custom content or styling.

### When to go custom

Use `asChild` only when the root must render a custom host element. Ark renders the root as a
`label`; the direct `asChild` element must stay label-compatible so the switch keeps a valid
accessible name.

Use `Switch.Context`, `useSwitchContext`, or `useSwitch` with `Switch.RootProvider` when surrounding
UI needs to read or own the Ark switch state.

### Default props

| Prop        | Default     | Notes                                       |
| ----------- | ----------- | ------------------------------------------- |
| `size`      | `md`        | moduix sugar: `xs`, `sm`, `md`, `lg`, `xl`. |
| `value`     | `on`        | Ark hidden input value for form submission. |
| `children`  | `undefined` | Compose Ark parts explicitly.               |
| `className` | `undefined` | Applied to the Ark root label.              |

### Behavioral notes

- Use `defaultChecked` for uncontrolled mode and `checked` with
  `onCheckedChange(details)` for controlled mode.
- `details.checked` is the next boolean state. The old raw boolean callback is removed.
- Include `Switch.HiddenInput` when the switch participates in native forms or form reset.
- Use `Switch.Context`, `useSwitchContext`, or `useSwitch` with `Switch.RootProvider` for state
  reads instead of wrapper-level callback adapters.
- Use `asChild` for a custom root host. The child must remain a single semantic label-compatible
  element.

### Controlled

Use controlled mode when the switch needs to coordinate with surrounding UI.

<Preview>
  <ControlledSwitchExample />

  <Preview.Code>
    {`

      import { useState } from "react";

      export function ControlledSwitchDemo() {
        const [checked, setChecked] = useState(true);

        return (
          <div className={styles.stack}>
            <Switch checked={checked} onCheckedChange={(details) => setChecked(details.checked)}>
              <Switch.Control />
              <Switch.Label>{checked ? "On" : "Off"}</Switch.Label>
              <Switch.HiddenInput />

### Root Provider

        return (
          <div className={styles.stack}>

## Styling

Style `Switch` through `className`, public `data-slot` hooks, Ark state attributes, and the public
`--switch-*` CSS variables shown below.

### Styling hooks

- `Switch` and `Switch.RootProvider` expose `data-slot="switch-root"` /
  `data-slot="switch-root-provider"` and write moduix `data-size`.
- `Switch.Control` exposes `data-slot="switch-control"` and receives Ark state attributes such as
  `data-state`, `data-focus-visible`, `data-hover`, `data-disabled`, `data-readonly`,
  `data-invalid`, and `data-required`.
- `Switch.Thumb` exposes `data-slot="switch-thumb"` and receives the same Ark state attributes for
  thumb movement and custom content styling.
- `Switch.Label` exposes `data-slot="switch-label"` for typography overrides.
- `Switch.HiddenInput` exposes `data-slot="switch-hidden-input"` for inspection hooks, but it should
  remain visually hidden by Ark.
