## Choosing the right component

Use `Switch` for immediate on/off settings where changing the control should toggle one boolean
state.

- Choose **`Switch`** for settings such as notifications, reminders, or feature flags.
- Choose **`Checkbox`** when the control behaves like a submitted selection rather than an immediate
  setting toggle.
- Choose **`RadioGroup`** when users must choose one option from several mutually exclusive values.

## Basic

## Anatomy

The normal consumer path is short:

```text
SwitchField
├─ Switch
│  └─ SwitchThumb
└─ SwitchLabel
```

Use `SwitchField` when the switch and label can live inside one wrapping `<label>`.

| Part          | Role                                                                 |
| ------------- | -------------------------------------------------------------------- |
| `Switch`      | Interactive root. Handles checked state, form props, and `size`.     |
| `SwitchThumb` | Optional thumb part for custom thumb content or primitive props.     |
| `SwitchField` | Optional wrapping label for the common inline field pattern.         |
| `SwitchLabel` | Optional text wrapper that inherits the switch label styling tokens. |

## Composition

### When to stay high-level

Stay with `Switch`, `SwitchField`, and `SwitchLabel` for most settings panels and boolean
preferences.

`Switch` keeps the underlying primitive root behavior intact. Use it for `checked`, `defaultChecked`,
`onCheckedChange`, `name`, `value`, `uncheckedValue`, `required`, `disabled`, `readOnly`, `form`,
`inputRef`, `nativeButton`, and `render`.

`size` is the only component-specific DX sugar. It scales the track and default thumb without
changing the composition model.

### When to go custom

Render `<Switch />` directly when the default thumb is enough. Compose `SwitchThumb` explicitly
when the thumb needs custom content or thumb-specific primitive props.

Render the root as a native button when the layout needs separate switch and label elements:

```tsx
<div>
  <Switch id="product-updates" nativeButton render={<button />} />
  <label htmlFor="product-updates">Receive product updates</label>
</div>
```

Use the `render` callback when a native-button switch must still stay inside a wrapping label, so
The primitive can keep the hidden input outside the label.

### Default props

| Prop        | Default     | Values / Notes                              |
| ----------- | ----------- | ------------------------------------------- |
| `size`      | `md`        | `xs`, `sm`, `md`, `lg`, `xl`                |
| `className` | `undefined` | Root class name or primitive state callback |
| `children`  | auto thumb  | Compose `SwitchThumb` manually when needed  |

### Behavioral notes

- Use `defaultChecked` for uncontrolled mode and `checked` with `onCheckedChange` for controlled
  mode.
- Every switch needs an accessible name. The default recommendation is
  `SwitchField` + `SwitchLabel`.
- Use `readOnly` when the value should stay visible but not change. Use `disabled` when the
  control is unavailable.
- Use `Field` when the switch needs form labeling, validation, and submission wiring.

## Styling

Style `Switch` through `className`, the public `data-slot` hooks, primitive state attributes, and
the public `--switch-*` CSS variables shown below.

### CSS Properties

### Styling hooks

- `Switch` exposes `data-slot="switch-root"` and writes `data-size` for the `size` presets.
- `SwitchThumb` exposes `data-slot="switch-thumb"` and receives primitive state attributes such as
  `data-checked`, `data-unchecked`, `data-disabled`, and `data-readonly`.
- `SwitchField` and `SwitchLabel` expose `data-slot="switch-field"` and `data-slot="switch-label"`
  for layout and typography overrides.
