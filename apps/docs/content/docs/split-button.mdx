## API Reference

`SplitButton` is a local moduix composition built from the moduix `Button` and `Menu` wrappers.
SplitButton is a local moduix composition built on Ark-backed Menu and Button parts.

## Basic

Start with `SplitButton`, `SplitButtonAction`, `SplitButtonTrigger`, and `SplitButtonContent`.
This is the recommended path when one action is the default path and the rest belong in a compact
dropdown.

## Anatomy

The normal consumer path is short:

```text
SplitButton
├─ SplitButtonAction
├─ SplitButtonTrigger
└─ SplitButtonContent
   ├─ Menu.Item
   ├─ Menu.Separator
   └─ other shared Menu parts
```

| Part                 | Role                                                                  |
| -------------------- | --------------------------------------------------------------------- |
| `SplitButton`        | Menu root behavior plus grouped visual wrapper.                       |
| `SplitButtonAction`  | Primary action button. Inherits root `variant` and `size` by default. |
| `SplitButtonTrigger` | Secondary trigger button. Defaults to a chevron icon.                 |
| `SplitButtonContent` | Default popup wrapper around `Menu.Content`.                          |

## Composition

### When to stay high-level

Stay with `SplitButtonAction`, `SplitButtonTrigger`, and `SplitButtonContent` for most grouped
action menus.

- Keep the primary action explicit on `SplitButtonAction`.
- Keep alternate actions on shared menu parts such as `Menu.Item`, `Menu.Separator`, `Menu.ItemGroup`, and
  `Menu.CheckboxItem`.
- Let the root `variant` and `size` flow to both grouped buttons unless one side needs an override.

### When to go custom

`SplitButton` does not fork the menu row API. The custom path is still plain `Button` and `Menu`
composition:

- Override `size` or `variant` on `SplitButtonAction` or `SplitButtonTrigger` when one side needs a
  different visual treatment.
- Pass custom trigger `children` when the default chevron is not enough.
- Use `asChild` on `SplitButtonAction` when the primary side should render as a link or another
  compatible element.
- Configure popup placement on `SplitButton` with Ark `positioning`; `SplitButtonContent` owns only the default portal, positioner, and content structure.

### Default props

| Part                 | Prop         | Default     | Notes                                                            |
| -------------------- | ------------ | ----------- | ---------------------------------------------------------------- |
| `SplitButton`        | `variant`    | `'default'` | Shared default for `SplitButtonAction` and `SplitButtonTrigger`. |
| `SplitButton`        | `size`       | `'md'`      | Shared default for `SplitButtonAction` and `SplitButtonTrigger`. |
| `SplitButtonContent` | `align`      | `'end'`     | Keeps the popup aligned with the trigger edge.                   |
| `SplitButtonContent` | `sideOffset` | `4`         | Small gap between the trigger and popup surface.                 |
| `SplitButtonTrigger` | `aria-label` | auto        | Defaults to `'More actions'` when it stays icon-only.            |

### Behavioral notes

- `SplitButton` forwards Ark menu root props such as `open`, `defaultOpen`, `onOpenChange(details)`, `closeOnSelect`, `positioning`, and `ids`.
- `SplitButtonTrigger` intentionally owns the rendered button and does not expose `render`.
- The root `variant` excludes the `link` button variant because grouped link-style split buttons do
  not form a coherent control.
- The root `size` excludes icon-only button sizes because the default path expects a visible label on
  the primary action.
- `SplitButtonContent` is a small wrapper over `Portal`, `Menu.Positioner`, and `Menu.Content`; the menu rows still come from the
  shared `Menu*` API.

### Controlled Menu State

Use the same controlled open-state contract as `Menu` when outside state should own the dropdown.

<Preview>
  <SplitButtonControlledExample />

  <Preview.Code>
    {`
      import {
        SplitButton,
        SplitButtonAction,
        SplitButtonContent,
        SplitButtonTrigger,
      } from "moduix";
      import { useState } from "react";

      export function ControlledSplitButton() {
        const [open, setOpen] = useState(false);

        return (
          <SplitButton open={open} onOpenChange={(details) => setOpen(details.open)} variant="outline">
            <SplitButtonAction>Share</SplitButtonAction>
            <SplitButtonTrigger aria-label="More share actions" />
            <SplitButtonContent>
              <Menu.Item value="action">Copy Link</Menu.Item>
              <Menu.Item value="action">Invite by Email</Menu.Item>
              <Menu.Separator />
              <Menu.Item value="close-menu" onSelect={() => setOpen(false)}>
                Close Menu
              </Menu.Item>
            </SplitButtonContent>
          </SplitButton>
        );
      }
    `}

  </Preview.Code>
</Preview>

## Styling

`SplitButton` exposes stable `data-slot` hooks, but it does not add dedicated `--split-button-*`
theme variables. The grouped buttons still rely on the shared `Button` and `Menu` styling
contracts.

### CSS Properties

`SplitButton` does not expose a dedicated CSS variable table. Use the shared `Button` and `Menu`
variables on the parts they style.

### Styling hooks

- `data-slot="split-button-root"` targets the grouped wrapper.
- `data-slot="split-button-action"` targets the primary action side.
- `data-slot="split-button-trigger"` targets the dropdown trigger side.
- `SplitButtonAction` and `SplitButtonTrigger` still carry the normal `Button` data attributes such
  as `data-size` and `data-variant`.
