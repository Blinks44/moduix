## Choosing the right component

Use `OTPField` when the value is a short fixed-length verification code and each character should be
entered into its own visible slot.

- Choose **`OTPField`** for one-time passwords, PINs, backup codes, and invite codes.
- Choose **`Input`** when one plain text input is enough.
- Choose **`PasswordInput`** when the value is a single secret string with a visibility toggle.

## Basic

Start with `OTPField` inside `Field`. When `children` is omitted, moduix renders one styled slot
per character in `length`.

## Anatomy

The public shape stays small:

```text
OTPField[length]
├─ OTPFieldInput
├─ OTPFieldInput
├─ OTPFieldSeparator (optional)
└─ OTPFieldInput
```

| Part                | Role                                                             |
| ------------------- | ---------------------------------------------------------------- |
| `OTPField`          | Root state machine and default slot renderer.                    |
| `OTPFieldInput`     | One visible character slot.                                      |
| `OTPFieldSeparator` | Optional visual separator between groups. Decorative by default. |

## Composition

### When to stay high-level

Stay with `OTPField` alone when every slot shares the same structure and styling.

Pass Base UI root props such as `value`, `defaultValue`, `onValueChange`, `onValueComplete`,
`validationType`, `normalizeValue`, `mask`, `autoSubmit`, `required`, `disabled`, and `readOnly`
directly to `OTPField`.

The default path keeps the field label association on the first slot from `htmlFor={id}`. The
remaining generated slots receive position-aware labels automatically.

### When to go custom

Switch to `OTPFieldInput` and `OTPFieldSeparator` when the layout needs grouped slots, separators,
placeholders, or per-slot native input props.

When composing manually, render exactly `length` inputs, keep the first slot unlabeled so the field
label still names the control, and give later slots labels such as `"Character 2 of 6"`.

### Default props

| Prop             | Default   | Values / Notes                        |
| ---------------- | --------- | ------------------------------------- |
| `validationType` | `numeric` | `numeric`, `alphanumeric`, `none`     |
| `mask`           | `false`   | Masks visible characters              |
| `autoSubmit`     | `false`   | Submits the owning form on completion |
| `className`      | -         | `string` or Base UI state callback    |

### Behavioral notes

- Use `value` with `onValueChange` for controlled fields, or `defaultValue` for uncontrolled ones.
- `length` must match the number of rendered `OTPFieldInput` parts in manual composition.
- `OTPFieldSeparator` is decorative by default with `aria-hidden="true"` and `role="presentation"`.
- Use `validationType="alphanumeric"` for recovery or invite codes.
- Use `validationType="none"` only when you also own normalization rules through `normalizeValue`.

## Examples

### Alphanumeric

Use `validationType="alphanumeric"` for recovery, backup, or invite codes.

<Preview>
  <OTPFieldAlphanumericExample />

  <Preview.Code>
    {`
      import {
        Field,
        FieldDescription,
        FieldLabel,
        OTPField,
      } from "moduix";
      import { useId, useState } from "react";

      export function AlphanumericOTPField() {
        const id = useId();
        const [value, setValue] = useState("");

        return (
          <Field>
            <FieldLabel htmlFor={id}>Recovery code</FieldLabel>
            <FieldDescription>
              Letters and numbers are allowed, for example <code>A7C9XZ</code>.
            </FieldDescription>
            <OTPField
              id={id}
              length={6}
              value={value}
              validationType="alphanumeric"
              onValueChange={setValue}
            />
          </Field>
        );
      }
    `}

  </Preview.Code>
</Preview>

### Auto Submit

Use `autoSubmit` when a complete code should submit its owning form immediately.

<Preview>
  <OTPFieldAutoSubmitExample />

  <Preview.Code>
    {`
      import { Field, FieldDescription, FieldLabel, OTPField } from "moduix";
      import { useId, useState } from "react";

      export function AutoSubmitOTPField() {
        const id = useId();
        const [completedValue, setCompletedValue] = useState("");
        const [submittedValue, setSubmittedValue] = useState("");

        return (
          <form
            className="form"
            onSubmit={(event) => {
              event.preventDefault();
              const formData = new FormData(event.currentTarget);
              setSubmittedValue(String(formData.get("verificationCode") ?? ""));
            }}
          >
            <Field>
              <FieldLabel htmlFor={id}>Verification code</FieldLabel>
              <FieldDescription>
                Form submits automatically when all slots are filled.
              </FieldDescription>
              <OTPField
                id={id}
                name="verificationCode"
                length={6}
                autoSubmit
                onValueComplete={(value) => {
                  setCompletedValue(value);
                }}
              />
            </Field>
            <span className="hint">Last completed value: {completedValue || "empty"}</span>
            <span className="hint">Last submitted value: {submittedValue || "empty"}</span>
          </form>
        );
      }
    `}

  </Preview.Code>

  <Preview.CSS>
    {`

### Custom Sanitization

Use `validationType="none"` with `normalizeValue` when pasted or typed input needs app-specific
sanitization.

<Preview>
  <OTPFieldCustomSanitizationExample />

  <Preview.Code>
    {`
      import { Field, FieldDescription, FieldLabel, OTPField } from "moduix";
      import { useId, useState } from "react";

      export function CustomSanitizationOTPField() {
        const id = useId();
        const [value, setValue] = useState("");
        const [invalidValue, setInvalidValue] = useState("");

        return (
          <div className="stack">
            <Field className="field">
              <FieldLabel htmlFor={id}>Invite code</FieldLabel>
              <FieldDescription>
                <code>validationType=&quot;none&quot;</code> with custom sanitization.
              </FieldDescription>
              <OTPField
                id={id}
                length={6}
                value={value}
                validationType="none"
                normalizeValue={(nextValue) => nextValue.toUpperCase().replace(/[^A-Z0-9]/g, "")}
                onValueChange={setValue}
                onValueInvalid={(nextValue) => {
                  setInvalidValue(nextValue);
                }}
              />
            </Field>
            <span className="hint">Current value: {value || "empty"}</span>
            <span className="hint">Last invalid attempt: {invalidValue || "none"}</span>
          </div>
        );
      }
    `}

  </Preview.Code>

  <Preview.CSS>
    {`
      .stack {
        display: grid;
        align-items: flex-start;
        gap: var(--spacing-3);
      }

      }

      .hint {

### Styling hooks

Use `className` on the root or parts for local overrides, and target the stable public slots when
you need CSS selectors.

| Part                | `data-slot`           | Useful state hooks                                 |
| ------------------- | --------------------- | -------------------------------------------------- |
| `OTPField`          | `otp-field-root`      | `data-disabled`, `data-readonly`                   |
| `OTPFieldInput`     | `otp-field-input`     | `data-filled`, `data-invalid`, `data-complete`     |
| `OTPFieldSeparator` | `otp-field-separator` | No component-specific state; decorative by default |

Base UI also exposes field-state attributes such as `data-focused`, `data-valid`, `data-touched`,
and `data-dirty` on relevant parts. moduix only styles a subset by default, but the attributes stay
available for consumer CSS.
