## API Reference

`Alert` is a standalone moduix component and does not wrap a Base UI primitive.

## Choosing the right component

Use `Alert` when the message should stay inline in the page layout and participate in the normal
document flow.

- Choose **`Alert`** for inline status, validation, and system messages inside forms, panels, or page content.
- Choose **`Toast`** for transient notifications that appear outside the document flow and dismiss on their own.

## Basic

Start with `Alert`, `AlertContent`, `AlertTitle`, and `AlertDescription`. Add `AlertIcon` only
when the message benefits from a leading visual cue.

## Anatomy

`AlertIcon` is optional. `AlertContent` groups the text slots and keeps the layout predictable.

```text
Alert
├─ AlertIcon (optional)
└─ AlertContent
   ├─ AlertTitle
   └─ AlertDescription
```

| Part               | Role                                                    |
| ------------------ | ------------------------------------------------------- |
| `Alert`            | Root slot. Applies variant styles and semantics.        |
| `AlertIcon`        | Optional icon slot. Sizes child SVG icons consistently. |
| `AlertContent`     | Text content wrapper for title and description.         |
| `AlertTitle`       | Alert heading slot. Renders `h3` by default.            |
| `AlertDescription` | Supporting message slot.                                |

## Composition

### When to stay high-level

Stay with `Alert`, `AlertContent`, `AlertTitle`, and `AlertDescription` for most inline feedback
messages.

Use `variant` for intent: `default`, `info`, `success`, `warning`, or `destructive`.

Keep text, links, and extra actions inside `AlertContent`. The component intentionally avoids
special-purpose action props or slot bags; normal composition already covers those cases cleanly.

### When to go custom

Switch to custom composition only when the alert needs extra inline controls, local dismiss state,
or a custom heading level. Keep that logic in the consuming component instead of pushing state or
actions into the alert API.

### Default props

| Prop      | Default   | Values / Notes                                           |
| --------- | --------- | -------------------------------------------------------- |
| `variant` | `default` | `default`, `info`, `success`, `warning`, `destructive`   |
| `role`    | Auto      | `status` by default, `alert` for `variant="destructive"` |

### Behavioral notes

- `role` defaults to `status`, and switches to `alert` for `variant="destructive"`.
- `AlertTitle` renders `h3` by default and accepts `as="h1"` through `as="h6"` when the document
  outline needs a different heading level.
- `AlertIcon` defaults to `aria-hidden="true"`, so essential meaning should stay in the title or
  description.
- The component is intentionally presentational. It does not manage dismiss state, focus, or
  keyboard behavior for embedded actions.

### Actions

Compose extra controls inside `AlertContent` when the message needs a recovery path or follow-up
action. Keep dismiss state in the consuming component when a local alert should be hidden.

<Preview>
  <AlertActionsExample />

  <Preview.Code>
    {`
      import {
        Alert,
        AlertContent,
        AlertDescription,
        AlertIcon,
        AlertTitle,
        Button,
        InfoIcon,
      } from "moduix";
      import { useState } from "react";

      export function AlertActionsDemo() {
        const [visible, setVisible] = useState(true);

        if (!visible) return null;

        return (
          <Alert variant="warning" className="customAlert">
            <AlertIcon>
              <InfoIcon />
            </AlertIcon>
            <AlertContent>
              <AlertTitle>Storage is almost full</AlertTitle>
              <AlertDescription>
                You are using 92% of the available storage. Archive old uploads or upgrade the plan.
              </AlertDescription>
              <div className="actions">
                <Button size="sm">Review uploads</Button>
                <Button size="sm" variant="outline" onClick={() => setVisible(false)}>
                  Dismiss
                </Button>
              </div>
            </AlertContent>
          </Alert>
        );
      }
    `}

  </Preview.Code>

  <Preview.CSS>
    {`

        --alert-icon-color: var(--color-primary);
        --alert-radius: var(--radius-md);

## Styling

Use `className` on the root or parts for local styles. Override `--alert-*` CSS variables on the
root when the alert needs local theming.

The built-in semantic variants also read shared palette tokens:

| Variant       | Shared token          | Default                      |
| ------------- | --------------------- | ---------------------------- |
| `success`     | `--color-success`     | `oklch(0.627 0.194 149.214)` |
| `warning`     | `--color-warning`     | `oklch(0.795 0.184 86.047)`  |
| `destructive` | `--color-destructive` | theme destructive            |

### Styling hooks

- `data-slot="alert-root"`, `alert-icon`, `alert-content`, `alert-title`, and `alert-description`
  are stable selectors on the exported parts.
- `Alert` writes the selected intent to `data-variant`, which is the main state hook for
  variant-specific styling.
