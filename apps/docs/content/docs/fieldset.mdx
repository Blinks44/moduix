## Choosing the right component

Use `Fieldset` when several related controls need one shared visible label and group semantics.

- Choose **`Fieldset`** when several controls belong to one section or question.
- Choose **`Field`** when only one control needs its own label, description, and validation state.
- Choose a standalone group such as **`RadioGroup`** only when the extra section wrapper and legend
  are not needed.

## Basic

Start with `Fieldset`, `FieldsetLegend`, and the related controls inside. This is the normal path
for grouped form sections such as billing details or profile settings.

## Anatomy

`Fieldset` groups related controls under one semantic legend. Keep the visible group label in
`FieldsetLegend`, then place related `Field` blocks or grouped option rows inside the same root.

```text
Fieldset
├─ FieldsetLegend
└─ grouped content
   ├─ Field
   │  ├─ FieldLabel
   │  ├─ FieldControl
   │  └─ FieldError
   └─ Field / FieldItem rows (optional)
```

| Part             | Role                                                                                                  |
| ---------------- | ----------------------------------------------------------------------------------------------------- |
| `Fieldset`       | Root semantic group. Renders native `fieldset`, carries shared disabled state, and supports `render`. |
| `FieldsetLegend` | Group label announced for the whole fieldset.                                                         |

## Composition

### When to stay high-level

Stay with `Fieldset`, `FieldsetLegend`, and regular child content for most grouped forms.

Use `Fieldset` when several controls belong to one section, should disable together, or should be
announced under one shared legend.

`Fieldset` and `FieldsetLegend` are the whole component surface. Pass `className` directly to the
part you want to customize.

### When to go custom

Use `render` only when the semantic fieldset also needs to be another Base UI root, such as
`RadioGroup` or `CheckboxGroup`.

That is the main composition escape hatch for `Fieldset`. The normal structure stays simple:
`Fieldset` owns the group semantics, and the nested controls keep their own behavior.

### Default props

| Prop        | Default    | Values                                       |
| ----------- | ---------- | -------------------------------------------- |
| `disabled`  | `false`    | `true`, `false`                              |
| `render`    | `fieldset` | Any valid React element or render callback   |
| `className` | _not set_  | `string` or `(state) => string \| undefined` |

### Behavioral notes

- `Fieldset` renders a native `<fieldset>` by default.
- `FieldsetLegend` is the shared visible label for the group.
- Use `disabled` on `Fieldset` to dim and disable the whole section instead of handling each child
  individually.
- Keep `render` for real root-level composition needs. Standard grouped fields do not need it.

## Examples

### Disabled

Use `disabled` on `Fieldset` to prevent interaction with every native control inside the group.

        Fieldset,
        FieldsetLegend,

### Custom Composition

Use `render` when the semantic fieldset also needs to be another Base UI root. This is the main
advanced composition path for radio groups and checkbox groups inside forms.

<Preview>
  <CustomCompositionFieldsetExample />

  <Preview.Code>
    {`
      import {
        Field,
        FieldItem,
        FieldLabel,
        Fieldset,
        FieldsetLegend,
        Radio,
        RadioGroup,
        RadioLabel,
      } from "moduix";

      export function StorageTypeFieldset() {
        return (
          <Field name="storageType">
            <Fieldset render={<RadioGroup defaultValue="ssd" />}>
              <FieldsetLegend>Storage type</FieldsetLegend>

              <FieldItem>
                <FieldLabel>
                  <Radio value="ssd" />
                  <RadioLabel>SSD</RadioLabel>
                </FieldLabel>
              </FieldItem>

              <FieldItem>
                <FieldLabel>
                  <Radio value="hdd" />
                  <RadioLabel>HDD</RadioLabel>
                </FieldLabel>
              </FieldItem>
            </Fieldset>
          </Field>
        );
      }
    `}

  </Preview.Code>
</Preview>

## Styling

Style `Fieldset` through `className`, `data-slot="fieldset-*"` hooks, and the public
`--fieldset-*` CSS variables shown below.

### Styling hooks

Use the part that owns the visual concern:

- `Fieldset` for overall width, spacing, padding, border, and radius
- `FieldsetLegend` for the group heading color, typography, and bottom border
- nested `Field` parts for the controls inside the group

The main public hooks include `data-slot="fieldset-root"` and `data-slot="fieldset-legend"`.
