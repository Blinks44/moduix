## Choosing the right component

Use `Fieldset` for multiple related controls with one legend. Use `Field` for one control with its
own label, description, and validation message.

## Basic

Compose `Fieldset.Root`, `Fieldset.Legend`, optional helper or error text, and the related controls.
The short `<Fieldset>` form is equivalent to `<Fieldset>`.

<Preview>
  <FieldsetExample />

  <Preview.Code>
    {`
      import { Field, Fieldset } from "moduix";

      export function ContactDetails() {
        return (
          <Fieldset>
            <Fieldset.Legend>Contact details</Fieldset.Legend>
            <Field.Root>
              <Field.Label>Name</Field.Label>
              <Field.Input name="name" />
            </Field.Root>
            <Field.Root>
              <Field.Label>Email</Field.Label>
              <Field.Input name="email" type="email" />
            </Field.Root>
            <Fieldset.HelperText>

## Anatomy

```text
Fieldset.Root | Fieldset.RootProvider
├─ Fieldset.Legend
├─ grouped controls
├─ Fieldset.HelperText (optional)
├─ Fieldset.ErrorText (optional)
└─ Fieldset.Context (optional)
```

| Part                         | Role                                                                  |
| ---------------------------- | --------------------------------------------------------------------- |
| `Fieldset` / `Fieldset.Root` | Creates state and renders a native `fieldset`.                        |
| `Fieldset.RootProvider`      | Renders externally created `useFieldset` state.                       |
| `Fieldset.Legend`            | Native `legend` and accessible group name.                            |
| `Fieldset.HelperText`        | Description linked to the root through `aria-describedby`.            |
| `Fieldset.ErrorText`         | Polite-live validation message rendered only while `invalid` is true. |
| `Fieldset.Context`           | Inline access to IDs, state, refs, and prop getters.                  |

## Composition

### Recommended composition

Keep one visible `Fieldset.Legend` inside the root. Pass `disabled`, `invalid`, and `id` directly to
the root. Ark links the legend, mounted helper text, and active error text to the fieldset.

Root, provider, legend, helper text, and error text forward refs to their rendered DOM elements.
Use `asChild` with one semantic child when a custom host must own a part.

### When to go custom

Use `Fieldset.Context` for an inline state read and `useFieldsetContext` in reusable descendants.
Use `useFieldset` with `Fieldset.RootProvider` when state is owned outside the rendered tree. Do not
wrap a provider in `Fieldset.Root` for the same state instance.

### Default props

| Surface         | Prop       | Default |
| --------------- | ---------- | ------- |
| `Fieldset.Root` | `disabled` | `false` |
| `Fieldset.Root` | `invalid`  | `false` |

### Behavioral notes

- `Fieldset.Legend` renders a native `legend`, unlike the removed legacy wrapper.
- Native controls inside a disabled fieldset are disabled by the browser.
- Ark-aware form controls can consume the shared fieldset context.
- `Fieldset.ErrorText` is absent from the DOM until `invalid` is true.
- Fieldset owns no value, keyboard navigation, callback, or `HiddenInput`.
- Legacy `render` and flat exports such as `FieldsetLegend` are not supported; use `asChild` and the
  namespace parts.

### Checkbox

Use the fieldset legend as the shared name for a related set of Ark checkbox controls. Keep each
`Checkbox.HiddenInput` for native form submission and reset behavior.

<Preview>
  <FieldsetCheckboxExample />

  <Preview.Code>
    {`
      import { Checkbox, Fieldset } from "moduix";

      export function EmailPreferences() {
        return (
          <Fieldset>
            <Fieldset.Legend>Email preferences</Fieldset.Legend>
            {preferences.map((preference) => (
              <Checkbox.Root key={preference.value} value={preference.value}>
                <Checkbox.Control>
                  <Checkbox.Indicator />
                </Checkbox.Control>
                <Checkbox.Label>{preference.label}</Checkbox.Label>
                <Checkbox.HiddenInput />
              </Checkbox.Root>
            ))}
          </Fieldset>
        );
      }
    `}

  </Preview.Code>
  <Preview.CSS>{fieldsetExampleCss}</Preview.CSS>

### Root Provider

Create fieldset state outside the rendered tree when sibling controls or reusable descendants need
to coordinate with it.

<Preview>
  <FieldsetRootProviderExample />

  <Preview.Code>
    {`
      import { Field, Fieldset, useFieldset } from "moduix";

      export function ExternalFieldsetState() {
        const fieldset = useFieldset({ invalid: true });

        return (
          <FieldsetProvider value={fieldset}>
            <Fieldset.Legend>Contact details</Fieldset.Legend>
            <Field.Root>
              <Field.Label>Email</Field.Label>
              <Field.Input type="email" defaultValue="invalid-address" />
            </Field.Root>
            <Fieldset.ErrorText>Enter a valid email address.</Fieldset.ErrorText>
            <Fieldset.Context>
              {(context) => <output>Invalid: {String(context.invalid)}</output>}
            </Fieldset.Context>
          </Fieldset.RootProvider>
        );
      }
    `}

  </Preview.Code>
  <Preview.CSS>{fieldsetExampleCss}</Preview.CSS>
  <Preview.Data>{fieldsetEmptyData}</Preview.Data>
</Preview>

### Styling hooks

Every DOM part has Ark `data-scope="fieldset"` and a `data-part` matching the anatomy. Root and
legend expose `data-disabled` and `data-invalid`; moduix also adds `data-slot="fieldset-*"` hooks.
Use `className` for local overrides and `--fieldset-*` variables for theme-level customization,
especially legend spacing and root border tokens.
