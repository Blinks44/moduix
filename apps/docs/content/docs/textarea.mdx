## Choosing the right component

Use `Textarea` when the value is plain text, can span multiple lines, and should keep the native
browser textarea behavior.

- Choose **`Input`** for short single-line values such as names, URLs, or compact filters.
- Choose **`Textarea`** for notes, comments, descriptions, and other longer free-form text.
- Choose **`Field` + `Textarea`** when labels, descriptions, and validation should stay
  declarative.

## Basic

Start with `Textarea` inside `Field`. This is the recommended path for ordinary forms, settings,
and issue or note entry.

## Anatomy

The public structure is intentionally small:

```text
Field (optional wrapper)
└─ Textarea
```

| Part       | Role                                                       |
| ---------- | ---------------------------------------------------------- |
| `Textarea` | Native textarea element with moduix styles and field data. |

`Textarea` does not add wrapper shells, sub-parts, or textarea-specific slot props.

## Composition

### When to stay high-level

Stay with `Textarea` for normal multi-line text entry.

Use `Textarea` directly for native props such as `rows`, `name`, `placeholder`, `maxLength`,
`readOnly`, `disabled`, `required`, `value`, and `defaultValue`.

Wrap it with `Field`, `Field.Label`, `Field.HelperText`, and `Field.ErrorText` when the input should
share declarative labels, helper text, and validation messaging.

### When to go custom

There is no lower-level `Textarea` composition path beyond the styled textarea root. When structure
needs to change, keep `Textarea` as the control and customize the surrounding layout with your own
wrappers, actions, or adjacent UI.

### Behavioral notes

- Use `value` with `onValueChange` for controlled text.
- Use `name` plus regular form submission when the value only needs to be read from `FormData`.
- Use `autoResize` when the field should grow with content. In browsers without
  `field-sizing: content`, the textarea keeps normal sizing behavior.
- For inline edit/read-only flows, keep the same mounted `Textarea` and toggle `readOnly` instead
  of swapping to separate display markup.

### Readonly Editing

For multiline inline editing, keep the same `Textarea` mounted and switch only `readOnly` plus the
action buttons. That avoids a text-to-textarea layout jump and lets you focus the field as soon as
edit mode starts.

<Preview>
  <TextareaReadOnlyEditingExample />

  <Preview.Code>
    {`
      import { Button, CheckIcon, CloseIcon, Field, PencilIcon, Textarea } from "moduix";
      import { useEffect, useRef, useState } from "react";

      export function ReadonlyEditingTextareaDemo() {
        const [editing, setEditing] = useState(false);
        const [value, setValue] = useState(
          "Build the docs examples first, then harden the public API around real usage.",
        );
        const [draft, setDraft] = useState(value);
        const textareaRef = useRef(null as HTMLTextAreaElement | null);

        useEffect(() => {
          if (editing) {
            textareaRef.current?.focus();
          }
        }, [editing]);

        const handleEdit = () => {
          setDraft(value);
          setEditing(true);
        };

        const handleCancel = () => {
          setDraft(value);
          setEditing(false);
        };

        const handleSave = () => {
          setValue(draft);
          setEditing(false);
        };

        return (
          <Field className={styles.field}>
            <Field.Label>Team note</Field.Label>
            <Field.HelperText>
              The textarea stays mounted and only switches between read-only and editable modes.
            </Field.HelperText>
            <Textarea
              ref={textareaRef}
              autoResize
              readOnly={!editing}
              rows={3}
              value={editing ? draft : value}
              onValueChange={setDraft}
            />
            <div className={styles.actions}>
              {editing ? (
                <>
                  <Button aria-label="Cancel editing" size="icon-md" variant="ghost" onClick={handleCancel}>
                    <CloseIcon />
                  </Button>
                  <Button aria-label="Save changes" size="icon-md" onClick={handleSave}>
                    <CheckIcon />
                  </Button>
                </>
              ) : (
                <Button aria-label="Edit team note" size="icon-md" variant="ghost" onClick={handleEdit}>
                  <PencilIcon />
                </Button>
              )}
            </div>
          </Field>
        );
      }
    `}

  </Preview.Code>

  <Preview.CSS>
    {`
      .field {
        width: min(28rem, calc(100vw - var(--spacing-8)));
      }

      .actions {
        display: flex;
        justify-content: flex-end;
        gap: var(--spacing-2);
      }
    `}

  </Preview.CSS>
</Preview>

### Styling hooks

Use the textarea root for the visual control itself and the surrounding `Field` for layout or text
around it.

- `Textarea` root for border, background, placeholder, resize behavior, and focus ring
- `Field` wrapper for width, max width, spacing, label layout, description, and validation text
- `data-slot="textarea-root"` for stable targeting in scoped styles
- `data-auto-resize` when `autoResize` is enabled
- `data-invalid`, `[disabled]`, `data-disabled`, `:read-only`, and `data-readonly` for state-aware
  styling
