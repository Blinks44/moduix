## API Reference

Ark UI does not ship a dedicated textarea primitive. Moduix builds this component with the official
[Ark factory](https://ark-ui.com/docs/guides/composition#the-ark-factory), follows Ark's
[composition](https://ark-ui.com/docs/guides/composition), [styling](https://ark-ui.com/docs/guides/styling), and
[ref](https://ark-ui.com/docs/guides/ref) guidance, and delegates field behavior to
[Ark Field.Textarea](https://ark-ui.com/docs/components/field).

## Choosing the right component

Use `Textarea` when the value is plain text, can span multiple lines, and should keep the native
browser textarea behavior.

- Choose **`Input`** for short single-line values such as names, URLs, or compact filters.
- Choose **`Textarea`** for notes, comments, descriptions, and other longer free-form text.
- Choose **`Editable`** for preview-first text that switches into edit mode.
- Choose **`Field` + `Textarea`** when labels, descriptions, and validation should stay
  declarative.

## Anatomy

The public structure is intentionally small:

```text
Field (optional wrapper)
└─ Textarea
```

| Part       | Role                                                       |
| ---------- | ---------------------------------------------------------- |
| `Textarea` | Native textarea element with moduix styles and field data. |

`Textarea` does not add wrapper shells, sub-parts, provider/context hooks, hidden inputs, or
textarea-specific slot props.

## Composition

### Recommended composition

Stay with `Textarea` for normal multi-line text entry.

Use `Textarea` directly for native props such as `rows`, `name`, `placeholder`, `maxLength`,
`readOnly`, `disabled`, `required`, `value`, and `defaultValue`.

Wrap it with `Field`, `Field.Label`, `Field.HelperText`, and `Field.ErrorText` when the input should
share declarative labels, helper text, and validation messaging.

### When to go custom

There is no lower-level `Textarea` composition path beyond the styled textarea root. When structure
needs to change, keep `Textarea` as the control and customize the surrounding layout with your own
wrappers, actions, or adjacent UI.

Use `Editable` instead of composing custom read-only/editing controls around `Textarea`.

### Default props

| Prop         | Default | Notes                                            |
| ------------ | ------- | ------------------------------------------------ |
| `autoresize` | `false` | Ark auto-resize behavior is opt-in per textarea. |

### Behavioral notes

- Use `value` with native `onChange` for controlled text.
- Use `name` plus regular form submission when the value only needs to be read from `FormData`.
- Use Ark `autoresize` when the field should grow with content.
- Use `Editable` for inline edit/read-only flows with preview text, edit triggers, and commit or
  cancel controls.

## Examples

### Controlled

Use `value` and native `onChange` when content should stay in React state.

<Preview>
  <ControlledTextareaExample />

  <Preview.Code>
    {`
      import { Field, Textarea } from "@moduix/react";
      import { useState } from "react";

      function ControlledTextarea() {
        const [value, setValue] = useState("");

        return (
          <Field className={styles.field}>
            <Field.Label>Feedback</Field.Label>
            <Textarea
              value={value}
              onChange={(event) => setValue(event.currentTarget.value)}
              placeholder="Type to control value"

## Styling

Style `Textarea` through `className`, `data-slot="textarea-root"`, and the public `--textarea-*`
variables defined in `theme.css`.

### Styling hooks

Use the textarea root for the visual control itself and the surrounding `Field` for layout or text
around it.

- `Textarea` root for border, background, placeholder, resize behavior, and focus ring
- `Field` wrapper for width, max width, spacing, label layout, description, and validation text
- `data-slot="textarea-root"` for stable targeting in scoped styles
- `data-autoresize` when `autoresize` is enabled
- `data-invalid`, `[aria-invalid='true']`, `[disabled]`, `data-disabled`, `:read-only`, and
  `data-readonly` for state-aware styling
