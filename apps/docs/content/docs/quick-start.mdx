# Quick start

moduix supports two installation paths:

- Install the library as a regular npm package when you want package-managed updates.
- Install components into your own project with `shadcn` when you want direct ownership over the
  source.

## Install As A Package

Use this path when you want moduix to stay in `node_modules` and receive updates through normal
package management.

### 1. Install the package

The components, CSS, and design tokens are shipped together, while Base UI stays as a peer
dependency in your application:

```bash
npm install moduix @base-ui/react
```

`@base-ui/react` stays in your application bundle, so moduix does not ship duplicate Base UI
runtime.

### 2. Add styles

Import the required foundation stylesheet once in your application entry point:

```tsx
import 'moduix/style.css';
```

This stylesheet is required. It provides the shared design tokens and base layer styles that
component CSS depends on. Component styles are still loaded alongside the components you import.

If you want the optional reset, import it explicitly before the main stylesheet:

```tsx
import 'moduix/reset.css';
import 'moduix/style.css';
```

That keeps `style.css` focused on the global foundation, while unused component CSS can stay out of
the consuming bundle.

### 3. Use components

Import the components and parts you need from `moduix`:

```tsx
import { Button, Dialog, DialogContent, DialogTitle, DialogTrigger } from 'moduix';

export function Example() {
  return (
    <Dialog>
      <DialogTrigger render={<Button />}>Open dialog</DialogTrigger>
      <DialogContent>
        <DialogTitle>Project settings</DialogTitle>
      </DialogContent>
    </Dialog>
  );
}
```

## Install Components Into Your Project

Use this path when you want the installed files to become part of your own codebase instead of
staying in `node_modules`.

### 1. Prepare the project for shadcn

Make sure your project resolves the `@/*` alias to `src/*`. For Vite, that usually means:

```json title="tsconfig.app.json"
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

```ts title="vite.config.ts"
resolve: {
  alias: {
    '@': path.resolve(__dirname, './src'),
  },
}
```

Initialize `shadcn` with the moduix registry base item:

```bash
npx shadcn@latest init Blinks44/moduix/init
```

This creates `components.json`, installs the shared moduix foundation files, and sets up the
project configuration that the CLI uses to place generated files in the correct `components` and
`lib` locations.

### 2. Install components with shadcn

Install registry items from the moduix GitHub registry:

```bash
npx shadcn@latest add Blinks44/moduix/button
npx shadcn@latest add Blinks44/moduix/accordion
```

The registry will also install the shared moduix foundation files those components need, including
styles, icons, and small utilities.

### 3. Add styles

Import the generated foundation stylesheet from the files that `shadcn` added to your project:

```tsx
import '@/lib/moduix/styles/style.css';
```

The examples in this guide assume the default shadcn-style aliases such as `@/components` and
`@/lib`. If your project uses different aliases, update `components.json` and import the generated
files from the matching paths.

### 4. Use the generated files

Import components from the generated files in your own project:

```tsx
import { Button } from '@/components/moduix/button';
import {
  Accordion,
  AccordionHeader,
  AccordionItem,
  AccordionPanel,
  AccordionTrigger,
  AccordionTriggerIcon,
} from '@/components/moduix/accordion';
```

Most components are composed from named parts. This keeps behavior accessible while letting you
control markup, layout, and styling in the consuming application.

This composition style is inspired by [shadcn/ui](https://ui.shadcn.com/): the public API should be
easy to read, and the component structure should stay understandable even when you decide to own the
source directly.

## Styling

moduix works with the styling approach your project already uses:

- Native CSS with global selectors, cascade layers, and CSS variables.
- CSS Modules through `className` and component slot class props.
- Tailwind CSS utility classes for layout and one-off styling.
- CSS-in-JS libraries that attach classes or pass scoped styles through your own wrappers.

Component pages list the available CSS variables in the `CSS playground` tab. Use it to test values
live, then move the result into your project stylesheet, module, theme layer, or component wrapper.

## Documentation Flow

Start with the component page that matches the interface you are building. Each page includes a
basic example, common usage scenarios, customization examples, and the CSS variables that are useful
for that component.

For styling-heavy work, keep the docs open next to your app and use the live previews as a small CSS
playground. It is the quickest way to check spacing, sizes, colors, and state styles before copying
the final values into your codebase.
