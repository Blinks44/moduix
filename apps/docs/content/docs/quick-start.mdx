# Quick start

moduix supports two production workflows:

- Install `@moduix/react` when you want the shortest setup and package-managed updates.
- Install components with `shadcn` when you want the source, styles, and supporting utilities in
  your own repository.

Start with the npm package unless source ownership is an explicit requirement. Both paths expose the
same Ark-aligned component contracts and token system.

| Choose this path            | When it fits best                                                                                                      |
| --------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| npm package                 | You want updates through package management, imports from `@moduix/react`, and styles from `@moduix/react/style.css`.  |
| `shadcn` copy-owned install | You want generated source files inside your app, direct local customization, and imports from `@/components/moduix/*`. |

## Install as a Package

Use this path when you want moduix to stay in `node_modules` and receive updates through normal
package management.

### 1. Install the package

The components, CSS, and design tokens are shipped together. React 18 or 19 and Ark UI are peer
dependencies:

<Tabs items={['npm', 'pnpm', 'bun', 'deno', 'aube']}>
  <Tab value="npm">

```bash
npm install @moduix/react @ark-ui/react
```

  </Tab>
  <Tab value="pnpm">

```bash
pnpm add @moduix/react @ark-ui/react
```

  </Tab>
  <Tab value="bun">

```bash
bun add @moduix/react @ark-ui/react
```

  </Tab>
  <Tab value="deno">

```bash
deno add @moduix/react @ark-ui/react
```

  </Tab>
  <Tab value="aube">

```bash
aube add @moduix/react @ark-ui/react
```

  </Tab>
</Tabs>

`react`, `react-dom`, and `@ark-ui/react` stay in your application bundle, so moduix does not
ship duplicate framework or primitive runtimes.

### 2. Add styles

Import the required foundation stylesheet once in your application entry point:

```tsx
import '@moduix/react/style.css';
```

This stylesheet is required. It provides the shared design tokens and base layer styles that
component CSS depends on. Component styles are still loaded alongside the components you import.

If you want the optional reset, import it explicitly before the main stylesheet:

```tsx
import '@moduix/react/reset.css';
import '@moduix/react/style.css';
```

That keeps `style.css` focused on the global foundation, while unused component CSS can stay out of
the consuming bundle.

For a curated visual starting point, import one optional preset after `style.css`; see
[Themes](/docs/themes).

### 3. Use components

Import the components and parts you need from `@moduix/react`:

```tsx
import { Button, Dialog } from '@moduix/react';

export function Example() {
  return (
    <Dialog.Root>
      <Dialog.Trigger asChild>
        <Button>Open dialog</Button>
      </Dialog.Trigger>
      <Dialog.Backdrop />
      <Dialog.Positioner>
        <Dialog.Content>
          <Dialog.Title>Project settings</Dialog.Title>
        </Dialog.Content>
      </Dialog.Positioner>
    </Dialog.Root>
  );
}
```

## Install Components Into Your Project

Use this path when you want the installed files to become part of your own codebase instead of
staying in `node_modules`.

### 1. Prepare the project for shadcn

For an existing React project, follow the
[shadcn installation guide](https://ui.shadcn.com/docs/installation) for your bundler and initialize
the project:

```bash
npx shadcn@latest init
```

This creates the base setup and `components.json` used by the CLI. moduix does not require Tailwind
for its component styles, but it uses the shadcn registry format and project aliases for source
delivery.

Make sure your project resolves the `@/*` alias to `src/*`. The exact setup depends on the bundler:

#### Vite

For Vite, put the alias in both `tsconfig.json` and `tsconfig.app.json`, then mirror it in
`vite.config.ts`:

<Tabs items={['tsconfig.json', 'tsconfig.app.json', 'vite.config.ts']}>
  <Tab value="tsconfig.json">

```json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  },
  "files": [],
  "references": [{ "path": "./tsconfig.app.json" }, { "path": "./tsconfig.node.json" }]
}
```

  </Tab>
  <Tab value="tsconfig.app.json">

```json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

  </Tab>
  <Tab value="vite.config.ts">

```ts
resolve: {
  alias: {
    '@': path.resolve(__dirname, './src'),
  },
}
```

  </Tab>
</Tabs>

#### Rsbuild

For Rsbuild, add the alias to `tsconfig.json`, then mirror it in `rsbuild.config.ts`:

<Tabs items={['tsconfig.json', 'rsbuild.config.ts']}>
  <Tab value="tsconfig.json">

```json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

  </Tab>
  <Tab value="rsbuild.config.ts">

```ts
resolve: {
  alias: {
    '@': './src',
  },
}
```

  </Tab>
</Tabs>

After initialization, register the hosted moduix namespace:

```bash
npx shadcn@latest registry add '@moduix-react=https://moduix.dev/r/react/{name}.json'
```

You can also add the same namespace manually under `registries` in `components.json`:

```json
{
  "registries": {
    "@moduix-react": "https://moduix.dev/r/react/{name}.json"
  }
}
```

This gives the `shadcn` CLI the aliases and registry URL it needs to place generated files in
`src/components` and `src/lib` instead of a literal `@` directory.

### 2. Discover and preview before installing

For a copy-owned project, inspect the current configuration and the item before writing files:

```bash
npx shadcn@latest info --json
npx shadcn@latest search @moduix-react --query "confirmation dialog"
npx shadcn@latest view @moduix-react/dialog
npx shadcn@latest add @moduix-react/dialog --dry-run
```

Search by the interface you need rather than a familiar library name. The registry contains
components, blocks, themes, and their declared dependencies. See the [AI guide](/docs/ai) to use
the same discovery flow through shadcn MCP.

### 3. Install components with shadcn

Install registry items from the moduix hosted registry:

The registry will also install the shared moduix foundation files those components need, including
styles, icons, and small utilities.

### 4. Add styles

Import the generated foundation stylesheet from the files that `shadcn` added to your project:

```tsx
import '@/lib/moduix/styles/style.css';
```

If you want the optional reset in the copy-owned path, import it explicitly before the main
stylesheet:

```tsx
import '@/lib/moduix/styles/reset.css';
import '@/lib/moduix/styles/style.css';
```

The examples below assume `@/components` and `@/lib`. If your project uses different aliases,
update `components.json` and import the generated files from the matching paths.

### 5. Use the generated files

Import components from the generated files in your own project:

```tsx
import { Button } from '@/components/moduix/button';
import { Accordion } from '@/components/moduix/accordion';
```

Most components are composed from named parts. This keeps behavior accessible while letting you
control markup, layout, and styling in the consuming application.

The ownership flow is inspired by [shadcn/ui](https://ui.shadcn.com/), while component anatomy and
interaction behavior remain aligned with [Ark UI](https://ark-ui.com/).

## Styling

moduix works with the styling approach your project already uses:

- Native CSS with global selectors, cascade layers, and CSS variables.
- CSS Modules through `className` and component slot class props.
- Tailwind CSS utility classes for layout and one-off styling.
- CSS-in-JS libraries that attach classes or pass scoped styles through your own wrappers.

The default density uses `--size-md: 36px` for primary controls and `--size-sm: 32px` for popup
rows. Because components consume the shared scale, changing those tokens updates related inputs,
triggers, buttons, and menu-like items together.

Component pages list their public CSS variables and styling hooks. Use shared semantic tokens for
system-wide changes and component variables only for deliberate exceptions. See
[Tokens](/docs/tokens) for the complete hierarchy and [Animations](/docs/animations) for motion
recipes.
