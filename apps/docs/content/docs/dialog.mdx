## Choosing the right component

Use `Dialog` for general modal or non-modal flows that should stay in the current page context.

- Choose **`Dialog`** for forms, detail panels, secondary tasks, and multi-step workflows.
- Choose **`AlertDialog`** when the user must explicitly confirm a destructive or irreversible action.
- Choose **`Drawer`** when the surface should stay attached to an edge or support swipe/snap behavior.

## Basic

Start with `Dialog`, `DialogTrigger`, and `DialogContent`. This is the recommended path for most
modal workflows.

## Anatomy

`DialogContent` is the high-level surface. It already renders the portal, viewport, popup, and the
modal backdrop when `modal={true}`.

```text
Dialog
├─ DialogTrigger
└─ DialogContent
   ├─ portal (service slot)
   │  ├─ backdrop (modal only)
   │  └─ viewport (service slot)
   │     └─ popup surface
   │        ├─ DialogCloseIcon (optional)
   │        ├─ DialogHeader
   │        │  ├─ DialogTitle
   │        │  └─ DialogDescription (optional)
   │        ├─ DialogBody (optional)
   │        └─ DialogFooter (optional)
   │           └─ DialogClose
```

| Part                | Role                                                                                           |
| ------------------- | ---------------------------------------------------------------------------------------------- |
| `Dialog`            | Root state and interaction model. Use `open`, `defaultOpen`, `onOpenChange`, and `modal` here. |
| `DialogTrigger`     | Opens the dialog from user interaction. Use `handle` when the trigger lives outside the tree.  |
| `DialogContent`     | Popup container. It renders the internal portal and viewport, plus the backdrop in modal mode. |
| `DialogPortal`      | Optional escape hatch for portal placement and mount behavior.                                 |
| `DialogBackdrop`    | Optional styled backdrop part for manual composition.                                          |
| `DialogViewport`    | Optional positioning and scroll container for manual composition.                              |
| `DialogPopup`       | Optional popup surface for manual composition.                                                 |
| `DialogHeader`      | Groups title-level content at the top of the popup.                                            |
| `DialogTitle`       | Accessible dialog title announced by assistive technology.                                     |
| `DialogDescription` | Optional supporting text announced with the title.                                             |
| `DialogBody`        | Optional content region for forms, text, or custom layout.                                     |
| `DialogFooter`      | Optional actions row for submit/cancel buttons.                                                |
| `DialogCloseIcon`   | Optional icon-only close button. Pass children to replace the default close glyph.             |
| `DialogClose`       | Closes the dialog from action buttons while preserving built-in behavior.                      |

## Composition

### When to stay high-level

Stay with `Dialog`, `DialogTrigger`, and `DialogContent` for most flows.

Use `Dialog` for state and interaction: `open`, `defaultOpen`, `onOpenChange`, `modal`, `handle`,
`actionsRef`, `triggerId`, and `defaultTriggerId`.

Use `DialogContent` for the visible overlay. It is intentionally thin: it accepts Base UI popup
props, applies the library surface styles, and renders the default portal/backdrop/viewport/popup
stack for you.

### When to go custom

Switch to `DialogPortal`, `DialogBackdrop`, `DialogViewport`, and `DialogPopup` when the default
overlay structure is not enough.

That is the right path for custom portal placement, `keepMounted`, explicit backdrop control, or
viewport-specific layout such as top-aligned dialogs.

Use `createDialogHandle()` when the trigger lives outside the dialog tree or the dialog needs to
open imperatively.

### Default props

| Prop / Part       | Default          | Values / Notes                            |
| ----------------- | ---------------- | ----------------------------------------- |
| `modal`           | `true`           | `true`, `false`, or `"trap-focus"`        |
| `defaultOpen`     | `false`          | Initial uncontrolled open state           |
| `DialogContent`   | -                | Renders backdrop only when `modal={true}` |
| `DialogCloseIcon` | `"Close dialog"` | Default accessible label                  |

### Behavioral notes

- `modal={false}` keeps the page interactive, removes the backdrop, and does not trap focus.
- `modal="trap-focus"` keeps keyboard focus inside the dialog while still allowing outside pointer interaction.
- Backdrop clicks and `Escape` request close through `onOpenChange`. Keep the dialog controlled when close behavior depends on validation or async work.
- Nested dialogs receive Base UI nested-dialog attributes and the exported `--dialog-nested-*` variables, so the parent surface can visually recede while the child dialog is open.

## Examples

### Controlled

Control the open state from React when the dialog needs to coordinate with other UI.

<Preview>
  <ControlledDialogExample />

  <Preview.Code>
    {`
      import {
        Button,
        Dialog,
        DialogContent,
        DialogHeader,
        DialogTitle,
        DialogDescription,
        DialogFooter,
        DialogClose,
      } from "moduix";
      import { useState, Fragment } from "react";

      export function ControlledDialogDemo() {
        const [open, setOpen] = useState(false);

        return (
          <Fragment>
            <Button type="button" onClick={() => setOpen(true)}>
              Open controlled dialog
            </Button>
            <Dialog open={open} onOpenChange={setOpen}>
              <DialogContent>
                <DialogHeader>
                  <DialogTitle>Publish changes?</DialogTitle>
                  <DialogDescription>
                    This will make the latest version visible to all users.
                  </DialogDescription>
                </DialogHeader>
                <DialogFooter>
                  <DialogClose render={<Button variant="outline" />}>Back to editing</DialogClose>
                  <DialogClose render={<Button />}>Publish</DialogClose>
                </DialogFooter>
              </DialogContent>
            </Dialog>
          </Fragment>
        );
      }
    `}

  </Preview.Code>
</Preview>

### Handle

Use `createDialogHandle()` when the trigger lives outside the dialog tree or the dialog opens
programmatically.

<Preview>
  <DialogHandleExample />

  <Preview.Code>
    {`
      import {
        createDialogHandle,
        DialogTrigger,
        Button,
        Dialog,
        DialogContent,
        DialogHeader,
        DialogTitle,
        DialogDescription,
        DialogFooter,
        DialogClose,
      } from "moduix";
      import { useMemo, Fragment } from "react";

      export function DialogHandleDemo() {
        const dialogHandle = useMemo(() => createDialogHandle(), []);

        return (
          <Fragment>
            <DialogTrigger handle={dialogHandle} render={<Button variant="outline" />}>
              Open from detached trigger
            </DialogTrigger>
            <Button type="button" onClick={() => dialogHandle.open(null)}>
              Open programmatically
            </Button>

            <Dialog handle={dialogHandle}>
              <DialogContent>
                <DialogHeader>
                  <DialogTitle>Detached trigger</DialogTitle>
                  <DialogDescription>
                    This dialog is connected via createDialogHandle().
                  </DialogDescription>
                </DialogHeader>
                <DialogFooter>
                  <DialogClose render={<Button variant="outline" />}>Close</DialogClose>
                </DialogFooter>
              </DialogContent>
            </Dialog>
          </Fragment>
        );
      }
    `}

  </Preview.Code>
</Preview>

### Non-Modal

Set `modal={false}` when the dialog should not trap focus, lock page scroll, or block pointer
interaction with the rest of the page. `DialogContent` handles the pointer-event behavior and skips
the backdrop automatically.

### Trap Focus Without Blocking the Page

Set `modal="trap-focus"` when focus should stay inside the dialog but the rest of the page should
remain scrollable and clickable. `DialogContent` uses the same backdrop-free path here as it does
for non-modal dialogs.

<Preview>
  <TrapFocusDialogExample />

  <Preview.Code>
    {`
      import {
        Dialog,
        DialogTrigger,
        Button,
        DialogCloseIcon,
        DialogContent,
        DialogHeader,
        DialogTitle,
        DialogDescription,
        DialogFooter,
        DialogClose,
      } from "moduix";

      export function TrapFocusDialogDemo() {
        return (
          <Dialog modal="trap-focus">
            <DialogTrigger render={<Button />}>Open focus-trapped dialog</DialogTrigger>
            <DialogContent>
              <DialogHeader>
                <DialogTitle>Focus stays inside the dialog</DialogTitle>
                <DialogCloseIcon />
                <DialogDescription>
                  Outside content remains clickable, but keyboard focus stays trapped until the
                  dialog closes.
                </DialogDescription>
              </DialogHeader>
              <DialogFooter>
                <DialogClose render={<Button variant="outline" />}>Close</DialogClose>
              </DialogFooter>
            </DialogContent>
          </Dialog>
        );
      }
    `}

  </Preview.Code>
</Preview>

## Styling

Use `className` on `DialogContent`, `DialogHeader`, `DialogBody`, `DialogFooter`, and
`DialogCloseIcon` for most visual overrides. Move to `DialogPortal`, `DialogBackdrop`,
`DialogViewport`, and `DialogPopup` only when the overlay structure itself needs to change.

### Styling hooks

- `data-slot="dialog-trigger"`, `dialog-portal`, `dialog-backdrop`, `dialog-viewport`, `dialog-popup`, `dialog-header`, `dialog-title`, `dialog-description`, `dialog-body`, `dialog-footer`, `dialog-close`, and `dialog-close-icon` are stable selectors on the exported parts.
- Base UI also adds state attributes such as `data-open`, `data-closed`, `data-popup-open`, `data-disabled`, `data-nested`, `data-nested-dialog-open`, `data-starting-style`, and `data-ending-style` on the relevant overlay parts.
