## Choosing the right component

Use `Dialog` for focused forms, confirmations, details, and secondary workflows that stay in the
current page context. Use `Drawer` when the surface should remain attached to a viewport edge.

## Basic

Compose the explicit Ark UI structure: `Dialog.Root`, `Portal`, `Dialog.Backdrop`,
`Dialog.Positioner`, and `Dialog.Content`.

## Anatomy

```text
Dialog.Root
├─ Dialog.Trigger
└─ Portal
   ├─ Dialog.Backdrop
   └─ Dialog.Positioner
      └─ Dialog.Content
         ├─ Dialog.CloseTrigger or Dialog.CloseIcon
         ├─ Dialog.Title
         ├─ Dialog.Description
         ├─ Dialog.Header (moduix layout sugar)
         ├─ Dialog.Body (moduix layout sugar)
         └─ Dialog.Footer (moduix layout sugar)

Dialog.RootProvider
└─ the same portal and content tree connected to useDialog()
```

| Part                     | Role                                                             |
| ------------------------ | ---------------------------------------------------------------- |
| `Dialog` / `Dialog.Root` | Owns dialog open state, focus lifecycle, and dismissal behavior. |
| `Dialog.Trigger`         | Opens the dialog.                                                |
| `Dialog.Backdrop`        | Optional modal backdrop layer.                                   |
| `Dialog.Positioner`      | Floating positioner for content placement.                       |
| `Dialog.Content`         | Dialog surface container.                                        |
| `Dialog.Title`           | Accessible dialog title.                                         |
| `Dialog.Description`     | Accessible supporting description.                               |
| `Dialog.CloseTrigger`    | Closes the dialog.                                               |
| `Dialog.RootProvider`    | Connects parts to state created with `useDialog()`.              |
| `Dialog.Context`         | Inline render-prop access to dialog state.                       |

`Dialog.Context` exposes inline state. `useDialogContext()` exposes the same state to reusable
child components.

## Composition

### Recommended composition

Keep `Portal`, `Backdrop`, `Positioner`, and `Content` explicit. This preserves structural override
points and matches Ark UI directly. Omit `Backdrop` for non-modal dialogs.

Use `Dialog.Trigger asChild` and `Dialog.CloseTrigger asChild` with one semantic button child.
`Dialog.CloseIcon` is moduix sugar that composes Ark `CloseTrigger` with the library close button.

### When to go custom

Use `open` with `onOpenChange(details)` for React-controlled state. The callback receives an Ark
detail object, so read `details.open`.

Use `useDialog()` with `Dialog.RootProvider` when state and methods must live outside the rendered
dialog tree. Do not render `Dialog.Root` for the same store.

Use `ids` when separately composed parts need stable shared accessibility IDs. Use
`initialFocusEl` and `finalFocusEl` for explicit focus targets. `role="alertdialog"` models
destructive or critical confirmation flows.

Use `lazyMount` and `unmountOnExit` instead of conditionally rendering the root. Use `present` only
when JavaScript-driven exit animation needs to control mounted state.

### Default props

| Surface                  | Default | Notes                                                  |
| ------------------------ | ------- | ------------------------------------------------------ |
| `Dialog` / `Dialog.Root` | —       | No additional behavioral defaults; Ark defaults apply. |

### Behavioral notes

- Modal dialogs trap focus, prevent background scrolling, hide underlying content from assistive
  technology, and restore focus after close.
- `modal={false}` disables focus trapping and scroll prevention and should omit the backdrop.
- `closeOnEscape`, `closeOnInteractOutside`, `trapFocus`, `preventScroll`, `restoreFocus`,
  `persistentElements`, and the outside-interaction callbacks pass through to Ark UI.
- `Dialog.Title` or root `aria-label` is required for an accessible name.
- Nested content receives `data-has-nested` and `--nested-layer-count`.
- Open-from-menu and rapid state changes use the controlled `open` contract. Unsaved-change
  confirmation uses two `useDialog()` stores and intercepts the parent store's `onOpenChange`.

## Examples

### Controlled

Use the Ark detail object when synchronizing dialog state with React.

<Preview>
  <ControlledDialogExample />
  <Preview.Code>
    {`
      import { Button, Dialog, Portal } from "moduix";
      import { useState } from "react";

      export function ControlledDialogDemo() {
        const [open, setOpen] = useState(false);

        return (
          <Dialog open={open} onOpenChange={(details) => setOpen(details.open)}>
            <Dialog.Trigger asChild>
              <Button>Open controlled dialog</Button>
            </Dialog.Trigger>
            <Portal>
              <Dialog.Backdrop />
              <Dialog.Positioner>
                <Dialog.Content>
                  <Dialog.Title>Publish changes?</Dialog.Title>
                  <Dialog.CloseIcon />
                </Dialog.Content>

`Dialog.Context`.

      import { Button, Dialog, Portal } from "moduix";
      import { useRef } from "react";

      export function DialogLifecycleDemo() {
        const inputRef = useRef(null as HTMLInputElement | null);
        const finalFocusRef = useRef(null as HTMLButtonElement | null);

        return (
          <>
            <Button ref={finalFocusRef}>Final focus target</Button>
            <Dialog

### Nested

Use separate `useDialog()` stores for nested layers. Ark UI provides stack-aware state and CSS
variables to the parent content.

<Preview>
  <NestedDialogExample />
  <Preview.Code>
    {`
      import { Button, Dialog, Portal, useDialog } from "moduix";

      export function NestedDialogDemo() {
        const parent = useDialog();
        const child = useDialog();

        return (
          <>
            <Button onClick={() => parent.setOpen(true)}>Open parent</Button>
            <DialogProvider value={parent}>
              <Portal>
                <Dialog.Backdrop />
                <Dialog.Positioner>
                  <Dialog.Content>
                    <Dialog.Title>Parent dialog</Dialog.Title>
                    <Button onClick={() => child.setOpen(true)}>Open nested</Button>
                  </Dialog.Content>
                </Dialog.Positioner>
              </Portal>
            </Dialog.RootProvider>
            <DialogProvider value={child}>
              <Portal>
                <Dialog.Backdrop />
                <Dialog.Positioner>
                  <Dialog.Content>
                    <Dialog.Title>Nested dialog</Dialog.Title>
                    <Dialog.CloseIcon />
                  </Dialog.Content>
                </Dialog.Positioner>
              </Portal>
            </Dialog.RootProvider>

### Multiple Triggers

Give each trigger a `value` and react to `onTriggerValueChange(details)` when one dialog edits
different records.

<Preview>
  <MultipleTriggersDialogExample />
  <Preview.Code>
    {`
      import { Button, Dialog, Portal } from "moduix";
      import { useState } from "react";

      export function MultipleTriggersDialogDemo() {
        const [activeUser, setActiveUser] = useState(null as (typeof users)[number] | null);

        return (
          <Dialog
            onTriggerValueChange={(details) => {
              setActiveUser(users.find((user) => user.id === details.value) ?? null);
            }}
          >
            {users.map((user) => (
              <Dialog.Trigger key={user.id} value={user.id} asChild>
                <Button variant="outline">Edit {user.name}</Button>
              </Dialog.Trigger>
            ))}
            <Portal>
              <Dialog.Backdrop />
              <Dialog.Positioner>
                <Dialog.Content>
                  <Dialog.Title>Edit user</Dialog.Title>
                  <Dialog.Description>{activeUser?.email}</Dialog.Description>
                  <Dialog.CloseIcon />
                </Dialog.Content>
              </Dialog.Positioner>
            </Portal>
          </Dialog>
        );
      }
    `}

### Styling hooks

- Moduix slots: `dialog-trigger`, `dialog-backdrop`, `dialog-positioner`, `dialog-content`,
  `dialog-title`, `dialog-description`, `dialog-close-trigger`, `dialog-close-icon`,
  `dialog-header`, `dialog-body`, and `dialog-footer`.
- Ark anatomy: `data-scope="dialog"` with `data-part="trigger|backdrop|positioner|content|title|description|close-trigger"`.
- State: `data-state="open|closed"`, `data-has-nested`, `data-nested`, `data-current`, and
  `data-value`.
- Runtime variables: `--layer-index` and `--nested-layer-count`.
- `Dialog.CloseIcon` is absolutely positioned against `Dialog.Content`; its placement does not
  depend on `Dialog.Header`.
