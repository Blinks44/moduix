## Choosing the right component

Use `Dialog` for focused forms, confirmations, details, and secondary workflows that stay in the
current page context. Use `Drawer` when the surface should remain attached to a viewport edge.

## Anatomy

```text
Dialog.Root
├─ Dialog.Trigger
└─ Overlay subtree (automatically portalled)
   ├─ Dialog.Backdrop
   └─ Dialog.Positioner
      └─ Dialog.Content
         ├─ Dialog.CloseTrigger or Dialog.CloseIcon
         ├─ Dialog.Title
         ├─ Dialog.Description
         ├─ Dialog.Header (moduix layout sugar)
         ├─ Dialog.Body (moduix layout sugar)
         └─ Dialog.Footer (moduix layout sugar)

Dialog.RootProvider
└─ the same automatically portalled content tree connected to moduix useDialog()
```

| Part                     | Role                                                             |
| ------------------------ | ---------------------------------------------------------------- |
| `Dialog` / `Dialog.Root` | Owns dialog open state, focus lifecycle, and dismissal behavior. |
| `Dialog.Trigger`         | Opens the dialog.                                                |
| `Dialog.Backdrop`        | Optional modal backdrop layer.                                   |
| `Dialog.Positioner`      | Floating positioner for content placement.                       |
| `Dialog.Content`         | Dialog surface container.                                        |
| `Dialog.Title`           | Accessible dialog title.                                         |
| `Dialog.Description`     | Accessible supporting description.                               |
| `Dialog.CloseTrigger`    | Closes the dialog.                                               |
| `Dialog.RootProvider`    | Connects parts to state created with moduix `useDialog()`.       |

Import `useDialog()` and `useDialogContext()` from `@moduix/react` when dialog state must live
outside the rendered tree or be read inside reusable children.

## Composition

The root portals `Backdrop` and `Positioner` automatically. Use `portalled={false}` for inline rendering and `portalRef` for a custom container; structural parts remain explicit and styleable.

### Recommended composition

Keep `Backdrop`, `Positioner`, and `Content` explicit. This preserves structural override
points and matches Ark UI directly. Omit `Backdrop` for non-modal dialogs.

Use `Dialog.Trigger asChild` and `Dialog.CloseTrigger asChild` with one semantic button child.
`Dialog.CloseIcon` is moduix sugar that composes Ark `CloseTrigger` with the library close button.

### When to go custom

Use `open` with `onOpenChange(details)` for React-controlled state. The callback receives an Ark
detail object, so read `details.open`.

Use moduix `useDialog()` with `Dialog.RootProvider` when state and methods must live outside the
rendered dialog tree. Do not render `Dialog.Root` for the same store.

Use `ids` when separately composed parts need stable shared accessibility IDs. Use
`initialFocusEl` and `finalFocusEl` for explicit focus targets. `role="alertdialog"` models
destructive or critical confirmation flows.

Use `lazyMount` and `unmountOnExit` instead of conditionally rendering the root. Use `present` only
when JavaScript-driven exit animation needs to control mounted state.

### Default props

| Surface                  | Default | Notes                                                  |
| ------------------------ | ------- | ------------------------------------------------------ |
| `Dialog` / `Dialog.Root` | —       | No additional behavioral defaults; Ark defaults apply. |

### Behavioral notes

- Modal dialogs trap focus, prevent background scrolling, hide underlying content from assistive
  technology, and restore focus after close.
- `modal={false}` disables focus trapping and scroll prevention and should omit the backdrop.
- `closeOnEscape`, `closeOnInteractOutside`, `trapFocus`, `preventScroll`, `restoreFocus`,
  `persistentElements`, and the outside-interaction callbacks pass through to Ark UI.
- `Dialog.Title` or root `aria-label` is required for an accessible name.
- Keep every rendered close part id unique. The canonical Basic composition uses one
  `Dialog.CloseIcon`; if a dialog needs multiple close controls, give the additional parts explicit
  unique `id` values.
- Nested content receives `data-has-nested` and `--nested-layer-count`. Moduix uses those hooks to
  scale the parent down and offset it downward so the active nested dialog still reads as layered.
- Open-from-menu and rapid state changes use the controlled `open` contract. Unsaved-change
  confirmation uses two moduix `useDialog()` stores and intercepts the parent store's `onOpenChange`.

### Styling hooks

- Moduix slots: `dialog-trigger`, `dialog-backdrop`, `dialog-positioner`, `dialog-content`,
  `dialog-title`, `dialog-description`, `dialog-close-trigger`, `dialog-close-icon`,
  `dialog-header`, `dialog-body`, and `dialog-footer`.
- Ark anatomy: `data-scope="dialog"` with `data-part="trigger|backdrop|positioner|content|title|description|close-trigger"`.
- State: `data-state="open|closed"`, `data-has-nested`, `data-nested`, `data-current`, and
  `data-value`.
- Runtime variables: `--layer-index` and `--nested-layer-count`.
- `Dialog.CloseIcon` is absolutely positioned against `Dialog.Content`; its placement does not
  depend on `Dialog.Header`.
