## Basic

Use `ContextMenuContent` for the default path. It keeps the portal and positioning structure
internal, while `ContextMenuTrigger` stays intentionally thin so it can wrap an existing card, row,
canvas, or editor surface without adding library-owned chrome.

## Anatomy

`ContextMenuContent` is the default path. It renders the portal, positioner, and popup for you, so
the common case stays small.

```text
ContextMenu
├─ ContextMenuTrigger
└─ ContextMenuContent
   ├─ ContextMenuItem
   ├─ ContextMenuSeparator
   └─ ContextMenuSubmenu
      ├─ ContextMenuSubmenuTrigger
      │  └─ ContextMenuSubmenuTriggerIcon
      └─ ContextMenuSubmenuContent
         └─ ContextMenuItem
```

| Part                           | Role                                                                   |
| ------------------------------ | ---------------------------------------------------------------------- |
| `ContextMenu`                  | Root state machine for open state, focus management, and interactions. |
| `ContextMenuTrigger`           | Thin trigger wrapper for right click or long press.                    |
| `ContextMenuContent`           | Default popup wrapper. Renders `Portal`, `Positioner`, and `Popup`.    |
| `ContextMenuItem`              | Action row with disabled state and optional `closeOnClick`.            |
| `ContextMenuLinkItem`          | Link row with the same menu highlight and focus behavior.              |
| `ContextMenuSeparator`         | Divider between related item groups.                                   |
| `ContextMenuSubmenu*`          | Nested actions (`Trigger`, `TriggerIcon`, `Content`).                  |
| `ContextMenuGroup*` / controls | Group labels plus radio and checkbox controls.                         |

## Composition

### When to stay high-level

Stay with `ContextMenu`, `ContextMenuTrigger`, and `ContextMenuContent` for most menus. This is the
intended path for action rows, shortcut hints, navigation items, groups, and nested submenus.

`ContextMenuContent` accepts popup props plus the shared positioning props `side`, `sideOffset`,
`align`, `alignOffset`, `collisionAvoidance`, `collisionBoundary`, and `collisionPadding`.

`ContextMenuTrigger` keeps only interaction and focus behavior by default. Add your own `className`
when the target needs a visible surface.

```tsx
<ContextMenuContent className={styles.popup} align="start" sideOffset={12}>
  {children}
</ContextMenuContent>
```

### When to go custom

Switch to `ContextMenuPortal`, `ContextMenuBackdrop`, `ContextMenuPositioner`, and
`ContextMenuPopup` only when you need a custom DOM structure, backdrop, or direct access to portal
and positioning parts.

For nested menus, prefer `ContextMenuSubmenuContent` over plain `ContextMenuContent`. It uses
submenu-tuned defaults for `sideOffset` and `alignOffset`.

`ContextMenuCheckboxItem` and `ContextMenuRadioItem` add one local DX helper:
`indicator="start" | "end" | "none"`. Use it when the indicator column needs to move or disappear
without rewriting the row structure.

### Default props

| Surface                     | Prop          | Default                | Values / Notes                       |
| --------------------------- | ------------- | ---------------------- | ------------------------------------ |
| `ContextMenuContent`        | `sideOffset`  | `8`                    | `number \| ({ side }) => number`     |
| `ContextMenuContent`        | `side`        | Base UI default        | `top`, `right`, `bottom`, `left`     |
| `ContextMenuContent`        | `align`       | Base UI default        | `start`, `center`, `end`             |
| `ContextMenuSubmenuContent` | `sideOffset`  | `({ side }) => 4 / -4` | Keeps nested menus visually attached |
| `ContextMenuSubmenuContent` | `alignOffset` | `({ side }) => 4 / -4` | Keeps submenu alignment consistent   |
| `ContextMenuCheckboxItem`   | `indicator`   | default start layout   | `start`, `end`, `none`               |
| `ContextMenuRadioItem`      | `indicator`   | default start layout   | `start`, `end`, `none`               |

### Behavioral notes

- `ContextMenuTrigger` supports right click and long press. In product UI, make the target visually
  obvious and large enough for touch when you rely on long press.
- `ContextMenuItemShortcut` is visual only. It does not bind keyboard shortcuts.
- Use `ContextMenuLinkItem` for navigation and `ContextMenuItem` for imperative actions.
- Do not place `ContextMenuPortal`, `ContextMenuPositioner`, or `ContextMenuPopup` inside
  `ContextMenuContent`; it already renders those parts.

### Groups and Controls

Use groups, radio items, and checkbox items when the menu changes application state.

<Preview>
  <GroupsAndControlsContextMenuExample />

  <Preview.Code>
    {`
      import { useState } from "react";
      import {
        ContextMenu,
        ContextMenuCheckboxItem,
        ContextMenuCheckboxItemIndicator,
        ContextMenuContent,
        ContextMenuGroup,
        ContextMenuGroupLabel,
        ContextMenuItemText,
        ContextMenuRadioGroup,
        ContextMenuRadioItem,
        ContextMenuRadioItemIndicator,
        ContextMenuSeparator,
        ContextMenuTrigger,
      } from "moduix";
      import styles from "./context-menu-demo.module.css";

      export function GroupsAndControlsContextMenuDemo() {
        const [sortBy, setSortBy] = useState("date");
        const [showMinimap, setShowMinimap] = useState(true);
        const [showSearch, setShowSearch] = useState(true);
        const [showSidebar, setShowSidebar] = useState(false);

        return (
          <ContextMenu>
            <ContextMenuTrigger className={styles.trigger}>Right click here</ContextMenuTrigger>
            <ContextMenuContent>
              <ContextMenuGroup>
                <ContextMenuGroupLabel>Sort</ContextMenuGroupLabel>
                <ContextMenuRadioGroup value={sortBy} onValueChange={setSortBy}>
                  <ContextMenuRadioItem value="date">
                    <ContextMenuRadioItemIndicator />

                  </ContextMenuRadioItem>
                  <ContextMenuRadioItem value="name">
                    <ContextMenuRadioItemIndicator />

                  </ContextMenuRadioItem>

                  </ContextMenuRadioItem>
                </ContextMenuRadioGroup>
              </ContextMenuGroup>
              <ContextMenuSeparator />
              <ContextMenuGroup>
                <ContextMenuGroupLabel>Workspace</ContextMenuGroupLabel>

                  <ContextMenuCheckboxItemIndicator />

## Styling

Use CSS variables on the trigger, popup, rows, groups, and optional backdrop when spacing, motion,
and contextual emphasis need to change. The current public contract does not include arrow styling;
the menu surface is arrowless by design.

### Styling hooks

- All exported parts expose stable `data-slot` selectors such as `context-menu-trigger`,
  `context-menu-popup`, `context-menu-item`, `context-menu-submenu-trigger`,
  `context-menu-checkbox-item`, and `context-menu-item-shortcut`.
- The shipped styles use Base UI state attributes directly: `data-starting-style` and
  `data-ending-style` on the backdrop and popup; `data-highlighted`, `data-popup-open`, and
  `data-disabled` on interactive rows.
- `ContextMenuCheckboxItem` and `ContextMenuRadioItem` also style against
  `data-indicator-position="start" | "end" | "none"`.
- `ContextMenuTrigger render={...}` opts out of the default trigger CSS class, so custom trigger
  hosts own their full layout and surface styling.
