## API Reference

`Card` is a standalone presentational component. It does not wrap a Base UI primitive, so the API
stays intentionally small: a styled root, one compact size option, and composable parts.

## Overview

Use `Card` to group related content on a visible surface. The default path stays short:

- `Card` for the visible surface
- `CardHeader` for heading content
- `CardTitle` and `CardDescription` for the primary label
- `CardContent` for the body
- `CardFooter` for actions or secondary metadata

`CardAction` is optional and only handles the common header-side layout for a badge or compact
action.

## Anatomy

Use the following composition to build a `Card`:

```text
Card
├─ CardHeader
│  ├─ CardTitle
│  ├─ CardDescription
│  └─ CardAction (optional)
├─ CardContent
└─ CardFooter
```

```tsx
<Card>
  <CardHeader>
    <CardTitle>Billing plan</CardTitle>
    <CardDescription>Team workspace, monthly billing.</CardDescription>
  </CardHeader>
  <CardContent>...</CardContent>
  <CardFooter>...</CardFooter>
</Card>
```

| Part              | Role                                                                 |
| ----------------- | -------------------------------------------------------------------- |
| `Card`            | Root container. Supports `className`, native props, and `size="sm"`. |
| `CardHeader`      | Header layout for title, description, and action content.            |
| `CardContent`     | Main content area with card body spacing.                            |
| `CardFooter`      | Footer layout for actions or secondary content.                      |
| `CardTitle`       | Card heading. Defaults to `h3` and supports `as`.                    |
| `CardDescription` | Muted supporting text under the title.                               |
| `CardAction`      | Right-aligned header action slot.                                    |

## Public Props

`Card`, `CardHeader`, `CardContent`, `CardFooter`, and `CardAction` extend their native `div` props.
`CardDescription` extends native `p` props.

| Component   | Prop   | Type                                           | Default     |
| ----------- | ------ | ---------------------------------------------- | ----------- |
| `Card`      | `size` | `'default' \| 'sm'`                            | `'default'` |
| `CardTitle` | `as`   | `'h1' \| 'h2' \| 'h3' \| 'h4' \| 'h5' \| 'h6'` | `'h3'`      |

Every part accepts `className`. `Card` writes `size` to `data-size`; all parts expose `data-slot`
attributes for styling and tests.

## Composition

Prefer the default composition when the card is just grouping content. Add `CardAction` only when the
header needs a compact status or control aligned to the trailing edge.

If the whole surface should navigate somewhere, keep the root presentational and place a link or
button inside the content instead of turning `Card` into an interactive wrapper.

Use `size="sm"` for denser dashboards, menus, and supporting panels. Keep the default size for
standard page sections and primary summaries.

Use `className` and `--card-*` CSS variables for visual customization. The layout parts stay the same
even when the visuals change.

## UX and Accessibility

- `Card` is a `div` and has no built-in landmark, button, link, or region semantics.
- Use `CardTitle as` when the surrounding document outline needs a heading level other than `h3`.
- Keep the root presentational. If a card needs navigation or actions, place explicit links or
  buttons inside it.
- `CardAction` is visual layout only; it does not connect controls to the title or description.
- There are no hover, open, checked, disabled, read-only, keyboard, or focus-management states because
  the component is not interactive by itself.
