## Choosing the right component

Use `NumberField` when the value is truly numeric and the control should parse, step, or format that
number for the user.

- Choose **`Input`** for plain text-like values, even when they happen to contain digits.
- Choose **`InputGroup`** when the field needs inline addons or actions, but not numeric parsing or stepping.
- Choose **`NumberField`** when users should type, increment, decrement, or scrub a numeric value.
- Choose **`Textarea`** when the content can span multiple lines.

## Basic

Start with `NumberField` inside `Field` when the UI needs a label, validation wiring, and the
default decrement/input/increment composition.

| Part                         | Role                                                                 |
| ---------------------------- | -------------------------------------------------------------------- |
| `NumberField`                | High-level number field with the default stepper group built in.     |
| `NumberFieldRoot`            | Low-level root state and form integration.                           |
| `NumberFieldScrubArea`       | Optional drag target for changing the value.                         |
| `NumberFieldScrubAreaCursor` | Optional scrub cursor affordance. Renders a default icon by default. |
| `NumberFieldGroup`           | Visual wrapper for the buttons and managed input.                    |
| `NumberFieldDecrement`       | Button that decreases the current value.                             |
| `NumberFieldInput`           | Editable numeric input.                                              |
| `NumberFieldIncrement`       | Button that increases the current value.                             |

## Composition

### When to stay high-level

Stay with `NumberField` for most numeric form controls. It already renders the default
decrement/input/increment group and keeps the Base UI root behavior intact for `value`,
`defaultValue`, `onValueChange`, `min`, `max`, `step`, `required`, `disabled`, `readOnly`, and
`format`.

Use `children` only for extra visible parts that should render above the built-in group, such as
`NumberFieldScrubArea`.

Keep `decrementLabel` and `incrementLabel` only for the built-in icon buttons rendered by
`NumberField`.

### When to go custom

Switch to `NumberFieldRoot` when the layout, icons, or rendered structure need explicit control.
That path keeps the same numeric behavior, but makes you responsible for rendering
`NumberFieldGroup`, `NumberFieldDecrement`, `NumberFieldInput`, and `NumberFieldIncrement`
yourself.

Use custom composition for cases such as:

- replacing the default plus and minus icons
- changing where the scrub affordance sits relative to the input
- applying different classes to the input and buttons independently

### Default props

| Prop             | Default            | Notes                                               |
| ---------------- | ------------------ | --------------------------------------------------- |
| `decrementLabel` | `"Decrease value"` | Accessible label for the built-in decrement button. |
| `incrementLabel` | `"Increase value"` | Accessible label for the built-in increment button. |

### Behavioral notes

- `id` on `NumberField` or `NumberFieldRoot` becomes the managed input id, so `Field.Label htmlFor={id}` works as expected.
- Use `value` with `onValueChange` for controlled numeric state, or `defaultValue` for uncontrolled usage.
- The empty state is `null`, not an empty string.
- `inputRef` belongs on the root when external code needs imperative access to the real input element.
- `allowWheelScrub` is opt-in. Keep it off unless wheel-based value changes are intentionally part of the workflow.

## Examples

### Controlled

Use a controlled value when adjacent UI should react to the current number immediately.

<Preview>
  <ControlledNumberFieldExample />

  <Preview.Code>
    {`
      import { Field, NumberField } from "moduix";
      import { useId, useState } from "react";

      export function ControlledNumberFieldDemo() {
        const id = useId();
        const [value, setValue] = useState(24 as number | null);

        return (
          <div>
            <Field>
              <Field.Label htmlFor={id}>Controlled value</Field.Label>
              <NumberField id={id} value={value} onValueChange={setValue} />
            </Field>
            <span>Current value: {value ?? "empty"}</span>
          </div>
        );
      }
    `}

  </Preview.Code>
</Preview>

### Styling hooks

The useful public hooks are:

- `className` on `NumberField`, `NumberFieldRoot`, `NumberFieldScrubArea`, `NumberFieldScrubAreaCursor`, `NumberFieldGroup`, `NumberFieldDecrement`, `NumberFieldInput`, and `NumberFieldIncrement`
- `data-slot="number-field-root"`, `number-field-scrub-area`, `number-field-scrub-area-cursor`, `number-field-group`, `number-field-decrement`, `number-field-input`, and `number-field-increment`
- `data-disabled` and `data-readonly` on the scrub area, input, and step buttons
- `data-invalid` on the input and step buttons
- additional Base UI field-state attributes such as `data-valid`, `data-dirty`, `data-touched`, `data-filled`, and `data-focused` when consumer styling needs them
