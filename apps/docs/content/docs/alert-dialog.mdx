## Choosing the right component

Use `AlertDialog` when the user must explicitly confirm a destructive, irreversible, or blocking
decision before continuing.

- Choose **`AlertDialog`** for modal confirmations such as delete, discard, archive, or reset actions.
- Choose **`Dialog`** for general modal content that is not strictly a confirmation flow.
- Choose **`Alert`** for inline feedback that should stay in the page layout and not block the workflow.

## Basic

Start with `AlertDialog`, `AlertDialogTrigger`, and `AlertDialogContent`. This is the recommended
path for destructive confirmation flows and other blocking decisions.

## Anatomy

`AlertDialogContent` is the high-level surface. It already renders the portal, backdrop, viewport,
and popup for the common case.

```text
AlertDialog
├─ AlertDialogTrigger
└─ AlertDialogContent
   ├─ AlertDialogHeader
   │  ├─ AlertDialogTitle
   │  └─ AlertDialogDescription
   ├─ AlertDialogBody (optional)
   └─ AlertDialogFooter
      ├─ AlertDialogCancel
      └─ AlertDialogAction
```

| Part                     | Role                                                                                              |
| ------------------------ | ------------------------------------------------------------------------------------------------- |
| `AlertDialog`            | Root state machine. Manages open state, controlled/uncontrolled behavior, and close interactions. |
| `AlertDialogTrigger`     | Opens the dialog with the default trigger styling.                                                |
| `AlertDialogContent`     | Main dialog surface. Renders popup plus portal/backdrop/viewport infrastructure.                  |
| `AlertDialogHeader`      | Layout group for title and description.                                                           |
| `AlertDialogTitle`       | Required accessible title.                                                                        |
| `AlertDialogDescription` | Optional supporting copy.                                                                         |
| `AlertDialogBody`        | Optional long-form or scrollable content area.                                                    |
| `AlertDialogFooter`      | Layout container for decision actions.                                                            |
| `AlertDialogCancel`      | Secondary close action with the default cancel styling.                                           |
| `AlertDialogAction`      | Primary confirm action with the default action styling.                                           |

## Composition

### When to stay high-level

Stay with `AlertDialog`, `AlertDialogTrigger`, and `AlertDialogContent` for most confirmation
dialogs.

Use `AlertDialog` for state: `open`, `defaultOpen`, `onOpenChange`, or `handle`.

Use `AlertDialogContent` for the visible surface. It is intentionally the high-level path, so most
consumers only need `className` there plus composition inside the content.

`AlertDialogAction` and `AlertDialogCancel` are small DX helpers for the common close-on-click
workflow.

### When to go custom

Switch to `AlertDialogPortal`, `AlertDialogBackdrop`, `AlertDialogViewport`, and
`AlertDialogPopup` only when the default overlay structure is not enough.

If the primary action must wait for async work, use a regular `Button` in `AlertDialogFooter` and
close the dialog from controlled state or `createAlertDialogHandle()` only after the request
succeeds.

### Default props

| Prop                 | Default | Values / Notes                                       |
| -------------------- | ------- | ---------------------------------------------------- |
| `defaultOpen`        | `false` | Initial uncontrolled open state                      |
| `open`               | -       | Controlled open state                                |
| `AlertDialogContent` | -       | Always renders portal, backdrop, viewport, and popup |
| `AlertDialogAction`  | -       | Closes immediately on click                          |
| `AlertDialogCancel`  | -       | Closes immediately on click                          |

### Behavioral notes

- `AlertDialogContent` is always modal and always renders a backdrop.
- Focus is trapped while the dialog is open and returns to the trigger or previous focus target on close.
- Backdrop clicks and `Escape` request close through `onOpenChange`. Keep the dialog controlled if
  you need to block closing for pending work or validation.
- `AlertDialogAction` is a close button, not an async submit helper.
- Use concise titles and descriptions. The action label should name the destructive action directly.

## Examples

### Controlled

Use controlled mode when another workflow owns the open state.

<Preview>
  <ControlledAlertDialogExample />

  <Preview.Code>
    {`
      import {
        AlertDialog,
        AlertDialogAction,
        AlertDialogCancel,
        AlertDialogContent,
        AlertDialogDescription,
        AlertDialogFooter,
        AlertDialogHeader,
        AlertDialogTitle,
        Button,
      } from "moduix";
      import { Fragment, useState } from "react";

      export function ControlledAlertDialogDemo() {
        const [open, setOpen] = useState(false);

        return (
          <Fragment>
            <Button type="button" onClick={() => setOpen(true)}>
              Open controlled dialog
            </Button>
            <AlertDialog open={open} onOpenChange={setOpen}>
              <AlertDialogContent>
                <AlertDialogHeader>
                  <AlertDialogTitle>Publish changes?</AlertDialogTitle>
                  <AlertDialogDescription>
                    This will make the latest version visible to all users.
                  </AlertDialogDescription>
                </AlertDialogHeader>
                <AlertDialogFooter>
                  <AlertDialogCancel>Back to editing</AlertDialogCancel>
                  <AlertDialogAction>Publish</AlertDialogAction>
                </AlertDialogFooter>
              </AlertDialogContent>
            </AlertDialog>
          </Fragment>
        );
      }
    `}

  </Preview.Code>
</Preview>

### Async Confirmation

Keep the dialog open during async work and close it only after the request succeeds.

<Preview>
  <AsyncAlertDialogExample />

  <Preview.Code>
    {`
      import {
        AlertDialog,
        AlertDialogBody,
        AlertDialogCancel,
        AlertDialogContent,
        AlertDialogDescription,
        AlertDialogFooter,
        AlertDialogHeader,
        AlertDialogTitle,
        AlertDialogTrigger,
        Button,
      } from "moduix";
      import { useState } from "react";

      export function AsyncAlertDialogDemo() {
        const [open, setOpen] = useState(false);
        const [pending, setPending] = useState(false);
        const [error, setError] = useState("");

        const handleArchive = async () => {
          setPending(true);
          setError("");

          try {
            await archiveWorkspace();
            setOpen(false);
          } catch {
            setError("Workspace could not be archived. Review the warning and try again.");
          } finally {
            setPending(false);
          }
        };

        return (
          <AlertDialog open={open} onOpenChange={setOpen}>
            <AlertDialogTrigger>Archive workspace</AlertDialogTrigger>
            <AlertDialogContent>
              <AlertDialogHeader>
                <AlertDialogTitle>Archive workspace?</AlertDialogTitle>
                <AlertDialogDescription>
                  Keep the dialog open while the request is pending, then close it only after success.
                </AlertDialogDescription>
              </AlertDialogHeader>
              {error ? (
                <AlertDialogBody>
                  <p>{error}</p>
                </AlertDialogBody>
              ) : null}
              <AlertDialogFooter>
                <AlertDialogCancel disabled={pending}>Cancel</AlertDialogCancel>
                <Button type="button" disabled={pending} onClick={handleArchive}>
                  {pending ? "Archiving..." : "Archive"}
                </Button>
              </AlertDialogFooter>
            </AlertDialogContent>
          </AlertDialog>
        );
      }
    `}

  </Preview.Code>
</Preview>

### Handle

Use `createAlertDialogHandle()` for detached triggers or imperative opens.

<Preview>
  <AlertDialogHandleExample />

  <Preview.Code>
    {`
      import {
        AlertDialog,
        AlertDialogAction,
        AlertDialogCancel,
        AlertDialogContent,
        AlertDialogDescription,
        AlertDialogFooter,
        AlertDialogHeader,
        AlertDialogTitle,
        AlertDialogTrigger,
        Button,
        createAlertDialogHandle,
      } from "moduix";
      import { Fragment, useMemo } from "react";

      export function AlertDialogHandleDemo() {
        const alertDialogHandle = useMemo(() => createAlertDialogHandle(), []);

        return (
          <Fragment>
            <AlertDialogTrigger handle={alertDialogHandle}>Open from detached trigger</AlertDialogTrigger>
            <Button type="button" onClick={() => alertDialogHandle.open(null)}>
              Open programmatically
            </Button>

            <AlertDialog handle={alertDialogHandle}>
              <AlertDialogContent>
                <AlertDialogHeader>
                  <AlertDialogTitle>Delete workspace?</AlertDialogTitle>
                  <AlertDialogDescription>
                    This alert dialog is connected via createAlertDialogHandle().
                  </AlertDialogDescription>
                </AlertDialogHeader>
                <AlertDialogFooter>
                  <AlertDialogCancel>Cancel</AlertDialogCancel>
                  <AlertDialogAction>Delete</AlertDialogAction>
                </AlertDialogFooter>
              </AlertDialogContent>
            </AlertDialog>
          </Fragment>
        );
      }
    `}

## Styling

Use `className` on `AlertDialogContent` for most visual overrides. It applies the popup styling and
keeps the default overlay structure intact.

For advanced overlay customization, move to `AlertDialogPortal`, `AlertDialogBackdrop`,
`AlertDialogViewport`, and `AlertDialogPopup`.

### Styling hooks

- `data-slot="alert-dialog-trigger"`, `alert-dialog-portal`, `alert-dialog-backdrop`,
  `alert-dialog-viewport`, `alert-dialog-popup`, `alert-dialog-header`, `alert-dialog-title`,
  `alert-dialog-description`, `alert-dialog-body`, `alert-dialog-footer`, and `alert-dialog-close`
  are stable selectors on the exported parts.
- Base UI also adds state attributes such as `data-open`, `data-closed`, `data-popup-open`,
  `data-disabled`, `data-nested`, `data-nested-dialog-open`, `data-starting-style`, and
  `data-ending-style` on the relevant overlay parts.
