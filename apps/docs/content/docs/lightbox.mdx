## Choosing the right component

Use `Lightbox` when the overlay is primarily about previewing imagery.

- Choose **`Lightbox`** for zoomed image previews, media galleries, and image-first overlays.
- Choose **`Dialog`** when the surface contains general application UI such as forms, actions, or
  mixed content that is not centered around one preview image.

## Basic

Start with `Lightbox`, `LightboxImage`, and `LightboxContent`. This is the recommended path for a
thumbnail that opens a larger preview.

## Anatomy

The common consumer path stays short:

```text
Lightbox
├─ LightboxTrigger
│  └─ LightboxImage (optional helper)
└─ LightboxContent
   ├─ LightboxPortal
   │  ├─ LightboxBackdrop
   │  └─ LightboxViewport
   │     ├─ LightboxCloseButton (optional)
   │     └─ LightboxPopup
   │        └─ LightboxFrame
   │           └─ preview content
```

`LightboxContent` already renders the default portal, backdrop, viewport, popup, frame, and close
button, so most usage stays at `Lightbox`, `LightboxImage`, and `LightboxContent`.

| Part                  | Role                                                             |
| --------------------- | ---------------------------------------------------------------- |
| `Lightbox`            | Root open state and dialog behavior.                             |
| `LightboxImage`       | Image trigger helper for the common thumbnail-to-preview flow.   |
| `LightboxTrigger`     | Opens the overlay from any custom trigger composition.           |
| `LightboxContent`     | High-level wrapper for the default overlay structure.            |
| `LightboxCloseButton` | Default close control rendered inside the viewport.              |
| `LightboxGallery`     | Delegated image capture for CMS-rendered or unknown image lists. |

## Composition

### When to stay high-level

Stay with `Lightbox`, `LightboxImage`, and `LightboxContent` for most preview flows.

`Lightbox` keeps the Base UI dialog root contract intact, so pass `open`, `defaultOpen`,
`onOpenChange`, `onOpenChangeComplete`, `modal`, and `handle` directly to it.

`LightboxImage` is the narrow helper for one trigger image opening one preview image. It stores the
active image for the root, reuses `src` by default, and switches to `fullSrc` when you provide a
larger asset. When `fullSrc` is present, the helper preloads it on pointer hover and keyboard
focus.

`LightboxContent` keeps the default path compact. When `children` are omitted, it renders the
active image from `LightboxImage` inside the default overlay structure.

### When to go custom

Use `LightboxTrigger` when the trigger is not an image.

Use `LightboxGallery` when the images come from delegated markup that you do not compose manually.
It reads `data-lightbox-src` first, then falls back to `currentSrc` or `src`, and it can target a
focusable wrapper with `selector` when the image is nested inside a button, card, or link-like
control.

Drop to `LightboxPortal`, `LightboxBackdrop`, `LightboxViewport`, `LightboxPopup`,
`LightboxFrame`, and `LightboxClose` when you need a different overlay structure, dismissal
behavior, or popup layout.

### Default props

| Part              | Prop                  | Default       | Values / Notes                                   |
| ----------------- | --------------------- | ------------- | ------------------------------------------------ |
| `LightboxContent` | `showCloseButton`     | `true`        | Shows the built-in `LightboxCloseButton`         |
| `LightboxContent` | `closeOnContentClick` | `true`        | Closes when the preview surface is clicked       |
| `LightboxContent` | `closeLabel`          | `Close image` | Accessible label for the close control           |
| `LightboxGallery` | `selector`            | `img`         | Any selector supported by `Element.closest(...)` |

### Behavioral notes

- Set `closeOnContentClick={false}` when the content contains links, buttons, or other interactive
  children.
- `LightboxGallery` supports keyboard activation only when the matched element is already
  keyboard-focusable.
- For detached or programmatic open flows, pass a Base UI `handle` to `Lightbox` or use the
  exported `createLightboxHandle()`.

### Dynamic Gallery Capture

Use `LightboxGallery` for delegated galleries where the image nodes come from CMS or other external
markup.

<Preview>
  <DynamicLightboxGalleryExample />

  <Preview.Code>
    {`
      import { LightboxGallery } from "moduix";
      import { Fragment, useRef } from "react";

      export function DynamicGalleryDemo() {
        const rootRef = useRef<HTMLDivElement | null>(null);

        return (
          <Fragment>
            <div ref={rootRef} className="lightbox-gallery">
              <button type="button" className="lightbox-gallery-item">
                <img
                  src={images.mountain}
                  data-lightbox-src={images.mountainLarge}
                  alt="Mountain landscape"
                  className="lightbox-gallery-image"
                />
              </button>
              <button type="button" className="lightbox-gallery-item">
                <img
                  src={images.sea}
                  data-lightbox-src={images.seaLarge}
                  alt="Sea at sunset"
                  className="lightbox-gallery-image"
                />
              </button>
              <button type="button" className="lightbox-gallery-item">
                <img
                  src={images.forest}
                  data-lightbox-src={images.forestLarge}
                  alt="Forest and mountain road"
                  className="lightbox-gallery-image"
                />
              </button>
            </div>
            <LightboxGallery rootRef={rootRef} selector="button" />
          </Fragment>
        );
      }
    `}

  </Preview.Code>

  <Preview.CSS>

Use a focusable wrapper when delegated gallery items must stay keyboard-accessible. `LightboxGallery`
does not add `tabIndex`, button semantics, or labels to arbitrary markup.

### Custom Composition

Use the low-level parts when the default wrapper is too opinionated for the popup structure or the
dismissal behavior.

<Preview>
  <CustomizedLightboxExample />

  <Preview.Code>
    {`
      import {
        Lightbox,
        LightboxBackdrop,
        LightboxClose,
        LightboxCloseButton,
        LightboxFrame,
        LightboxImage,
        LightboxPopup,
        LightboxPortal,
        LightboxViewport,
      } from "moduix";

      export function CustomizedLightboxDemo() {
        return (
          <Lightbox>
            <LightboxImage
              src={thumbnail}
              alt="Road through forest"
              className="lightbox-demo-image"
            />
            <LightboxPortal>
              <LightboxBackdrop className="lightbox-custom-backdrop" />
              <LightboxViewport>
                <LightboxCloseButton className="lightbox-custom-close" aria-label="Close preview" />
                <LightboxPopup className="lightbox-custom-popup">
                  <LightboxFrame>
                    <LightboxClose nativeButton={false} render={<div />}>
                      <img src={fullSize} alt="Road through forest" />
                    </LightboxClose>
                  </LightboxFrame>
                </LightboxPopup>
              </LightboxViewport>
            </LightboxPortal>
          </Lightbox>
        );
      }
    `}

## Styling

Use `className` on `LightboxImage`, `LightboxTrigger`, `LightboxContent`, and the low-level parts
for most visual overrides. Move to low-level composition when the overlay structure itself needs to
change.

### Styling hooks

- `data-slot="lightbox-trigger"`, `lightbox-portal`, `lightbox-backdrop`, `lightbox-viewport`,
  `lightbox-popup`, `lightbox-close`, `lightbox-close-button`, `lightbox-frame`, `lightbox-image`,
  and `lightbox-content-image` are stable selectors on the exported parts.
- Base UI also adds state attributes such as `data-open`, `data-closed`, `data-popup-open`,
  `data-nested`, `data-nested-dialog-open`, `data-starting-style`, and `data-ending-style` on the
  relevant overlay parts.
