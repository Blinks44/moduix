# Dark mode

moduix ships light tokens by default and a dark token set behind the `dark` class. Import the
foundation stylesheet once, then add or remove that class where your application owns its theme.

```tsx
import type { ReactNode } from 'react';

import '@moduix/react/style.css';
```

## Apply the theme at the document root

Put `dark` on `<html>` for an application-wide theme. This is the recommended setup because
portalled components such as Dialog, Popover, and Select also inherit the same tokens.

```tsx
import type { ReactNode } from 'react';

export function Document({ children, theme }: { children: ReactNode; theme: 'light' | 'dark' }) {
  return <html className={theme === 'dark' ? 'dark' : undefined}>{children}</html>;
}
```

The class can be toggled by any theme solution. moduix does not add a provider or persist a user
preference for you.

If your app renders on the server, put the resolved class on the first HTML response. It prevents a
brief light theme before client JavaScript applies the saved preference.

## Scope a preview or embedded surface

You can place `dark` on a regular wrapper when only that subtree needs the dark tokens:

```tsx
<section className="dark">
  <Card>Dark preview</Card>
</section>
```

This is useful for a settings preview or an embedded widget. Portalled content is outside that
wrapper, so keep the class on `<html>`, `body`, or your portal container when the subtree opens
overlays.

## Customize both themes

Override the same semantic tokens in `:root` and `.dark`. Components read semantic aliases such as
`--color-background`, `--color-foreground`, and `--color-border`, so the values stay consistent
across the library.

```css
:root {
  --background: oklch(0.99 0.004 285);
  --primary: oklch(0.5 0.18 260);
}

.dark {
  --background: oklch(0.16 0.01 285);
  --primary: oklch(0.76 0.13 260);
}
```

Use [Tokens](/docs/tokens) for the complete token reference and component pages for local visual
overrides.
