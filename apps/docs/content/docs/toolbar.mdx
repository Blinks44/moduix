## Choosing the right component

Use `Toolbar` when related controls should share roving focus and behave like one compact action
region.

- Choose **`ToggleGroup`** when the main problem is shared pressed-state selection.
- Choose a plain layout container when the row does not need toolbar keyboard behavior.
- Choose **`Toolbar`** for compact editor controls, document actions, and small mixed-action rows.

## Basic

Start with `Toolbar`, `ToolbarGroup`, `ToolbarButton`, and `ToolbarSeparator`. This is the
recommended path for ordinary action rows.

## Anatomy

The exported parts cover the common toolbar surface directly:

```text
Toolbar[aria-label]
├─ ToolbarGroup[aria-label]
│  └─ ToolbarButton
├─ ToolbarSeparator
├─ ToolbarButton
├─ ToolbarLink
└─ ToolbarInput
```

| Part               | Role                                                         |
| ------------------ | ------------------------------------------------------------ |
| `Toolbar`          | Root roving-focus region with shared orientation and layout. |
| `ToolbarGroup`     | Optional subgroup for related controls.                      |
| `ToolbarButton`    | Toolbar-aware button and trigger composition surface.        |
| `ToolbarLink`      | Toolbar-aware anchor.                                        |
| `ToolbarInput`     | Input that participates in toolbar keyboard behavior.        |
| `ToolbarSeparator` | Visual split between toolbar clusters.                       |

## Composition

### When to stay high-level

Stay with `Toolbar`, `ToolbarGroup`, `ToolbarButton`, `ToolbarLink`, `ToolbarInput`, and
`ToolbarSeparator` for most action rows.

Use `ToolbarGroup` only when a subset of items needs its own accessible name. Use
`ToolbarSeparator` to split clusters. Keep `ToolbarInput` rare, compact, and usually at the end of
horizontal toolbars.

### When to go custom

Use `ToolbarButton render={...}` when another primitive should own the interactive surface without
nesting interactive elements.

That is the main escape hatch for:

- `Toggle` inside a toolbar item
- popup triggers such as `SelectTrigger`
- other trigger-like primitives that should keep their own semantics and state

### Default props

| Prop                           | Default                   | Values                                                             |
| ------------------------------ | ------------------------- | ------------------------------------------------------------------ |
| `orientation`                  | `horizontal`              | `horizontal`, `vertical`                                           |
| `disabled`                     | `false`                   | `true`, `false`                                                    |
| `ToolbarSeparator.orientation` | auto-opposite orientation | `vertical` in horizontal toolbar, `horizontal` in vertical toolbar |

### Behavioral notes

- `Toolbar` needs an accessible name through `aria-label` or `aria-labelledby`.
- Icon-only `ToolbarButton` items need their own accessible label.
- In horizontal toolbars, place text input after the main button cluster to avoid fighting cursor
  and roving-focus arrow keys.
- Use `ToolbarButton render={...}` for toggles and popup triggers so one interactive surface owns
  the semantics.
- `ToolbarLink` is for actual navigation or related metadata. If the item behaves like an action,
  use `ToolbarButton`.

### Toggle Composition

Compose `ToolbarButton` with `Toggle` when items need pressed state but should still live inside the
toolbar interaction model.

<Preview>
  <ToolbarToggleGroupExample />

  <Preview.Code>
    {`
      import {
        BellIcon,
        StarIcon,
        Toggle,
        ToggleGroup,
        Toolbar,
        ToolbarButton,
        ToolbarGroup,
        ToolbarSeparator,
      } from "moduix";

      export function ToolbarToggleGroupDemo() {
        return (
          <Toolbar aria-label="Editor formatting">
            <ToggleGroup multiple defaultValue={["bold"]} aria-label="Text formatting" variant="ghost">
              <ToolbarButton render={<Toggle variant="ghost" />} value="bold" aria-label="Bold">
                <strong>B</strong>
              </ToolbarButton>

                <em>I</em>

                <u>U</u>

            <ToolbarGroup aria-label="Insert">

              </ToolbarButton>

### Select Trigger Composition

Use `render` to turn a toolbar button into a popup trigger while keeping popup composition on the
popup primitive itself.

<Preview>
  <ToolbarSelectTriggerExample />

  <Preview.Code>
    {`
      import {
        BellIcon,
        ChevronUpDownIcon,
        Select,
        SelectContent,
        SelectIcon,
        SelectItem,
        SelectItemIndicator,
        SelectItemText,
        SelectList,
        SelectTrigger,
        SelectValue,
        Toolbar,
        ToolbarButton,
        ToolbarLink,
        ToolbarSeparator,
      } from "moduix";

      export function ToolbarSelectTriggerDemo() {
        return (
          <Toolbar aria-label="Text properties">
            <Select defaultValue="Inter">
              <ToolbarButton render={<SelectTrigger />} aria-label="Font family">
                <SelectValue />
                <SelectIcon>
                  <ChevronUpDownIcon />
                </SelectIcon>
              </ToolbarButton>
              <SelectContent>
                <SelectList>
                  {fonts.map((font) => (
                    <SelectItem key={font} value={font}>
                      <SelectItemIndicator />
                      <SelectItemText>{font}</SelectItemText>
                    </SelectItem>
                  ))}
                </SelectList>
              </SelectContent>
            </Select>

            <ToolbarSeparator />
            <ToolbarButton aria-label="Notifications">
              <BellIcon />
            </ToolbarButton>
            <ToolbarLink href="#">Edited 51m ago</ToolbarLink>
          </Toolbar>
        );
      }
    `}

  </Preview.Code>

  <Preview.Data>
    {`
      const fonts = ["Inter", "Arial", "Helvetica", "Georgia"];
    `}

### Styling hooks

Use the part that owns the visual concern:

- `Toolbar` for overall surface, spacing, orientation, and shared color context
- `ToolbarGroup` for subgroup spacing and vertical stacking
- `ToolbarButton` and `ToolbarLink` for interactive control visuals
- `ToolbarInput` for inline field styling
- `ToolbarSeparator` for divider length, thickness, and color

The main public hooks include `data-slot="toolbar-root"`, `toolbar-group`, `toolbar-button`,
`toolbar-link`, `toolbar-input`, and `toolbar-separator`.

Useful state and layout hooks include:

- `data-orientation` on `Toolbar`, `ToolbarGroup`, and `ToolbarSeparator`
- `data-disabled` and `[disabled]` on `ToolbarButton`, `ToolbarLink`, and `ToolbarInput`
- `data-pressed` and `data-popup-open` on `ToolbarButton` when it composes toggles or popup
  triggers
