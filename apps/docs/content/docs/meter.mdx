## Choosing the right component

Use `Meter` when the value is always known and represents a bounded measurement such as storage
usage, quota consumption, or service health.

- Choose **`Meter`** for known values that describe the current level of something.
- Choose **`Progress`** when work can be indeterminate or the UI represents ongoing completion.

## Basic

Start with `Meter`, `MeterLabel`, and `MeterValue`. This is the recommended path when the default
track and indicator structure already fits.

## Anatomy

The normal consumer path is short:

```text
Meter[value]
├─ MeterLabel
├─ MeterValue
└─ track
   └─ indicator
```

| Part             | Role                                                                         |
| ---------------- | ---------------------------------------------------------------------------- |
| `Meter`          | High-level root wrapper with formatting, ARIA state, and default bar output. |
| `MeterLabel`     | Accessible label associated with the meter root.                             |
| `MeterValue`     | Formatted value output, with optional custom text rendering.                 |
| `track`          | Default visual rail rendered by `Meter`.                                     |
| `indicator`      | Default filled bar rendered inside the track.                                |
| `MeterRoot`      | Low-level root for explicit composition and custom structure.                |
| `MeterTrack`     | Explicit track part for custom layouts.                                      |
| `MeterIndicator` | Explicit indicator part for custom layouts.                                  |

## Composition

### When to stay high-level

Stay with `Meter` for most dashboards, forms, and settings panels where the default track and
indicator belong after the visible text content.

Use `Meter` or `MeterRoot` for the public root props: `value`, `min`, `max`, `locale`, `format`,
`aria-valuetext`, and `getAriaValueText`.

### When to go custom

Switch to `MeterRoot`, `MeterTrack`, and `MeterIndicator` when the bar needs a different DOM
position, separate part-level selectors, or custom layout around the default structure.

### Default props

| Prop     | Default | Values                     |
| -------- | ------- | -------------------------- |
| `value`  | -       | Required `number`          |
| `min`    | `0`     | Any `number`               |
| `max`    | `100`   | Any `number`               |
| `format` | -       | `Intl.NumberFormatOptions` |
| `locale` | runtime | `Intl.LocalesArgument`     |

### Behavioral notes

- `Meter` always expects a concrete `number`. It does not support `null` or indeterminate state.
- Use `Progress` when the current amount is unknown or the UI represents ongoing completion.
- Prefer `MeterValue` for visible numeric output so formatting stays aligned with the root value.
- Use `aria-valuetext` or `getAriaValueText` when the numeric value needs domain-specific wording
  for assistive technology.

### Controlled

Keep the value in React state when the meter mirrors another control.

<Preview>
  <ControlledMeterExample />

  <Preview.Code>
    {`
      import {
        Meter,
        MeterLabel,
        MeterValue,
      } from "moduix";
      import { useState } from "react";

      export function ControlledMeterDemo() {
        const [value, setValue] = useState(45);

        return (
          <div className="stack">
            <Meter value={value}>
              <MeterLabel>Capacity</MeterLabel>
              <MeterValue />
            </Meter>
            <input
              className="range"
              type="range"
              min={0}
              max={100}
              value={value}
              onChange={(event) => {
                setValue(Number(event.target.value));
              }}
            />
          </div>
        );
      }
    `}

  </Preview.Code>

  <Preview.CSS>
    {`
      .stack {
        display: grid;
        gap: var(--spacing-4);
      }

      .range {
        width: 12rem;
      }
    `}

  </Preview.CSS>
</Preview>

### Locale And Format

Use `locale`, `format`, and a custom `MeterValue` child function for localized value text.

<Preview>
  <LocaleAndFormatMeterExample />

  <Preview.Code>
    {`
      import {
        Meter,
        MeterLabel,
        MeterValue,
      } from "moduix";

      export function LocaleAndFormatMeterDemo() {
        return (
          <Meter
            value={0.64}
            min={0}
            max={1}
            locale="de-DE"
            format={{ style: "percent", maximumFractionDigits: 0 }}
          >
            <MeterLabel>Storage usage</MeterLabel>
            <MeterValue>{(formattedValue) => \`\${formattedValue} belegt\`}</MeterValue>
          </Meter>
        );
      }
    `}

  </Preview.Code>
</Preview>

### Custom Value Text

Use `getAriaValueText` when visible and accessible wording should stay aligned but should not use
the default number formatting alone.

<Preview>
  <CustomValueTextMeterExample />

  <Preview.Code>
    {`
      import {
        Meter,
        MeterLabel,
        MeterValue,
      } from "moduix";

      export function CustomValueTextMeterDemo() {
        return (
          <Meter
            value={68}
            getAriaValueText={(formattedValue, value) =>
              \`\${formattedValue} percent of storage used (\${value} of 100)\`
            }
          >
            <MeterLabel>Storage usage</MeterLabel>
            <MeterValue>{(formattedValue) => \`\${formattedValue}% used\`}</MeterValue>
          </Meter>
        );
      }
    `}

  </Preview.Code>
</Preview>

### Styling hooks

Use the part that owns the visual concern:

- `Meter` or `MeterRoot` for layout width, gap, and shared text color
- `MeterLabel` and `MeterValue` for typography overrides
- `MeterTrack` for rail background, border, and radius
- `MeterIndicator` for fill color, gradient, and transition overrides

The main public hooks are `data-slot="meter-root"`, `data-slot="meter-label"`,
`data-slot="meter-value"`, `data-slot="meter-track"`, and `data-slot="meter-indicator"`.
