## Choosing the right component

Use `PasswordInput` when a single secure value should stay masked by default while still allowing a
user-controlled reveal action.

- Choose **`Input`** for plain text-like fields without a visibility toggle.
- Choose **`PasswordInput`** for passwords, API keys, and other secure strings.
- Choose **`InputGroup`** when the field needs a custom inline action that is not password visibility.

## Anatomy

```tsx
<PasswordInput>
  <PasswordInput.Label />
  <PasswordInput.Field />
</PasswordInput>
```

For advanced customization, replace `PasswordInput.Field` with its explicit Ark parts:

```tsx
<PasswordInput>
  <PasswordInput.Label />
  <PasswordInput.Control>
    <PasswordInput.Input />
    <PasswordInput.VisibilityTrigger>
      <PasswordInput.Indicator />
    </PasswordInput.VisibilityTrigger>
  </PasswordInput.Control>
</PasswordInput>
```

| Part                                   | Role                                                                           |
| -------------------------------------- | ------------------------------------------------------------------------------ |
| `PasswordInput` / `PasswordInput.Root` | Owns password visibility state and field state.                                |
| `PasswordInput.RootProvider`           | Renders the same root part from a `usePasswordInput` state object.             |
| `PasswordInput.Label`                  | Labels the input and receives required, disabled, invalid, and readonly state. |
| `PasswordInput.Field`                  | Renders the default control, input, visibility trigger, and indicator.         |
| `PasswordInput.Control`                | Positions the input and visibility trigger together.                           |
| `PasswordInput.Input`                  | Native input part whose `type` switches between `password` and `text`.         |
| `PasswordInput.VisibilityTrigger`      | Button that toggles the visibility state.                                      |
| `PasswordInput.Indicator`              | Shows visible/hidden content; moduix supplies default icons.                   |

## Composition

### Recommended composition

Use `PasswordInput.Field` for the default control, input, visibility trigger, and indicator. Pass
root visibility, form-state, ids, translations, and password-manager props to `PasswordInput`.

### When to go custom

Use `Control`, `Input`, `VisibilityTrigger`, and `Indicator` directly when the structure, icons, or
part styling must differ from `PasswordInput.Field`. Use `asChild` only when a part must render one
custom semantic element.

### Default props

| Prop                             | Default              | Notes                                                  |
| -------------------------------- | -------------------- | ------------------------------------------------------ |
| `autoComplete`                   | `"current-password"` | Set on the input through the root machine.             |
| `defaultVisible`                 | `false`              | Initial uncontrolled visibility state.                 |
| `visible`                        | uncontrolled         | Controlled visibility state.                           |
| `PasswordInput.Field`            | —                    | Renders the default visible parts with no local props. |
| `onVisibilityChange`             | -                    | Receives Ark details; read `details.visible`.          |
| `translations.visibilityTrigger` | Ark default          | Function used for localized visibility trigger labels. |
| `ignorePasswordManagers`         | `false`              | Adds supported password-manager ignore hints.          |

### Behavioral notes

- Control visibility with `visible` and `onVisibilityChange(details)`.
- Control the field value directly on `PasswordInput.Input` with native `value` and `onChange`.
- Use root props such as `name`, `required`, `disabled`, `readOnly`, `invalid`, `ids`, and
  `translations` directly from Ark UI.
- Use `usePasswordInput` with `PasswordInput.RootProvider` when another component needs the same
  visibility state outside the rendered tree.
- `PasswordInput.Input`, `PasswordInput.VisibilityTrigger`, and `PasswordInput.Indicator` expose
  `data-state="visible" | "hidden"` for styling.

`--moduix-password-input-*` tokens layered over the shared input contract.

### Styling Hooks

- Root and root provider expose `data-slot="password-input-root"` and
  `data-slot="password-input-root-provider"`.
- Ark parts keep `data-scope="password-input"` and `data-part` values for `root`, `label`,
  `control`, `input`, `visibility-trigger`, and `indicator`.
- State styling uses `data-state="visible" | "hidden"` on input, trigger, and indicator, plus
  `data-disabled`, `data-invalid`, `data-readonly`, and `data-required` where Ark provides them.
- Ark does not expose password-input-specific runtime CSS variables; moduix adds
  `--moduix-password-input-*` variables for visual defaults.
