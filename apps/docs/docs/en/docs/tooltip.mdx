## Choosing the right component

Use `Tooltip` for short, non-essential labels or hints that appear on hover or focus.

- Choose **`Tooltip`** for compact labels and short contextual hints.
- Choose **`Popover`** when the popup needs richer layout, actions, or interactive content.
- Choose **`HoverCard`** when the popup should expose a richer preview surface tied to a trigger.

## Basic

Start with `Tooltip`, `Tooltip.Trigger`, and `Tooltip.Body`. `Tooltip.Body` is the narrow shortcut
for the common `Tooltip.Positioner + Tooltip.Content` path. Keep an accessible name on the trigger
itself: tooltips do not replace `aria-label` or visible text.

## Anatomy

```text
Tooltip / Tooltip.Root
├─ Tooltip.Trigger
└─ Overlay subtree (automatically portalled)
   └─ Tooltip.Positioner
      └─ Tooltip.Content
         └─ content
```

Add `Tooltip.Arrow` and `Tooltip.ArrowTip` inside the content when the popup should point back to
the trigger. `Tooltip.Arrow` renders `Tooltip.ArrowTip` by default.

```text
Tooltip / Tooltip.Root
├─ Tooltip.Trigger
└─ Overlay subtree (automatically portalled)
   └─ Tooltip.Positioner
      └─ Tooltip.Content
         ├─ Tooltip.Arrow
         │  └─ Tooltip.ArrowTip
         └─ content
```

| Part                       | Role                                                                                        |
| -------------------------- | ------------------------------------------------------------------------------------------- |
| `Tooltip` / `Tooltip.Root` | Root state, controlled/uncontrolled open state, delays, positioning options, and lifecycle. |
| `Tooltip.RootProvider`     | Renders a tooltip from a `useTooltip()` state object owned outside the tree.                |
| `Tooltip.Body`             | Convenience wrapper that renders `Tooltip.Positioner` and `Tooltip.Content` together.       |
| `Tooltip.Trigger`          | Interactive anchor element that opens the tooltip on hover or focus.                        |
| `Tooltip.Positioner`       | Floating layout layer that consumes Ark positioning options and runtime CSS variables.      |
| `Tooltip.Content`          | Visible tooltip surface.                                                                    |
| `Tooltip.Arrow`            | Optional arrow wrapper.                                                                     |
| `Tooltip.ArrowTip`         | Arrow tip element; rendered by default from `Tooltip.Arrow`.                                |

## Composition

The root portals `Positioner` automatically. Use `portalled={false}` for inline rendering and `portalRef` for a custom container; structural parts remain explicit and styleable.

### Recommended composition

Use `Tooltip.Trigger` with `Tooltip.Body` for the common path. `Tooltip.Body` keeps the root-level
positioning API intact and renders `Tooltip.Positioner` with `Tooltip.Content` for you. Moduix also
keeps the explicit Ark parts available when you need full structural control.

If you're moving from shadcn, `Tooltip.Body` is the closest equivalent to the common
`TooltipContent` step, while the explicit Ark parts stay available for advanced structure.

### When to go custom

Use `asChild` on `Tooltip.Trigger` when the trigger should be a moduix `Button`, link, or another
semantic control. The child must remain a single focusable element that can support the trigger
interaction.

Use `positioning` on `Tooltip` for placement, strategy, offset, collision behavior, and the rest of
Ark's floating options. Use explicit `Tooltip.Positioner` and `Tooltip.Content` when you need
positioner-specific styling, inline context consumers that should own the content part, or a lower
level Ark-shaped structure. Use `ids` when separate tooltip parts need stable shared IDs.

### Default props

| Prop                   | Default | Notes                                                             |
| ---------------------- | ------- | ----------------------------------------------------------------- |
| `openDelay`            | `400`   | Delay before opening on hover or focus.                           |
| `closeDelay`           | `150`   | Delay before closing after hover or focus leaves.                 |
| `closeOnClick`         | `true`  | Closes when the trigger is clicked.                               |
| `closeOnEscape`        | `true`  | Closes when Escape is pressed.                                    |
| `closeOnPointerDown`   | `true`  | Closes on pointer down.                                           |
| `closeOnScroll`        | `true`  | Closes on scroll.                                                 |
| `interactive`          | `false` | Keeps the tooltip open while hovering content when set to `true`. |
| `lazyMount`            | `false` | Defers mounting until the tooltip first opens.                    |
| `unmountOnExit`        | `false` | Unmounts content after exit when enabled.                         |
| `skipAnimationOnMount` | `false` | Allows the first presence animation unless explicitly skipped.    |

### Behavioral notes

- Controlled state uses Ark's `open` and `onOpenChange(details)` shape. Read `details.open`.
- `defaultOpen` provides uncontrolled initial state, while `disabled` turns off tooltip
  interaction without changing the trigger's own semantics.
- Ark presence props such as `present`, `immediate`, `onExitComplete`, `lazyMount`, and
  `unmountOnExit` pass through unchanged.
- `Tooltip.RootProvider`, `useTooltip`, and `useTooltipContext` are available from `@moduix/react`
  for external or inline advanced state access.
- `Tooltip.Trigger` receives the moduix trigger skin only when `asChild` is not used. With
  `asChild`, styling belongs to the child element.
- `Tooltip.Body` forwards content props and children to `Tooltip.Content`. It does not introduce a
  new DOM part or a separate positioning prop surface.
- `Tooltip.Content` owns the visible surface and z-index. `Tooltip.Positioner` owns floating
  layout and exposes Ark variables such as `--available-width`, `--available-height`, and
  `--transform-origin`.
- Ark emits `data-state="open" | "closed"` on trigger and content; `data-value`, `data-current`,
  and `data-expanded` on trigger; and placement attributes such as `data-placement` and
  `data-side` on content.

### Styling hooks

Use the part that owns the visual concern:

- `Tooltip.Body` does not create its own DOM node; style `Tooltip.Positioner` and
  `Tooltip.Content` when you switch to the explicit path.
- `Tooltip.Trigger` for trigger chrome, hover/open feedback, and focus styling.
- `Tooltip.Positioner` for floating layout constraints.
- `Tooltip.Content` for popup surface overrides.
- `Tooltip.Arrow` and `Tooltip.ArrowTip` for arrow size, background, and stroke.

The main public hooks are `data-slot="tooltip-trigger"`, `data-slot="tooltip-positioner"`,
`data-slot="tooltip-content"`, `data-slot="tooltip-arrow"`, and
`data-slot="tooltip-arrow-tip"`.

Ark also adds `data-scope="tooltip"`, `data-part`, `data-state="open" | "closed"`,
`data-value`, `data-current`, `data-expanded`, `data-instant`, `data-placement`, and `data-side`
on the relevant parts. The positioner exposes runtime CSS variables including
`--available-width`, `--available-height`, `--reference-width`, `--reference-height`, `--x`,
`--y`, `--z-index`, and `--transform-origin`.
