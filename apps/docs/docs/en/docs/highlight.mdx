## Choosing the right component

Use `Highlight` when one or more query terms should stay inline with surrounding copy but receive a
shared visual emphasis.

- Choose **`Highlight`** when matched ranges inside an existing sentence or label should stand out.
- Choose **`Text`** when the whole string only needs typography styling and no query-based matching.

## Basic

Place `Highlight` inside the text container that already owns layout and semantics. `Highlight`
styles only the matched ranges; it does not render a wrapper element around the whole sentence.

## Anatomy

`Highlight` is root-only, but the root contract applies to every matched native `<mark>` segment.
Unmatched text stays plain text nodes in the surrounding flow.

```text
Highlight / Highlight.Root
└─ repeated <mark> segments for matched ranges inside the provided text
```

| Part                           | Role                                                                         |
| ------------------------------ | ---------------------------------------------------------------------------- |
| `Highlight` / `Highlight.Root` | Applies moduix mark styling to every matched range inside the provided text. |

## Composition

### Recommended composition

Use the short `<Highlight>` root inside `Text`, `Heading`, a list item, or another text-owning
container. It is equivalent to `<Highlight.Root>`.

### When to go custom

`Highlight` intentionally renders native `<mark>` elements for matched ranges and does not expose a
chunk renderer. When a different element or per-match rendering is required, compose that markup
directly instead of adding a second rendering API to `Highlight`.

### Default props

| Prop         | Default | Notes                                                   |
| ------------ | ------- | ------------------------------------------------------- |
| `query`      | -       | String or string array of terms to highlight.           |
| `text`       | -       | Required full source string.                            |
| `ignoreCase` | `false` | Makes matching case-insensitive.                        |
| `matchAll`   | `false` | Highlights all matches instead of only the first match. |
| `exactMatch` | `false` | Restricts matching to exact whole words.                |

### Behavioral notes

`className`, `style`, `title`, `id`, and other mark-element props apply to every matched `<mark>`.
If the query does not match, the component renders only plain text and no mark element at all.

`Highlight` has no ref target, state machine, callbacks, `Field`/`Fieldset` integration,
`HiddenInput`, `ids`, context hooks, or `RootProvider`.

## Styling

Use `className` for local one-off mark styling, and use `--highlight-*` variables when the same
visual treatment should stay consistent across several highlight instances.

### Styling hooks

- Every matched `<mark>` writes `data-scope="highlight"` and `data-part="root"`.
- `data-slot="highlight-root"` is the stable moduix selector on each matched mark.
- There is no wrapper element, state attribute, or Ark runtime CSS variable because `Highlight`
  renders plain text nodes plus matched `<mark>` segments only.
